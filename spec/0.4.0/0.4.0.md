# Overwatch Homelab — Product Specification v0.4.0 (DRAFT)

> **Status:** Design draft for review. Replaces the prior stale v0.3.0 duplicate that was checked into this directory by mistake. No code yet. The choices called out as "Open questions" at the bottom need owner review before implementation starts.

## Overview

v0.4.0 drives Overwatch below the OS line into the **silicon** of each monitored box — the parts that fail slowly and expensively if nobody is watching. The milestone ships under the theme **"Know your hardware before it dies."**

Four coherent collectors land together, each designed as an additive capability on top of the v0.3.0 agent:

1. **Disk SMART + array health (F1)** — per-physical-disk SMART attribute sampling via `smartctl -a`, plus ZFS pool state (`zpool status -x`) and Linux mdadm arrays (`/proc/mdstat`). A new `DISK_HEALTH` alert metric fires on predictive-failure flags, degraded pools, and resyncing arrays.
2. **GPU telemetry (F2)** — NVIDIA (`nvidia-smi --query-gpu`) and AMD (`rocm-smi`) collectors producing per-GPU utilisation, VRAM, temperature, power, and fan. A new `GPU` alert metric covers temp and VRAM pressure.
3. **UPS / power via NUT (F3)** — agent talks the NUT protocol to a local or remote `upsd`, surfacing battery %, runtime estimate, input voltage, load %, and on-battery state. A new `POWER` alert metric fires on on-battery + low-battery conditions.
4. **Expanded board sensors (F4)** — extends the existing `systeminformation`-based host collector with `lm-sensors` (Linux), `powermetrics` (macOS) and WMI/LibreHardwareMonitor (Windows) to surface motherboard temps, per-NVMe drive temps, fan RPM, and voltage rails.

The spec is deliberately additive. Every new collector plugs into the **agent capability negotiation** introduced in v0.3.0 (S1) — a v0.2.0 or v0.3.0 agent connecting to a v0.4.0 hub keeps working unchanged, and a v0.4.0 agent on a box that lacks `smartctl` / a GPU / a UPS / `lm-sensors` simply declares `false` for that capability and the dashboard hides the tab.

Two tracks that ship together:

1. **Feature track** — SMART + array collector + Disks tab, GPU collector + Monitor-tab GPU card, NUT/UPS collector + Power card, expanded board-sensors collector + Sensors section.
2. **Supporting track** — capability-set extension, per-disk / per-GPU / per-UPS persistence tables, `AlertMetric` enum additions (`DISK_HEALTH | GPU | POWER`), privilege documentation in the Help Center, fixture-based parser unit tests, retention-pruner extension.

## Foundation from v0.2.0 / v0.3.0

Not in scope to re-implement:

- Auth, JWT handshake, cursor pagination, retention pruner, stale-agent pruner — v0.2.0.
- Threshold alert evaluator (CPU / memory / disk) + `MetricSnapshot` persistence — v0.2.0.
- Docker container monitoring, Uptime checks, multi-channel alert delivery, maintenance windows — v0.3.0.
- **Agent capability negotiation (S1)** — v0.3.0. The `capabilities: { host: true, docker?: boolean }` handshake is extended here with six new flags; the negotiation protocol itself is unchanged.
- **Alert evaluator refactor** — v0.3.0 already abstracted the evaluator to take a "source" + "rule" pair. v0.4.0 adds three new source kinds (`disk-health`, `gpu`, `power`) without touching the evaluator's control flow.
- **Alert channels + routing** — v0.3.0. The `metrics` filter on `AlertRoute` gains three new values (`disk-health`, `gpu`, `power`) so existing routes can opt in.
- **Help Center + structured logging + Prometheus `/metrics`** — v0.3.0 primitives are reused; v0.4.0 adds four new Help Center topics and four new Prometheus gauges.

v0.4.0 does **not** touch Docker containers, uptime checks, or alert delivery beyond the enum additions above.

## Scope

### In scope — feature track

#### F1. Disk SMART + array health

1. **Agent SMART collector** — new `apps/lab-agent/src/collectors/smart.ts`. Runs `smartctl --scan-open -j` once per tick to enumerate devices, then `smartctl -a -j <device>` per device (JSON output — `smartmontools >= 7.0` required). Parsed attributes per device: `deviceName` (`/dev/sda`, `/dev/nvme0n1`), `modelFamily`, `modelName`, `serialNumber`, `firmwareVersion`, `capacityBytes`, `rotationRateRpm` (0 for SSD/NVMe), `smartStatusPassed` (bool), `temperatureCelsius`, `powerOnHours`, `powerCycleCount`, plus the attributes of concern (ATA IDs where applicable) — `reallocatedSectorsCt` (5), `currentPendingSectorCt` (197), `offlineUncorrectable` (198), `udmaCrcErrorCount` (199), `wearLevellingCount` (177 / NVMe `percentage_used`), `mediaAndDataIntegrityErrors` (NVMe only).
2. **Tick budget** — SMART polling is slow (0.2–2 s per disk; NVMe is fastest, spinning disks the slowest). The collector runs on a **separate, slower schedule** from the 60 s host tick — default `SMART_INTERVAL_MS=600000` (10 min). Results are cached and attached to every `agent:metrics` payload in a `smart?: SmartSnapshot[]` slice until replaced.
3. **ZFS collector** — runs `zpool list -H -o name,health,size,alloc,free,frag,cap,dedup` plus `zpool status -x` (to detect degraded / resilvering state) and `zpool status -p` per pool for device counts. Parsed: `poolName`, `health` (`ONLINE | DEGRADED | FAULTED | OFFLINE | UNAVAIL | REMOVED`), `sizeBytes`, `allocBytes`, `freeBytes`, `fragPercent`, `capacityPercent`, `dedupRatio`, `state` (`normal | resilvering | scrubbing | degraded`), `vdevs: { name, state, read, write, cksum }[]`. Same 10 min cadence.
4. **mdadm collector** — reads `/proc/mdstat` directly (no shell-out). Parsed per array: `arrayName` (`md0`), `level` (`raid1`, `raid5`, etc.), `state` (`active | resyncing | degraded | recovery | reshape`), `devices: { name, state }[]`, `resyncProgressPercent?`, `resyncSpeedKBs?`, `resyncEtaMinutes?`.
5. **Opt-in** — three independent env flags: `COLLECT_SMART` (default `false`), `COLLECT_ZFS` (default `false`), `COLLECT_MDADM` (default `auto` — detects `/proc/mdstat` on Linux, declares `false` if absent). Each failure mode is the same as v0.3.0 Docker: log once, unset capability, stop trying until restart.
6. **Socket payload** — `agent:metrics` gains `smart?: SmartSnapshot[]`, `zfs?: ZfsPoolSnapshot[]`, `mdadm?: MdadmArraySnapshot[]`. Cap: 64 disks, 16 pools, 16 arrays per tick; overflow is logged and truncated.
7. **Hub-server persistence** — `DiskHealthSnapshot`, `ZfsPoolSnapshot`, `MdadmArraySnapshot` tables, written out-of-band (same pattern as `MetricSnapshot`). Keyed by `(labId, recordedAt, deviceId | poolName | arrayName)`. Retention follows lab `retentionDays`.
8. **Alerting — `DISK_HEALTH`** — evaluator fires an alert when any of the following are true for a device or array in the latest snapshot:
   - `smartStatusPassed === false` (SMART self-assessment failed — the catch-all predictive flag)
   - `reallocatedSectorsCt > 0` **and** increasing over the last 3 snapshots (persistent reallocation trend)
   - `currentPendingSectorCt > 0` (any pending sector at all — these are the leading edge of bad blocks)
   - `temperatureCelsius >= DISK_TEMP_ALERT_C` (default 60 °C)
   - `zfs.health !== 'ONLINE'`
   - `mdadm.state` in `{ degraded, recovery, reshape }`
   Severity mapping: `smartStatusPassed=false` → `critical`; ZFS `FAULTED | UNAVAIL | REMOVED` → `critical`; `DEGRADED`, mdadm `degraded` → `critical`; `recovery | reshape | resilvering | scrubbing` → `warning`; pending-sector / reallocation trend / over-temp → `warning`. Resolve rule: the triggering condition no longer holds for 2 consecutive snapshots (debounced to avoid flapping on scrub-complete transitions).
9. **UI — Disks tab** — new tab on the HomeLab detail page, inserted between **Monitor** and **Services** when the lab reports `smart || zfs || mdadm` capability. Sections:
   - **Physical disks** table: device, model, serial (obfuscated by default, click to reveal), capacity, temperature, power-on hours, reallocated / pending / CRC counters, SMART self-test pill (`PASSED | FAILED | UNKNOWN`), wear-level % for SSD/NVMe.
   - **ZFS pools** list: per-pool health pill, capacity bar, frag %, dedup ratio, a small vdev tree showing device state.
   - **mdadm arrays** list: array, level, state pill, device list, resync progress bar + ETA when in recovery.
   - Row click on a physical disk opens a drawer with the last 30 days of temperature and power-on hours as a sparkline (requires `DiskHealthSnapshot` history — cheap because the collector tick is 10 min, i.e. ~4.3k rows/disk/month).
10. **Overview card** — HomeLab card gains a "`N disks · M healthy`" sub-line when the lab reports SMART capability; sub-line turns red if any disk is in alert.

#### F2. GPU telemetry

1. **Agent GPU collector** — new `apps/lab-agent/src/collectors/gpu.ts`. At agent startup, probes `nvidia-smi` and `rocm-smi` (both via `which` / `where` first, then a one-shot version call). Sets internal `gpu` capability to `true` iff at least one tool is usable and returns >= 1 GPU.
2. **NVIDIA path** — `nvidia-smi --query-gpu=index,name,uuid,utilization.gpu,utilization.memory,memory.total,memory.used,temperature.gpu,power.draw,power.limit,fan.speed,pstate,driver_version --format=csv,noheader,nounits` every tick. Parsed fields become a `GpuSnapshot`. `fan.speed` is optional (not all NVIDIA cards expose it — data-center cards lack fans).
3. **AMD path** — `rocm-smi --showid --showuse --showmemuse --showmeminfo vram --showtemp --showpower --showfan --showproductname --json` per tick. Fields mapped to the same `GpuSnapshot` schema. AMD `temperature` may return multiple sensors (`edge`, `junction`, `mem`); `temperatureCelsius` stores `edge`, `junctionTemperatureCelsius` stores `junction` when present.
4. **GpuSnapshot fields** — `vendor` (`nvidia | amd`), `index` (int), `uuid`, `name`, `driverVersion`, `utilizationGpuPercent`, `utilizationMemoryPercent`, `memoryUsedBytes`, `memoryTotalBytes`, `temperatureCelsius`, `junctionTemperatureCelsius?`, `powerDrawWatts`, `powerLimitWatts`, `fanSpeedPercent?`, `performanceState?` (NVIDIA `P0..P15`).
5. **Cadence** — GPU queries are cheap (~50–150 ms). Runs on the standard 60 s tick; no separate interval.
6. **Opt-in** — single env flag `COLLECT_GPU` (default `auto`). `auto` enables collection iff the runtime probe finds at least one supported tool. Explicit `false` disables regardless.
7. **Socket payload** — `agent:metrics` gains `gpus?: GpuSnapshot[]`. Cap 8 GPUs per tick (homelab-realistic upper bound; lifted via `GPU_MAX` env var).
8. **Hub-server persistence** — `GpuSnapshot` rows on the out-of-band path, keyed by `(labId, recordedAt, gpuUuid)`.
9. **Alerting — `GPU`** — fires when, for any GPU in the latest snapshot:
   - `temperatureCelsius >= GPU_TEMP_ALERT_C` (default `85`) for `consecutiveBreaches` ticks (reuses the existing `AlertThresholds.consecutiveBreaches`)
   - `utilizationMemoryPercent >= GPU_VRAM_ALERT_PERCENT` (default `95`) for `consecutiveBreaches` ticks
   Severity: always `warning` (card-damage scenarios are already covered by the driver's own throttling; Overwatch's role is to tell the operator). Resolve: first tick below the threshold.
10. **UI — GPU card on Monitor tab** — rendered when `agentCapabilities.gpu === true`. Per-GPU card showing: name + driver, utilisation ring, VRAM used/total bar, temperature value with a sparkline, power draw / limit bar, fan %. Multi-GPU hosts get a horizontal carousel. Sparklines pull from a new `GET /api/homelabs/:labId/gpus/:gpuUuid/history` endpoint.

#### F3. UPS / power via NUT

1. **Agent NUT collector** — new `apps/lab-agent/src/collectors/ups.ts`. Speaks the [NUT protocol](https://networkupstools.org/docs/developer-guide.chunked/ar01s09.html) directly over TCP (port 3493) using a small in-repo client — no native dependency. Default target `NUT_HOST=localhost`, `NUT_PORT=3493`, `NUT_UPS_NAME=ups` (NUT's conventional default name). Optional `NUT_USERNAME` / `NUT_PASSWORD` for authenticated UPSes.
2. **Remote NUT support** — the collector does not require `upsd` on the agent's host. A user can run `upsd` on a dedicated "UPS host" and point multiple agents at it (homelabs often have one physical UPS for multiple machines). This is controlled entirely by `NUT_HOST`.
3. **Commands used** — `LIST UPS` (enumerate), `LIST VAR <upsname>` (read all variables). The collector does **not** send `INSTCMD` — no shutdown, no beeper control, no test triggers. Read-only by design.
4. **Parsed variables** (subset from the NUT variable namespace — only stable widely-supported names):
   - `battery.charge` (%)
   - `battery.runtime` (seconds)
   - `battery.voltage` (V, optional)
   - `input.voltage` (V)
   - `input.frequency` (Hz, optional)
   - `output.voltage` (V, optional)
   - `ups.load` (%)
   - `ups.status` (raw string — `OL`, `OB`, `OB DISCHRG`, `LB`, `CHRG`, etc.)
   - `ups.model`, `ups.mfr`, `ups.serial` (static metadata)
   Derived booleans: `onBattery = status contains 'OB'`, `lowBattery = status contains 'LB'`, `online = status contains 'OL'`, `charging = status contains 'CHRG'`.
5. **Cadence** — UPS variables change quickly when the power flickers. Collector runs every `UPS_INTERVAL_MS` (default `30000` — 30 s). Separate from the 60 s host tick because operators need fast "on battery" detection.
6. **Opt-in** — env flag `COLLECT_UPS` (default `false` — NUT requires an installed daemon, never a safe default). On first failure to connect, log once, unset capability, stop retrying until the agent restarts.
7. **Socket payload** — dedicated `agent:ups` event rather than riding inside `agent:metrics`, because the cadence differs. Payload: `{ labId, recordedAt, ups: UpsSnapshot }`.
8. **Hub-server persistence** — `UpsSnapshot` rows keyed by `(labId, recordedAt)`. Retention follows lab `retentionDays`.
9. **Alerting — `POWER`** — two rules, both `critical` severity:
   - **On-battery** — fires on the first snapshot where `onBattery === true`. Resolves when `onBattery === false` for 2 consecutive snapshots (ride-through debounce).
   - **Low-battery** — fires when `lowBattery === true` OR `battery.charge <= LOW_BATTERY_ALERT_PERCENT` (default `25`) OR `battery.runtime <= LOW_RUNTIME_ALERT_SECONDS` (default `300`). Resolves when all three clear.
   Delivery is intentionally routed with severity `critical` because a missed on-battery event is the difference between a clean shutdown and data loss.
10. **UI — Power card** on Monitor tab — rendered when `agentCapabilities.ups === true`. Big status pill (`ONLINE | ON BATTERY | LOW BATTERY`) in traffic-light colour, battery % ring gauge, runtime estimate in MM:SS, input-voltage number, load % bar, UPS model/serial in a footer. A small "history" strip below shows the last 24 h of on-battery events as tick marks.

#### F4. Expanded board sensors

1. **Agent board-sensors collector** — new `apps/lab-agent/src/collectors/sensors.ts`. Extends, rather than replaces, the `systeminformation`-based host collector. Runs alongside the existing `collectMetrics()` call on the normal 60 s tick.
2. **Linux path** — `lm-sensors` via `sensors -j` (JSON output from `lm-sensors >= 3.6`). Parses the nested `chip → feature → sub-feature` tree into a flat array of `SensorReading { chip, label, kind: 'temp' | 'fan' | 'voltage' | 'power' | 'other', valueNumeric, unit, critical?, min?, max? }`. Per-NVMe drive temperatures exposed by the kernel's `nvme` driver are captured via the `nvme-*` chips in `sensors -j`; no separate `smartctl` call needed for that specific reading.
3. **macOS path** — `powermetrics -n 1 -i 1000 --samplers smc --format plist`. Parsed via `plist` library. Extracts the package / P-core / E-core temperatures, fan RPM (where present — MacBook Pros have them, minis don't), and the power subsystem's rail voltages. Requires `sudo` — if `powermetrics` exits with permission-denied, collector logs once, declares `sensors: false`, and gives up. Help Center documents `sudo -n powermetrics` via a sudoers rule.
4. **Windows path** — WMI via [`node-wmi`](https://www.npmjs.com/package/node-wmi) calling `Win32_TemperatureProbe`, `Win32_Fan`, and `Win32_VoltageProbe`, plus an optional LibreHardwareMonitor bridge if `LHM_URL` is configured (pointing at an instance exposing its HTTP JSON endpoint). Windows native WMI classes are famously sparse — most homebuilt boards return nothing. The Help Center calls out that LibreHardwareMonitor is the practical route on Windows.
5. **Cadence** — 60 s tick (rides the existing host metrics schedule — no separate interval). `lm-sensors` and `powermetrics` reads are 20–100 ms.
6. **Opt-in** — env flag `COLLECT_SENSORS` (default `auto` — enables on Linux if `sensors` binary exists, enables on macOS if `powermetrics` is available without password prompt, enables on Windows only if `LHM_URL` is set or WMI returned non-empty at startup probe).
7. **Socket payload** — `agent:metrics` gains `sensors?: SensorReading[]`. Cap 256 readings per tick (a typical desktop board reports 20–40; a server board can reach ~100 including per-DIMM temps).
8. **Hub-server persistence** — `SensorSnapshot` stores `(labId, recordedAt, chip, label, kind, valueNumeric, unit)`. **Not indexed per-label** — the cardinality is user-dependent; instead `(labId, recordedAt)` is the primary index and per-label queries scan within a window. This is cheap up to the ~100-reading-per-tick upper bound.
9. **UI — Sensors section on Monitor tab** — rendered when `agentCapabilities.sensors === true`. Collapsible by default (most users will want it hidden). When expanded: three sub-tables — Temperatures (chip, label, value, chart icon), Fans (chip, label, RPM), Voltages (chip, label, value, nominal rail). Click a row to open a drawer with that sensor's last 1 h sparkline. NVMe drive temperatures surface here **and** as the `temperatureCelsius` field on the matching `DiskHealthSnapshot` row — they're the same physical reading.
10. **Alert policy** — deliberately **no new alert metric** for sensors. Board sensors are highly vendor-specific and their nominal ranges vary wildly (a Xeon at 75 °C is fine, a Ryzen at 75 °C is concerning); alerting would need per-chip rules that belong in a later milestone. Over-temp conditions that actually matter for storage and GPUs are already covered by `DISK_HEALTH` and `GPU` alerts.

### In scope — supporting track

- **S1 — Capability-set extension.** `AgentCapabilitiesSchema` from v0.3.0 gains `smart?: boolean`, `zfs?: boolean`, `mdadm?: boolean`, `gpu?: boolean`, `ups?: boolean`, `sensors?: boolean`. Hub treats all absent keys as `false`. The UI gating helper `useAgentCapability(labId, 'gpu')` that v0.3.0 introduced for Docker continues to work unchanged.
- **S2 — Per-disk / per-GPU / per-UPS persistence.** Six new Prisma tables (`DiskHealthSnapshot`, `ZfsPoolSnapshot`, `MdadmArraySnapshot`, `GpuSnapshot`, `UpsSnapshot`, `SensorSnapshot`). Retention pruner (v0.2.0 background job) extended to cover all six in the same 6 h tick.
- **S3 — `AlertMetric` enum additions.** `DISK_HEALTH`, `GPU`, `POWER` (all-caps per the convention set by `UPTIME` in v0.3.0; note the historical lower-case `cpu | memory | disk` values remain — migrating them is out of scope and called out in the Open Questions). `AlertRoute.metrics` string-array filter accepts the new values.
- **S4 — Privilege documentation.** Four new Help Center topics: `disk-health`, `gpu-telemetry`, `ups-nut`, `board-sensors`. Each includes a **Privileges** section with the exact commands/groups needed (`sudo smartctl`, `disk` group, `nvidia-persistenced`, `upsmon` ACL config on `upsd`, `sudoers` rule for `powermetrics`). Also a **How to test** section per the v0.3.0 topic template.
- **S5 — Collector unit tests with fixtures.** Real sample outputs captured as fixtures under `apps/lab-agent/test/fixtures/`:
  - `smartctl-seagate-spinning.json`, `smartctl-samsung-nvme.json`, `smartctl-wd-ssd.json`, `smartctl-failing-pending-sectors.json`
  - `zpool-status-healthy.txt`, `zpool-status-degraded.txt`, `zpool-status-resilvering.txt`, `zpool-list.txt`
  - `mdstat-raid1-clean.txt`, `mdstat-raid5-recovering.txt`
  - `nvidia-smi-rtx3090.csv`, `nvidia-smi-multi-gpu.csv`, `rocm-smi-7900xtx.json`
  - `nut-list-var-apc-smart-1500.txt`, `nut-list-var-cyberpower.txt`, `nut-ups-on-battery.txt`, `nut-ups-low-battery.txt`
  - `sensors-linux-ryzen.json`, `sensors-linux-xeon-with-nvme.json`, `powermetrics-m2-pro.plist`, `lhm-windows.json`
  Each parser has a test that asserts the exact shape it produces from the fixture — regression protection for vendors changing output formats.
- **S6 — Prometheus `/metrics` extension.** Four new gauges: `overwatch_disk_health_alerts_active`, `overwatch_gpu_temperature_celsius{lab_id, gpu_index}`, `overwatch_ups_on_battery{lab_id}`, `overwatch_sensors_reported{lab_id}` (count of readings, not values — keeps cardinality sane). Emitted per scrape via the v0.3.0 `prom-client` wiring.
- **S7 — Client test harness extension.** Add component tests for `DisksTab`, `GpuCard`, `PowerCard`, `SensorsSection` and hook tests for `useDiskHealth(labId)`, `useGpuSnapshots(labId)`, `useUpsSnapshot(labId)`. Target: +20 tests on top of v0.3.0's 25.
- **S8 — Playwright scenario.** Mocked-agent scenario that injects synthetic `agent:metrics` payloads with `smart`, `gpus`, and `sensors` slices and asserts the three surfaces render correctly; a second scenario fires `DISK_HEALTH` + `POWER` alerts and verifies they route through the v0.3.0 delivery pipeline end-to-end. (The v0.3.0 uptime scenario is unchanged.)

### Out of scope (deferred to v0.5.0+)

- **Network device SNMP discovery / polling** — routers, switches, PDUs. Separate milestone.
- **Speedtest / gateway latency / DNS health** — synthetic internet probes — separate "network observability" milestone.
- **Agent packaging** (deb / rpm / Homebrew formula / signed releases / auto-update).
- **TimescaleDB migration / tiered downsampling** — `DiskHealthSnapshot` at 10 min cadence for 30 days is only ~4.3k rows per disk, so a tiered store is not required yet.
- **Multi-user RBAC / team workspaces.**
- **TLS / mTLS between agent and hub.**
- **PWA mobile layout / public status pages.**
- **Kubernetes / Podman pod awareness** beyond v0.3.0's Docker collector.
- **Container log tailing.**
- **Backup job ingestion** (restic / borg / rsnapshot exit codes).
- **Per-sensor alert rules + chip-specific nominal ranges** for F4 — needs a sensor-profile library first.
- **SMART self-test scheduling** (triggering `smartctl -t short` on a cron) — Overwatch observes, doesn't drive self-tests in v0.4.0.
- **NUT `INSTCMD` (remote shutdown / beeper)** — read-only for v0.4.0; writeable UPS control is a separate security review.
- **User-editable alert templates.**
- **OpenAPI spec generation** from Zod.

## User Stories

User-story numbers continue from v0.3.0's US-57.

### Disks (F1)

- **US-58** — As a user, I can enable SMART collection on an agent by installing `smartmontools` and setting `COLLECT_SMART=true`. If `smartctl` isn't on PATH or I lack permissions, the agent logs why and keeps running with host metrics only.
- **US-59** — As a user, I see a **Disks** tab on any HomeLab whose agent reports SMART / ZFS / mdadm capability, listing every physical disk with model, serial (obfuscated), temperature, power-on hours, and SMART self-assessment pill.
- **US-60** — As a user, I see my ZFS pools' health, capacity, fragmentation, and vdev state on the Disks tab — including a clear "resilvering — 42 %" bar during rebuilds.
- **US-61** — As a user, I see mdadm arrays with their RAID level, device list, and a resync progress + ETA bar while they're recovering.
- **US-62** — As a user, I get a `DISK_HEALTH` alert through the v0.3.0 alert routes when any physical disk reports a failed self-assessment, pending sectors, a reallocation trend, or over-temperature, or when a ZFS pool goes DEGRADED or an mdadm array enters degraded/recovery state.
- **US-63** — As a user, clicking a physical disk opens a drawer with the last 30 days of temperature and power-on-hours sparklines — I can spot the drive that's been creeping warmer.

### GPU (F2)

- **US-64** — As a user with an NVIDIA or AMD GPU and the appropriate driver CLI installed, the agent detects it automatically (`COLLECT_GPU=auto`) and reports utilisation, VRAM, temperature, power, and fan on the Monitor tab.
- **US-65** — As a user, a GPU card on the Monitor tab shows per-GPU live telemetry with a temperature sparkline and VRAM bar, and I get a `GPU` alert when temperature crosses my threshold for N ticks.
- **US-66** — As a user with multiple GPUs, each GPU gets its own card in a horizontal carousel and each emits alerts keyed by GPU UUID so I can tell *which* one overheated.

### UPS (F3)

- **US-67** — As a user running NUT on my homelab, I set `COLLECT_UPS=true`, `NUT_HOST`, and `NUT_UPS_NAME` on any agent — the UPS does not have to be plugged into the agent's host.
- **US-68** — As a user, I see a Power card on the Monitor tab showing battery %, runtime estimate, input voltage, load %, and a big ON BATTERY / ONLINE / LOW BATTERY status pill.
- **US-69** — As a user, I get a **critical** `POWER` alert routed to my configured alert channels within 30 s of my UPS switching to battery, and a second critical alert when the battery drops below my low-battery threshold — so I have time to trigger my own shutdown procedure.

### Board sensors (F4)

- **US-70** — As a user on Linux with `lm-sensors` configured, the agent picks up motherboard temps, per-NVMe drive temps, fan RPMs, and voltage rails automatically.
- **US-71** — As a user, I see a collapsible Sensors section on the Monitor tab with Temperatures, Fans, and Voltages sub-tables; clicking any row shows a 1 h sparkline of that specific sensor.
- **US-72** — As a user, I understand that board sensors are informational only — Overwatch does not alert on raw sensor readings in v0.4.0, and the Help Center explains why.

### Capability coherence

- **US-73** — As a user, every new tab and card respects the v0.3.0 capability gating — if my agent reports `gpu: false` I don't see a GPU card, if it reports `ups: false` I don't see a Power card, and if it reports nothing new (v0.3.0 agent connected to v0.4.0 hub) the dashboard looks exactly like v0.3.0.

## API Changes

### New endpoints

#### `GET /api/homelabs/:labId/disks/latest`

Returns the most recent `DiskHealthSnapshot` per `deviceName` for the lab, plus latest `ZfsPoolSnapshot` and `MdadmArraySnapshot` rows.

{
  "success": true,
  "data": {
    "recordedAt": "ISO8601",
    "disks": [
      {
        "deviceName": "/dev/nvme0n1",
        "modelFamily": "Samsung SSD 980 PRO",
        "modelName": "Samsung SSD 980 PRO 1TB",
        "serialNumber": "S5P2NS0…",
        "firmwareVersion": "5B2QGXA7",
        "capacityBytes": 1000204886016,
        "rotationRateRpm": 0,
        "smartStatusPassed": true,
        "temperatureCelsius": 42,
        "powerOnHours": 14520,

### Socket.IO contract changes

- **`agent:register`** — `capabilities` payload extended with the six new flags from S1. Backwards compatible: absent = `false`.

- **`lab:ups`** — new broadcast carrying the `ups` snapshot. Fires on every UPS tick.
- **`lab:sensors`** — new broadcast carrying the `sensors` slice. Fires on every host tick that includes a sensors payload.
- **`lab:alert`** (existing) — `alertMetric` enum gains `DISK_HEALTH | GPU | POWER`; payload gains optional `deviceName?`, `gpuUuid?` to scope the alert to a specific device.

## Architecture Changes

### Hub-server additions

- **`src/collectors/` mirror** — parsing logic that's shared with agent lives in `packages/shared-types` (since the agent runs the parsers against live tool output and the hub just deserialises the resulting schemas). No SMART/ZFS parsing on the hub; the hub is a dumb persister + alert evaluator.
- **`src/persist/disks.ts`, `src/persist/gpus.ts`, `src/persist/ups.ts`, `src/persist/sensors.ts`** — one module per new snapshot family, each subscribing to the relevant `agent:*` socket event via the v0.3.0 `onAgentMetrics` / `onAgentUps` pattern. Each module batch-inserts with `prisma.$transaction` of `createMany` calls keyed by the appropriate uniqueness tuple.
- **`src/alerts/evaluators/diskHealth.ts`** — new evaluator implementing the rules in F1.8. Takes the latest + previous-two `DiskHealthSnapshot` rows for a lab (to compute the reallocation trend) and the latest `ZfsPoolSnapshot` + `MdadmArraySnapshot`. Outputs fire/resolve events into the v0.3.0 alert-router pipeline.
- **`src/alerts/evaluators/gpu.ts`** — reads the last N (= `consecutiveBreaches`) `GpuSnapshot` rows per GPU and applies the temp / VRAM rules.
- **`src/alerts/evaluators/power.ts`** — reads the latest `UpsSnapshot`, applies on-battery + low-battery rules, always-critical severity.
- **`src/env.ts`** — adds `DISK_TEMP_ALERT_C` (default 60), `GPU_TEMP_ALERT_C` (default 85), `GPU_VRAM_ALERT_PERCENT` (default 95), `LOW_BATTERY_ALERT_PERCENT` (default 25), `LOW_RUNTIME_ALERT_SECONDS` (default 300). All optional.
- **Retention pruner** extended to prune `DiskHealthSnapshot`, `ZfsPoolSnapshot`, `MdadmArraySnapshot`, `GpuSnapshot`, `UpsSnapshot`, `SensorSnapshot` on the same 6 h schedule using each lab's `retentionDays`.
- **`src/metrics/prometheus.ts`** — adds the four gauges from S6. Labels chosen to keep cardinality bounded: `gpu_index` rather than `gpu_uuid`, no per-disk labels.
- **No new worker.** F1–F4 fit inside the v0.3.0 out-of-band persistence path; no new `Job` kinds.

### Hub-client additions

- **`DisksTab`** on HomeLab detail page — rendered when `agentCapabilities.smart || zfs || mdadm`. Three collapsible sub-sections; row-click drawer for per-disk history.
- **`GpuCard`** on Monitor tab — rendered when `agentCapabilities.gpu === true`. Per-GPU carousel.
- **`PowerCard`** on Monitor tab — rendered when `agentCapabilities.ups === true`.
- **`SensorsSection`** on Monitor tab — rendered when `agentCapabilities.sensors === true`. Collapsed by default; `localStorage` key `ui.sensorsExpanded` persists the user's choice.
- **Hooks** — `useDiskHealth(labId)`, `useGpuSnapshots(labId)`, `useUpsSnapshot(labId)`, `useSensorSnapshots(labId)`. All TanStack Query v5 with socket-driven invalidation plus a poll fallback (30 s for `disk`, 60 s for `gpu`/`sensors`, 30 s for `ups`).
- **`AlertsTab`** — filter chips gain `DISK_HEALTH | GPU | POWER`. `AlertBanner` / sidebar badge automatically pick up the new enum values via the existing severity-agnostic counting.
- **Help Center** — new routes `/help/disk-health`, `/help/gpu-telemetry`, `/help/ups-nut`, `/help/board-sensors`.
- **New overview-card sub-line** — `N disks · M healthy` (red when any disk is in alert) when `smart === true`.

### Lab-agent additions

- **`src/collectors/smart.ts`** — spawns `smartctl` with a 5 s wall-clock timeout per disk, bounded concurrency of 4. Tracks per-disk failure count; 3 consecutive failures on the same device moves that device to a "skip until restart" list to avoid log spam.
- **`src/collectors/zfs.ts`** — spawns `zpool`; parses text output for `zpool status -x` (structure is stable but not JSON), parses tabular output for `zpool list -H`.
- **`src/collectors/mdadm.ts`** — reads `/proc/mdstat` as a file (no shell-out). Linux-only.
- **`src/collectors/gpu.ts`** — dispatches to either the NVIDIA or AMD code path based on which tool the startup probe found. Multi-vendor hosts (rare but supported) run both.
- **`src/collectors/ups.ts`** — tiny NUT protocol client (~100 LOC, in-repo). Connects on demand, 2 s connect timeout, 3 s total budget, closes after each poll.
- **`src/collectors/sensors.ts`** — platform-switched: `sensors -j` on Linux, `powermetrics` on macOS, `node-wmi` + optional LHM on Windows.
- **`src/capabilities.ts`** — extended from v0.3.0 to probe the six new tools at startup: `which smartctl`, `which zpool`, `stat /proc/mdstat`, `which nvidia-smi || which rocm-smi`, TCP-connect to `NUT_HOST:NUT_PORT`, `which sensors || stat /usr/bin/powermetrics || env LHM_URL`. Each probe is `Promise.race`d with a 1 s timeout so agent startup stays fast even on a broken box.
- **Scheduling** — the agent now runs three independent tick cadences on top of the existing heartbeat:
  - Fast tick (`METRICS_INTERVAL_MS`, default 60 s) — host metrics + GPU + sensors.
  - Slow tick (`SMART_INTERVAL_MS`, default 600 s) — SMART + ZFS + mdadm.
  - UPS tick (`UPS_INTERVAL_MS`, default 30 s) — NUT only.
  Each cadence is its own `setInterval(...).unref()` so the existing shutdown path stays clean.

## Security Considerations

1. **`smartctl` privileges** — `smartctl` needs raw access to block devices. On Linux that usually means running the agent as root or granting `CAP_SYS_RAWIO` + membership in the `disk` group. The Help Center `disk-health` topic spells out both options and warns that running the whole agent as root is the blunt choice; the per-cap option (`setcap cap_sys_rawio+ep /usr/sbin/smartctl` + `usermod -aG disk overwatch`) is the recommended one.
2. **Disk serial numbers are identifying data** — `DiskHealthSnapshot.serialNumber` is stored plaintext (operators need it for RMAs) but the UI obfuscates by default and requires a click to reveal. API responses return the full serial — not a leak (API is owner-scoped) but documented.
3. **`nvidia-smi` / `rocm-smi` don't need root** on properly configured systems, but AMD `rocm-smi` requires the user to be in the `video` or `render` group. The Help Center `gpu-telemetry` topic documents the group add.
4. **NUT credentials** — `NUT_USERNAME` + `NUT_PASSWORD` are env vars on the agent. Never written to persistent storage on the hub, never transmitted in the socket handshake (the agent authenticates to `upsd` locally on its own host). Pino redaction pattern from v0.3.0 (`/token|secret|password|credential|authorization/i`) already catches `NUT_PASSWORD` in logs — verified by adding a fixture test.
5. **NUT protocol is cleartext** — NUT does not offer native TLS. When `NUT_HOST` is on the agent's own loopback this is fine; cross-host NUT should run over a VPN (Tailscale / WireGuard) and the Help Center states this explicitly.
6. **No `INSTCMD` / writeable UPS control** — the collector never issues commands that change UPS state. Prevents a compromised hub from triggering a shutdown on every connected lab's UPS.
7. **`lm-sensors` + `powermetrics`** — `lm-sensors` needs no privilege beyond read access to `/sys`. `powermetrics` on macOS requires `sudo`; the Help Center `board-sensors` topic documents a sudoers rule scoped to the exact binary + flags (`%overwatch ALL = (root) NOPASSWD: /usr/bin/powermetrics -n 1 -i 1000 --samplers smc --format plist`).
8. **WMI on Windows** — requires the agent to run under a user with WMI query rights. No new privilege surface beyond what a monitoring agent typically needs.
9. **GPU card vendor strings** — `name`, `driverVersion`, `performanceState` are freeform strings from the vendor CLI. Parser rejects values longer than 256 chars to prevent a malformed vendor update from ballooning rows.
10. **Capability-gate leakage** — the new `agentCapabilities` JSON is returned by `GET /api/homelabs/:id` as it was in v0.3.0, scoped to the owner. No leakage of disk models / GPU models / UPS serials across users.

## Known Limitations (v0.4.0)

1. **SMART is point-in-time sampled, not event-driven.** Some SMART attributes (self-test-errors, log-page reads) would ideally push immediately on change, but `smartctl` exposes no event mode. Operators waiting on a brand-new bad-block event will see it within one `SMART_INTERVAL_MS` tick (default 10 min).
2. **ZFS scrubs report state but Overwatch does not schedule them.** Observability only.
3. **mdadm is Linux-only.** BSD `graid` and Windows Storage Spaces are out of scope in v0.4.0.
4. **No per-sensor alert rules for F4.** Informational only by design — see the reasoning in F4.10.
5. **NUT-only UPS support.** APC PowerChute, Eaton IPM, and vendor-specific agents are out of scope; users on those stacks install NUT alongside or skip F3.
6. **No GPU process list.** `nvidia-smi pmon` could tell the user *which* process is using VRAM — deferred to keep F2 focused on card-level telemetry.
7. **Integrated GPUs (Intel iGPU, Apple Silicon GPU, AMD APU) are ignored.** The vendor CLIs don't report them in a consistent way; `powermetrics` partially covers Apple Silicon GPU but is routed through F4 sensors only.
8. **Disk temperature appears in two places** (`DiskHealthSnapshot.temperatureCelsius` and `SensorSnapshot` with `kind=TEMP` and a matching `nvme-*` chip) — by design, since SMART is the authoritative source but `lm-sensors` may be the only one available on hosts without `smartmontools`. The UI picks one per-host (SMART wins when both exist).
9. **Serial numbers are not bcrypt-hashed.** They're stored plaintext for operator RMA workflows. Acceptable because the API is owner-scoped and the UI obfuscates by default.
10. **No cross-agent UPS aggregation.** If one UPS powers three hosts and each host's agent points at the same `NUT_HOST`, Overwatch stores three `UpsSnapshot` streams (one per lab). Deduping to a single "UPS resource" is a v0.5.0+ concern once the "network device" resource type lands.

## Rollback & Compatibility

- All schema changes additive. Existing monitoring resources, alerts, channels, routes, and uptime checks keep working unchanged.
- **Agent compatibility:** v0.2.0 and v0.3.0 agents connect without the new `capabilities` keys — hub treats every new flag as `false` and the dashboard hides Disks / GPU / Power / Sensors for those labs. v0.4.0 agents with no new env vars set behave identically to v0.3.0 agents.
- **Alert compatibility:** existing `Alert` rows keep their `metric` value unchanged. The enum extension is PostgreSQL-safe because Prisma's enum migrations are additive by default. Existing `AlertRoute.metrics` filters continue to work; routes that want the new values must be updated by the user.
- **Capability additions are independent.** Disabling `COLLECT_SMART` on a specific agent does not affect GPU or UPS on the same agent.
- Socket.IO event names are additive. v0.2.0 / v0.3.0 dashboards ignore `lab:disks`, `lab:gpus`, `lab:ups`, `lab:sensors`, and the new `agent:ups` event on the agent side is never emitted unless `COLLECT_UPS=true`.
- Rolling the hub back to v0.3.0 requires: stop agents with any of the four new `COLLECT_*` flags enabled (so they stop emitting the new slices), drop the six new tables (or leave them — v0.3.0 simply won't touch them), revert env vars. No destructive cross-release data reshape.
- **Retention pruner rollback** — extending the pruner is additive; on rollback the v0.3.0 pruner simply ignores the new tables.

Users running NVMe-only boxes might reasonably want 60 s. Confirm 10 min default, or split into `SMART_INTERVAL_NVME_MS` vs `SMART_INTERVAL_ATA_MS`?

2. **Disk-serial obfuscation.** Proposing UI obfuscation by default with click-to-reveal. Alternative: obfuscate in the API response too (send `ending in XYZ` rather than the full serial) and only reveal via a separate endpoint that audit-logs the access. Operator-friendliness vs defence-in-depth — which side?

3. **NUT write commands (`INSTCMD`).** v0.4.0 is strictly read-only. A future "trigger graceful shutdown on low battery" feature would need `INSTCMD`. Do you want a gated `UPS_ALLOW_WRITE=false` env knob now (even if we don't use it yet) to signal the design direction, or leave it for v0.5.0?

4. **Alert severity for `DISK_HEALTH` pending sectors.** Proposing `warning` because pending sectors *usually* resolve on the next write — but some operators treat the first pending sector as the "drop everything and clone the drive" signal. Promote to `critical` by default, or keep `warning` with a documented override?

5. **GPU alert for idle temperature creep.** The current rule is "temp >= threshold for N ticks". It catches runaway workloads but misses "idle temp has climbed 15 °C over the last month" (the early-warning signal for a dying fan). Add a secondary trend-based rule, or defer trend alerts to a future milestone?

6. **Lower-case legacy `AlertMetric` values.** `cpu | memory | disk | UPTIME | DISK_HEALTH | GPU | POWER` is inconsistent. Options: (a) leave it — purely cosmetic, migrations are a pain; (b) ship a migration that renames `cpu → CPU`, `memory → MEMORY`, `disk → DISK` in this milestone — cleanest long-term but touches every v0.2.0 alert row and the evaluator. Confirm (a)?

7. **macOS `powermetrics` sudoers rule.** Proposing a scoped sudoers rule shipped as a Help Center snippet. Alternative: don't support macOS board sensors at all in v0.4.0 — macOS's useful sensor set is thin anyway. Half-support now, or wait for a proper macOS collector?

8. **GPU vendor coverage.** Proposing NVIDIA + AMD only. Intel Arc (`intel_gpu_top`) and Apple Silicon GPU both have increasing homelab relevance. Ship v0.4.0 with two vendors and add in v0.4.1, or hold until all four land?

9. **Cross-agent UPS deduplication.** If three hosts share one physical UPS via NUT, v0.4.0 stores three parallel `UpsSnapshot` streams. Should we add a `UpsDevice` model (one per `(NUT_HOST, NUT_UPS_NAME)` tuple) and relate `UpsSnapshot` to that instead of to the lab? Cleaner model but more migration surface — worth it now or defer?

10. **Per-disk retention.** Some operators want long-lived SMART history (tracking power-on hours and reallocation trends across years) even when their metric retention is 30 days. Introduce a separate `DISK_RETENTION_DAYS` env (default 365) vs the shared `retentionDays`, or keep everything on the one knob for v0.4.0?
