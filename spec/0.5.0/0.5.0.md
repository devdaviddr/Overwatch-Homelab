# Overwatch Homelab ‚Äî Product Specification v0.5.0 (DRAFT)

> **Status:** Design draft for review. Builds on v0.3.0 (containers + uptime checks + multi-channel alert delivery) and v0.4.0 (SMART / GPU / UPS / environmental sensors ‚Äî assumed already shipped). No code yet. Choices flagged "Open questions" at the bottom need owner review before implementation starts.

## Overview

v0.5.0 pushes Overwatch past host-local monitoring into the **network layer** ‚Äî the part of a homelab that, when it misbehaves, makes every host look "fine" while nothing actually works. Three coherent features ship together under the theme **"See the network, not just the hosts."**

1. **SNMP-based network devices** ‚Äî routers, switches, APs, NAS, firewalls become first-class monitored resources without needing an agent on them. A new `NETWORK_DEVICE` resource type is polled by the hub via SNMP (v2c / v3) on a configurable interval; the dashboard gains per-interface up/down/speed/utilisation, throughput sparklines pulled from `ifHCInOctets` / `ifHCOutOctets` counters, and device-level identity / uptime / CPU / memory when a recognised vendor MIB is present.
2. **Synthetic network health** ‚Äî the hub itself runs probes that aren't tied to any specific device: public-internet speedtest (rate-limited), continuous ICMP to each discovered default gateway, and A/AAAA resolution against a user-configured resolver set. Results flow through a new `NETWORK_HEALTH` alert metric so sustained degradation actually pages.
3. **Network dashboard surface** ‚Äî a new top-level `/network` route aggregates devices + synthetic health + a best-effort topology hint (gateway ‚Üí device associations inferred from ARP tables where an SNMP-readable ARP/IP-MIB is available; otherwise a flat list).

The spec deliberately keeps v0.3.0 / v0.4.0 data models and agent capability negotiation backward-compatible. Every new data model is additive; the new *hub* capability flags (see S1) fail feature availability gracefully when the required binaries / libraries aren't present. No existing agent needs to change to benefit from any v0.5.0 feature ‚Äî network devices are agent-less by design.

Two tracks that ship together:

1. **Feature track** ‚Äî `NETWORK_DEVICE` resource type + SNMP poller (F1, F2), network device UI + per-interface alerts (F3), SNMP discovery wizard (F4), synthetic network health probes (F5), `/network` dashboard surface (F6).
2. **Supporting track** ‚Äî hub capability flags, per-device credential encryption reusing the v0.3.0 AES-256-GCM helper, speedtest rate limiting, alert route integration, Help Center entries, test coverage.

## Foundation from v0.4.0 / earlier

Not in scope to re-implement:

- Auth, cursor pagination, profile, Help Center, sidebar ‚Äî v0.2.0.
- Threshold alert evaluator (CPU / memory / disk) + `MetricSnapshot` persistence + retention pruner ‚Äî v0.2.0.
- Docker container monitoring, uptime checks, alert channels + routes + delivery worker, maintenance windows, agent capability negotiation, agent token auth, Prometheus `/metrics`, pino structured logging, job runner (alert-delivery-kind) ‚Äî v0.3.0.
- `CREDENTIAL_ENCRYPTION_KEY` + AES-256-GCM helper at `apps/hub-server/src/lib/crypto.ts` ‚Äî v0.3.0.
- SMART / ZFS / mdadm disk health, GPU telemetry (`nvidia-smi` / `rocm-smi`), NUT/UPS integration, thermal / fan / board sensors ‚Äî v0.4.0 (assumed shipped; their capability flags live on the agent, not the hub).

Synthetic network health and SNMP polling **reuse** the v0.3.0 job runner by registering new job kinds (`snmp-poll`, `network-probe`); the runner's DB-backed `Job` table stays a single row type. The new `NETWORK_HEALTH` and `INTERFACE_UTILISATION` alerts flow through the existing `Alert` table, evaluator, router, and delivery worker ‚Äî no parallel alert path.

## Scope

### In scope ‚Äî feature track

#### F1. `NETWORK_DEVICE` resource type

1. **Enum extension** ‚Äî `ResourceType` in `apps/hub-server/prisma/schema.prisma` gains `NETWORK_DEVICE`, joining the existing `HOMELAB | SERVER | PC` set. Existing rows are unaffected; new validation treats `NETWORK_DEVICE` as agent-less, so `agentHubUrl`, `heartbeatIntervalMs`, `metricsIntervalMs`, and the stale-agent pruner all skip these rows.
2. **Storage decision ‚Äî extend `HomeLab`, not a parallel `NetworkDevice` table.** Rationale: the user-facing mental model already treats every monitored thing as a "resource" (the UI even calls the column "Resource"), ownership / labels / alert routes / maintenance windows already key off `HomeLab.id`, and every existing route (`/api/homelabs/‚Ä¶`) keeps working for listing and scoping without a second ownership story. A sibling table would duplicate every one of those scaffolds. The agent-linked columns (`agentHubUrl`, interval overrides, `agentCapabilities`, `agentTokenHash`, `agentAuthMode`) are simply unused on `NETWORK_DEVICE` rows ‚Äî this is cheaper than a polymorphic table and doesn't break any existing query.
3. **New `HomeLab` columns** (all nullable, applicable only to `NETWORK_DEVICE`):
   - `networkHostname: String?` ‚Äî DNS name or IP of the device.
   - `networkDeviceKind: enum NetworkDeviceKind { ROUTER | SWITCH | AP | NAS | FIREWALL | OTHER }?` ‚Äî drives icon + default interface-filter behaviour in the UI.
   - `snmpVersion: enum SnmpVersion { V2C | V3 }?`
   - `snmpPort: Int?` ‚Äî default `161` when null; stored explicitly so non-standard ports are first-class.
   - `snmpPollIntervalSec: Int?` ‚Äî default `60`; same range-clamp as v0.3.0 (30 / 60 / 300 / 900).
   - `snmpCredentialCiphertext: Bytes?` ‚Äî AES-256-GCM blob holding the v2c community string **or** the v3 credential set (`{ username, authProtocol, authPassword, privProtocol, privPassword, contextName? }`). Encryption reuses `apps/hub-server/src/lib/crypto.ts` (the v0.3.0 helper) driven by the existing `CREDENTIAL_ENCRYPTION_KEY` env var ‚Äî no new key, no new crypto.
4. **Create flow** ‚Äî the existing "New Resource" wizard grows a "Network Device" option. Fields are type-gated: picking `NETWORK_DEVICE` hides agent / interval fields and reveals hostname / device-kind / SNMP-version / credential fields. Successful create fires a single synchronous `sysDescr` GET so the user sees "device identified" before leaving the wizard.
5. **Backwards compatibility** ‚Äî existing `HOMELAB | SERVER | PC` rows are not touched. `POST /api/homelabs` continues to accept the v0.3.0 payload shape; a `NETWORK_DEVICE` create requires the new fields instead of the agent ones (Zod discriminated union on `resourceType`).

#### F2. SNMP collection

1. **Library** ‚Äî `net-snmp` (npm) for polling. Chosen over `node-net-snmp` variants because it's actively maintained, supports SNMPv3 auth+priv without native deps, and handles 64-bit counters (`ifHCInOctets` / `ifHCOutOctets`) natively ‚Äî critical because 32-bit counters wrap in seconds on modern gigabit links.
2. **Scheduler** ‚Äî `apps/hub-server/src/network/snmpScheduler.ts`. Ticks every 1 s, picks up `NETWORK_DEVICE` rows whose next poll is due (`lastPolledAt + snmpPollIntervalSec`), enqueues an `snmp-poll` job. Mirrors the v0.3.0 uptime-scheduler pattern for consistency ‚Äî same lease model, same `setInterval(...).unref()` hygiene.
3. **Job kind ‚Äî `snmp-poll`** ‚Äî runs the full poll: `sysDescr.0`, `sysUpTime.0`, `sysName.0`, `sysObjectID.0`, walks the IF-MIB (`ifDescr`, `ifType`, `ifOperStatus`, `ifAdminStatus`, `ifSpeed` / `ifHighSpeed`, `ifHCInOctets`, `ifHCOutOctets`, `ifInErrors`, `ifOutErrors`), and conditionally walks vendor MIBs based on the `sysObjectID` OID prefix:
   - Cisco (`1.3.6.1.4.1.9`) ‚Üí `CISCO-PROCESS-MIB` for CPU, `CISCO-MEMORY-POOL-MIB` for memory.
   - MikroTik (`1.3.6.1.4.1.14988`) ‚Üí `MIKROTIK-MIB::mtxrHlProcessorTemperature`, `mtxrHlTotalMemory`, `mtxrHlUsedMemory`.
   - UniFi / Ubiquiti (`1.3.6.1.4.1.41112`) ‚Üí UniFi OIDs for CPU / memory / channel utilisation (APs).
   - Synology (`1.3.6.1.4.1.6574`) ‚Üí `SYNOLOGY-SYSTEM-MIB` CPU + mem + temperature.
   - QNAP (`1.3.6.1.4.1.24681`) ‚Üí NAS-MIB CPU + mem + temperature.
   - Unknown vendor ‚Üí skip, record only the standard MIB-II subset.
4. **Throughput computation** ‚Äî bytes/sec per interface is **computed on write**, not at read time. Each poll persists the raw 64-bit counters plus a computed `bpsIn` / `bpsOut` derived from `(current - previous) / elapsedSec`. Counter resets (value decreasing, or uptime reset detected) clamp to `null` for that one sample. Reasoning: sparklines stay cheap, and the counter-wrap / reboot handling logic lives in one place.
5. **Persistence** ‚Äî two new tables, both on the v0.3.0 retention pruner schedule:
   - `NetworkSnapshot` ‚Äî per-device, per-poll: `labId`, `polledAt`, `sysUpTimeSec`, `sysDescr`, `cpuPercent?`, `memUsedBytes?`, `memTotalBytes?`, `tempCelsius?`, `rawVendor JSON` (vendor MIB dump, size-capped 16 KB).
   - `InterfaceSnapshot` ‚Äî per-device, per-interface, per-poll: `labId`, `polledAt`, `ifIndex`, `ifDescr`, `ifAlias?`, `ifType`, `ifOperStatus`, `ifAdminStatus`, `speedBps` (from `ifHighSpeed * 1_000_000` when present, else `ifSpeed`), `inOctets BigInt`, `outOctets BigInt`, `inErrors BigInt`, `outErrors BigInt`, `bpsIn Float?`, `bpsOut Float?`.
6. **Broadcast** ‚Äî `lab:network-snapshot` Socket.IO event fires to `lab:<labId>` room on every successful poll; payload is the full latest snapshot. Mirrors the v0.3.0 `lab:containers` broadcast pattern. Room ownership check is unchanged from v0.2.0.
7. **Error handling** ‚Äî timeouts, auth failures, and MIB-parse errors are rate-limited-logged (max 1 entry per `(labId, errorKind)` per 15 min), never throw up to the job runner; the job itself always returns "succeeded" so the retry backoff isn't triggered by a persistently-unreachable device. A contiguous sequence of 3 failed polls fires a `NETWORK_HEALTH` alert with `component='device'`.

#### F3. Network device UI

1. **Detail page** ‚Äî new route `/labs/:labId` already exists; when the lab's `resourceType === NETWORK_DEVICE` the page renders a different tab set: **Overview**, **Interfaces**, **Alerts**, **Configuration**. The existing Monitor / Services / Alerts tabs are hidden (they reference agent-only concepts).
2. **Overview tab** ‚Äî sysName, sysDescr, uptime, device kind, last-poll timestamp, vendor CPU% / mem% / temperature when available, and a small "N interfaces ¬∑ M up ¬∑ K errored" summary line.
3. **Interfaces tab** ‚Äî dense table with columns: `ifDescr` / `ifAlias`, oper/admin status badges, link speed, last-sample bps-in / bps-out, utilisation% (bps / speedBps), error counters (1h delta), sparkline thumbnail. Row click opens a drawer with 1h / 6h / 24h throughput sparklines (in + out on the same axis, different colours). Default filter hides interfaces with `ifType == 24` (loopback) and `ifOperStatus == down` unless "Show all" is toggled.
4. **Configuration tab** ‚Äî edit hostname, port, poll interval, SNMP version, credentials. "Test now" button fires a one-shot poll and returns the result inline (same UX shape as the v0.3.0 "Test now" for uptime checks).
5. **New alert metric ‚Äî `INTERFACE_UTILISATION`** ‚Äî fires when `bpsIn / speedBps` or `bpsOut / speedBps` exceeds a configured threshold (default 90%) for N consecutive polls (default 3). Threshold + N live on `HomeLab.alertThresholds` under a new `networkDevice: { interfaceUtilisationPercent, consecutiveBreaches }` subkey ‚Äî piggybacks the existing `alertThresholds` JSON column rather than adding a column.
6. **Capability-gated visibility** ‚Äî if the hub is missing its `snmp` capability (see S1), the "New Network Device" option is hidden from the create wizard and the `NETWORK_DEVICE` branch of the detail page renders a placeholder ("SNMP collection isn't available on this hub ‚Äî see Help Center ‚Ä∫ Network Devices"). Existing rows stay in the database; no destructive behaviour.

#### F4. Discovery wizard

1. **Entry point** ‚Äî `/network` page gains a "Discover devices" button (see F6). Not surfaced on the existing resource-list page ‚Äî subnet discovery is a fundamentally network-layer operation and belongs on the network surface.
2. **Inputs** ‚Äî CIDR (default filled from the hub's detected default-gateway subnet when it has the `arp` capability; otherwise blank), default SNMP version, default community string (v2c) or v3 credentials, SNMP port (default 161), scan timeout per host (default 800 ms).
3. **Safety** ‚Äî CIDR prefix length is clamped `>= /22` (max 1024 hosts/scan) unless an env flag `DISCOVERY_LARGE_SUBNETS=true` is set. Reasoning: a `/16` sweep with a 1 s timeout is a 17 min scan that will concern a non-trivial fraction of corporate networks and pile on the job runner; homelab subnets are typically `/24`.
4. **Scan mechanics** ‚Äî parallel `sysDescr.0` SNMP GETs, bounded concurrency (default 32, env-flag `SNMP_DISCOVERY_CONCURRENCY`). Results classified: **answered** (SNMP reply ‚Üí offer for one-click add, vendor guessed from `sysObjectID`), **unreachable** (no SNMP reply ‚Äî omitted from UI), **auth-failure** (reply but wrong credential ‚Äî shown with a "credential mismatch" hint).
5. **One-click add** ‚Äî each discovered device shows its guessed name, guessed kind, sysDescr, and a pre-filled "Add as Network Device" button. Clicking creates a `HomeLab` row with `resourceType=NETWORK_DEVICE` using the scan's credentials; the row's individual credential is encrypted afresh (the scan's credentials are not cached server-side beyond the duration of the scan).
6. **Storage** ‚Äî scan results are **not persisted**. The wizard is a stateful UI flow only; walking away and coming back starts a new scan.

#### F5. Synthetic network health

1. **Not a resource** ‚Äî synthetic probes are user-scoped, not lab-scoped. Every user gets one `NetworkHealthConfig` row on first use (lazy-created) with sensible defaults. Reasoning: speedtest is a bandwidth-cost operation shared across the whole hub; attaching it to a specific lab muddies the alert story and creates "do I pay for speedtest per NAS?" confusion.
2. **Probe types**
   - `SPEEDTEST` ‚Äî public-internet bandwidth test. Invokes the Ookla `speedtest` CLI (preferred ‚Äî better accuracy, accepts license-prompt suppression via `--accept-license --accept-gdpr`) or `librespeed-cli` if only the latter is present. Hub capability flag `speedtest` (see S1) gates availability. Default schedule: every 6 h. Records `downloadMbps`, `uploadMbps`, `pingMs`, `jitterMs`, `server`.
   - `GATEWAY_LATENCY` ‚Äî continuous ICMP every 30 s to each detected default gateway. Uses the same `net-ping` library v0.3.0 introduced for ICMP uptime checks; reuses the existing `NET_RAW` cap documentation. Records `targetIp`, `rttMs`, `lossPercent` (rolling over last 10 probes).
   - `DNS_HEALTH` ‚Äî A/AAAA lookup every 60 s for a user-configured name set against each of their configured resolvers. Records `name`, `resolverIp`, `rttMs`, `answerCount`, `errorKind?`. Uses Node's `dns.promises` with an explicit `setServers()` per resolver so results are per-resolver, not per-system-default.
3. **Schedule & job kind** ‚Äî each probe type registers a job kind on the v0.3.0 job runner: `network-probe-speedtest`, `network-probe-gateway`, `network-probe-dns`. Each has its own `NetworkHealthConfig` entry with `enabled: bool` and `intervalSec: int`.
4. **Speedtest rate limit** ‚Äî hard-capped at 1 run per hour per hub, irrespective of scheduling config. Default schedule is 6 h. Attempts to run sooner than 1 h after the last run are silently deferred (logged at debug). This is a hub-level policy, not a per-user one, because speedtest cost is paid by the hub's link.
5. **Storage ‚Äî `NetworkHealthSample`** ‚Äî one row per probe execution. Fields: `ownerId`, `probeKind` (`SPEEDTEST | GATEWAY_LATENCY | DNS_HEALTH`), `executedAt`, `status` (`ok | degraded | failed`), `target?` (gateway IP / DNS name / resolver IP / speedtest server), `metricJson` (probe-specific payload ‚Äî `downloadMbps` / `rttMs` / `lossPercent` / etc.), `errorMessage?`. Retention pruner extension covers this table on the same 6h/30d schedule.
6. **Alerting ‚Äî new metric `NETWORK_HEALTH`** ‚Äî `Alert.metric` enum gains `NETWORK_HEALTH`. Fires when:
   - `GATEWAY_LATENCY.lossPercent > 20` sustained over 3 consecutive samples, OR
   - `GATEWAY_LATENCY.rttMs > 200` sustained over 5 consecutive samples, OR
   - `DNS_HEALTH` failure over 5 consecutive samples for a given `(name, resolver)` pair, OR
   - `SPEEDTEST.downloadMbps` falls below a user-configured floor (default unset ‚Äî no alert unless configured).
   All thresholds live on the `NetworkHealthConfig` row. Resolves on the first clean sample, per the v0.3.0 evaluator pattern.
7. **Alert route integration** ‚Äî `AlertRoute.metrics[]` (v0.3.0) now accepts `network-health` and `interface-utilisation` alongside `cpu / memory / disk / uptime`. No schema change ‚Äî it's a `String[]`. Default routes (`*` / `all` severity) include the new metrics automatically.

#### F6. `/network` dashboard surface

1. **New top-level route ‚Äî `/network`** ‚Äî sibling of `/uptime` and `/alert-channels`. Sidebar entry "Network" with a small signal-strength icon.
2. **Page layout**
   - **Synthetic health cards** ‚Äî three cards along the top (Speedtest, Gateway latency, DNS health), each showing the latest result + a 24h sparkline + an "enabled / disabled" toggle.
   - **Devices table** ‚Äî every `NETWORK_DEVICE` resource the user owns, with columns: name, device kind, hostname, last-poll timestamp, uptime, interface up/down counts, active alerts count. Row click routes to `/labs/:labId` (the existing detail route handles the `NETWORK_DEVICE` branch).
   - **Topology hint pane** ‚Äî for each discovered default gateway, lists the devices whose ARP table entry points at that gateway's MAC. Rendered as an indented list (gateway ‚Üí children), not a visual graph. Visual topology is explicitly deferred to v0.6.0+.
3. **ARP-based association** ‚Äî when a device's SNMP walk succeeds on `ipNetToMediaPhysAddress` (or the newer `ipNetToPhysicalPhysAddress`), the hub extracts the IP‚ÜîMAC pairs and persists them on `NetworkSnapshot.arpJson`. The topology-hint pane queries "which MACs are behind which gateway" by joining the latest snapshots. Devices whose SNMP agent refuses the ARP walk (common on switches with restricted views) simply don't appear in the hint ‚Äî that's fine.
4. **Empty states** ‚Äî when no network devices exist the page shows only the synthetic-health cards plus a primary "Add network device" / "Discover devices" CTA.

### In scope ‚Äî supporting track

- **S1 ‚Äî Hub capability negotiation.** The *hub* now declares its own capabilities, mirroring the v0.3.0 agent capability pattern. `GET /api/hub/capabilities` (authed) returns `{ snmp: boolean, speedtest: boolean, icmp: boolean, dns: boolean }`. The hub determines these at boot:
  - `snmp` ‚Äî the `net-snmp` module loaded without error.
  - `speedtest` ‚Äî either `speedtest` (Ookla) or `librespeed-cli` binary resolves on `$PATH` (checked via `which`-style shellout).
  - `icmp` ‚Äî `net-ping` constructs a session without `EPERM` (so the `NET_RAW` cap story from v0.3.0 doesn't regress silently).
  - `dns` ‚Äî always `true` (Node built-in).
  Individual features degrade visibly when their capability is missing: the discovery wizard, "New Network Device", and the speedtest card all render disabled-with-explanation states rather than failing opaquely.
- **S2 ‚Äî Per-device credential encryption.** Reuses `apps/hub-server/src/lib/crypto.ts` (AES-256-GCM, `CREDENTIAL_ENCRYPTION_KEY` 32-byte hex env var, both from v0.3.0). `HomeLab.snmpCredentialCiphertext` is the only new consumer. Lazy-validated: hub boots without the key; attempting to create a `NETWORK_DEVICE` row fails fast with a clear error. Plaintext credentials are never logged ‚Äî extends the v0.3.0 pino redaction list to cover `community`, `authPassword`, `privPassword` field names.
- **S3 ‚Äî Alert route integration.** No schema change; `AlertRoute.metrics[]` is a string array. Server-side validator extends the accepted value set from `{cpu, memory, disk, uptime}` to `{cpu, memory, disk, uptime, network-health, interface-utilisation}`. UI multi-select on the route edit form gains the two new entries; existing routes created before v0.5.0 are unaffected.
- **S4 ‚Äî Speedtest binary bootstrap.** Dockerfile for `hub-server` optionally installs the Ookla CLI when the `INSTALL_SPEEDTEST_CLI=true` build arg is set. Default stays `false` so the image size isn't forced on users who don't want speedtest. The Help Center documents both the build-arg path and the "bring your own binary mounted at `/usr/local/bin/speedtest`" path.
- **S5 ‚Äî Retention pruner extension.** `MetricSnapshot` + `ContainerSnapshot` + `UptimeResult` (v0.3.0) now joined by `NetworkSnapshot`, `InterfaceSnapshot`, `NetworkHealthSample`. Same 6 h cadence, same 1000-row-per-iteration batching. `InterfaceSnapshot` is the highest-volume table (per-interface √ó per-device √ó per-poll) ‚Äî gets its own query and an explicit `(labId, polledAt)` composite index on the prune path.
- **S6 ‚Äî Prometheus `/metrics` extension.** v0.3.0's endpoint gains: `overwatch_network_devices_total{kind}`, `overwatch_network_snmp_poll_duration_seconds{labId}`, `overwatch_network_snmp_poll_errors_total{labId,kind}`, `overwatch_network_health_probe_duration_seconds{probe}`, `overwatch_network_health_status{probe}` (0/1/2 for ok/degraded/failed). Everything aggregate; no user identifiers.
- **S7 ‚Äî Test coverage.** Vitest units for: `ifHCInOctets` delta calculation (including counter wrap + reboot detection), vendor-MIB dispatcher (each supported vendor + unknown fallback), SNMP credential encrypt/decrypt round-trip, discovery-wizard concurrency + CIDR clamp, speedtest rate-limit enforcement, gateway-loss rolling window, DNS probe per-resolver `setServers()` isolation, interface-utilisation alert state machine, alert route accepts the two new metric values. Target: 30+ new tests, < 2 s runtime addition.
- **S8 ‚Äî Help Center entries.** Four new topics: `network-devices`, `snmp-setup`, `network-discovery`, `network-health-probes`. Each includes "how to test" (Docker sandbox with `polinux/mikrotik`-style image for SNMP, public `speedtest.net` server, etc.) and a "common gotchas" section (v3 auth/priv mismatches, ICMP cap on bare metal, DNS resolver reachability through VPNs).

### Out of scope (deferred to v0.6.0+)

- **SNMP trap receiver** (inbound UDP 162). Would add a long-lived UDP listener + trap-to-alert mapping + bind-port privilege story. The push model is valuable but large enough to warrant its own milestone; polling covers the critical states in the meantime.
- **Active visual topology map.** The ARP-hint list in F6 is explicitly the cheap version; a drag-able graph with LLDP/CDP neighbour-table walks + layout engine + real-time update stream is its own milestone.
- **Netflow / sFlow / IPFIX ingestion.** Per-flow traffic analysis is a different data-scale problem and deserves its own storage story (probably parquet + a query engine, not PostgreSQL).
- **Wireless client tracking** (associated clients per AP, RSSI, roaming events). Vendor-MIB-heavy; belongs after traps so client-leave/join events can push.
- **DHCP lease monitoring.** Requires either pulling ISC/Kea/dnsmasq lease files from the agent or polling router-specific lease OIDs; depends on F6 shape and user demand.
- **Agent-side synthetic probes.** Running the same speedtest / gateway-latency / DNS probes from each agent would give the "inside-the-LAN-looking-out" perspective ‚Äî defer with the same rationale as the v0.3.0 agent-side-uptime-probes defer.
- **Custom OID / MIB support.** v0.5.0 ships a fixed vendor dispatcher (Cisco / MikroTik / UniFi / Synology / QNAP + MIB-II fallback). User-defined OID watchers land in v0.6.0+ so we don't prematurely design a rule language.
- **TLS / DTLS / SNMPv3 over TLS.** SNMPv3 auth+priv is sufficient for the homelab threat model; SNMP-over-TLS is niche.
- **Agent packaging (deb/rpm/brew), TimescaleDB migration, multi-user RBAC, full mTLS hub‚Üîagent, PWA, public status pages** ‚Äî all carried forward from the v0.3.0 out-of-scope list unchanged.

## User Stories

> US numbering continues from v0.4.0 (which consumed US-58 through US-75).

### Network devices (F1, F2, F3)

- **US-76** ‚Äî As a user, I can add a router / switch / AP / NAS / firewall as a **Network Device** resource without installing an agent on it, by entering its hostname, SNMP version, and credentials.
- **US-77** ‚Äî As a user, I see the device's identity (sysName / sysDescr), uptime, and ‚Äî on supported vendors ‚Äî CPU and memory utilisation on its detail page.
- **US-78** ‚Äî As a user, I see every interface on the device with its up/down state, link speed, last-sample throughput, and utilisation %. Loopback and down interfaces are hidden by default.
- **US-79** ‚Äî As a user, clicking an interface row shows 1h / 6h / 24h throughput sparklines (in + out) so I can spot the saturated uplink.
- **US-80** ‚Äî As a user, when an interface sustains over 90% of its link speed for N consecutive polls, an `INTERFACE_UTILISATION` alert fires through the same alert system as CPU/memory/disk/uptime.
- **US-81** ‚Äî As a user, if my hub doesn't have the SNMP library loaded, the network-device create option is hidden and the `/network` page tells me what's missing rather than rendering a broken state.

### Discovery (F4)

- **US-82** ‚Äî As a user, I can press "Discover devices" on the `/network` page, provide a CIDR + default SNMP creds, and get back a list of devices that answered SNMP ‚Äî each with a one-click "Add" button.
- **US-83** ‚Äî As a user, the discovery scan refuses CIDRs larger than `/22` by default so I don't accidentally sweep a corporate `/16`.

### Synthetic network health (F5)

- **US-84** ‚Äî As a user, I can see my public-internet download / upload / latency / jitter on the `/network` page, refreshed every 6 h by default.
- **US-85** ‚Äî As a user, I can see continuous latency + packet loss to my default gateway(s), refreshed every 30 s.
- **US-86** ‚Äî As a user, I can configure a list of DNS names and resolvers and see resolution success + latency per `(name, resolver)` pair.
- **US-87** ‚Äî As a user, sustained gateway packet loss (or a flapping resolver, or chronically-slow speedtest) fires a `NETWORK_HEALTH` alert through the same routes and channels as my other alerts.

### Network surface (F6)

- **US-88** ‚Äî As a user, the `/network` route gives me one place to see synthetic health + all my network devices + a best-effort topology hint showing which devices sit behind which gateway.
- **US-89** ‚Äî As a user, if my hub can't run speedtest (no binary), I see a disabled card with a clear "install the CLI or set the build arg" link to the Help Center ‚Äî not a broken widget.

### Supporting (S-track)

### Socket.IO contract changes

- **`lab:network-snapshot`** ‚Äî broadcast to `lab:<labId>` room on every successful SNMP poll. Payload is the full latest `NetworkSnapshot` + its `InterfaceSnapshot[]` (the same shape as `GET /network/latest`). Ownership check identical to v0.2.0's `lab:metrics`.

- **`network:discovery-complete`** ‚Äî terminal event, same room. Payload: `{ scanId, found: [...], auth FailureCount, unreachableCount }`.
- **`network:health-sample`** ‚Äî broadcast to `user:<userId>` (a new per-user room joined automatically on dashboard-socket handshake) on every new `NetworkHealthSample`. Lets the `/network` page update without re-polling.
- **`lab:alert`** (existing) now carries `alertMetric` values of `NETWORK_HEALTH` or `INTERFACE_UTILISATION` alongside the v0.3.0 `UPTIME` value; optional `ifIndex` for interface-utilisation alerts so the UI can deep-link.

## Architecture Changes

### Hub-server additions

- **`src/network/snmp/client.ts`** ‚Äî thin `net-snmp` wrapper. Per-device connection is transient (session per poll) to keep the scheduler stateless; v3 auth setup is cached per `(labId, credentialsHash)` for the lifetime of the process. All errors routed through `src/lib/logger.ts` with the v0.3.0 redaction filter (extended to include SNMP auth/priv fields).
- **`src/network/snmp/mibs/`** ‚Äî vendor dispatcher with one file per supported vendor: `cisco.ts`, `mikrotik.ts`, `unifi.ts`, `synology.ts`, `qnap.ts`, plus `standard.ts` for MIB-II fallback and `dispatch.ts` that picks based on `sysObjectID` OID prefix.
- **`src/network/snmpScheduler.ts`** ‚Äî 1 s tick picking up due devices (`lastSnmpPolledAt + snmpPollIntervalSec < now()`); enqueues `snmp-poll` jobs into the v0.3.0 job runner. Bounded concurrency (default 50, env flag `SNMP_POLL_CONCURRENCY`).
- **`src/network/interfaceDelta.ts`** ‚Äî counter-delta calculator. Takes `(prev InterfaceSnapshot, current InterfaceSnapshot)` and returns `{bpsIn, bpsOut}` or `null` on detected counter wrap / reboot. Pure function, heavily unit-tested.
- **`src/network/discovery.ts`** ‚Äî CIDR expansion + bounded-concurrency scan + room-scoped progress emission.
- **`src/network/health/probes/`** ‚Äî one file per probe: `speedtest.ts`, `gatewayLatency.ts`, `dnsHealth.ts`. Each exports `run(config): Promise<ProbeResult>` with a strict timeout; the speedtest wrapper detects which CLI is present at boot and caches the path.
- **`src/network/health/scheduler.ts`** ‚Äî per-user tick; enqueues the three probe job kinds respecting the 1/hour speedtest rate limit.
- **`src/alerts/evaluator.ts`** ‚Äî extended to handle `NETWORK_HEALTH` (on `NetworkHealthSample` insert) and `INTERFACE_UTILISATION` (on `InterfaceSnapshot` insert). Both reuse the v0.3.0 state-machine refactor; no new alert-lifecycle path.
- **`src/alerts/router.ts`** ‚Äî filter accepts `network-health` and `interface-utilisation` in `AlertRoute.metrics`.
- **`src/hub/capabilities.ts`** ‚Äî boot-time capability probe. Emits a structured log line listing each capability + its status, and serves `GET /api/hub/capabilities`.
- **`src/network/persist.ts`** ‚Äî on-poll handler that writes `NetworkSnapshot` + batch-inserts `InterfaceSnapshot` rows. Shares the out-of-band pattern with v0.2.0's `MetricSnapshot` persistence.
- **`src/lib/env.ts`** ‚Äî adds `SNMP_POLL_CONCURRENCY` (default 50), `SNMP_DISCOVERY_CONCURRENCY` (default 32), `DISCOVERY_LARGE_SUBNETS` (default false), `SPEEDTEST_PROVIDER` (`ookla | librespeed | auto`, default `auto`). All lazy.

## Security Considerations

1. **SNMP credentials at rest** ‚Äî AES-256-GCM via the existing v0.3.0 `CREDENTIAL_ENCRYPTION_KEY`. Covers v2c community strings and the full v3 credential tuple. Encryption happens on the request path before insert; decryption happens only in the scheduler immediately before session construction. Plaintext never touches Prisma logs.
2. **SNMP credentials in logs** ‚Äî pino redaction list extended: `community`, `authPassword`, `privPassword`, `snmpCredentials`, `credentials`. Verified by unit test.
3. **Discovery scan abuse** ‚Äî CIDR clamped to `>= /22` by default (max 1024 hosts); `DISCOVERY_LARGE_SUBNETS=true` required to override. Per-host timeout clamped to `[200, 2000] ms`. Concurrency capped at 64 regardless of env. Scan endpoint rate-limited 4/hour/user because a `/22` sweep is already 4√ó the per-user default API budget.
4. **Speedtest bandwidth cost** ‚Äî hard-capped 1/hour/hub regardless of user count or scheduling config. Third-party terms (Ookla's license/GDPR prompts) must be answered at install time via `--accept-license --accept-gdpr` flags; the Help Center documents this explicitly.
5. **ICMP privilege** ‚Äî reuses the v0.3.0 `NET_RAW` cap documentation; no new privilege surface. When the capability is absent, the `icmp` hub capability flips to `false` and the gateway-latency card becomes disabled-with-explanation.
6. **DNS probe leakage** ‚Äî DNS names in `NetworkHealthConfig.dnsHealthNames` are not secret (they're chosen names like `google.com`), but if a user lists an internal name like `vault.corp.lab` the probe will leak it to external resolvers. Help Center calls this out; no server-side policy enforcement (would be more surprising than safe).
7. **ARP data exposure** ‚Äî `NetworkSnapshot.arpJson` is owner-scoped via the standard `HomeLab.ownerId` relation, same as every other monitored resource. Never broadcast to any room other than `lab:<labId>`.
8. **Credential testing endpoint** ‚Äî `POST /api/homelabs/:labId/snmp/test` runs a single SNMP GET, not a walk; chosen deliberately to prevent using the endpoint as an amplification oracle against arbitrary hosts.
9. **No inbound UDP surface.** SNMP trap receiver is out of scope; the hub opens no new inbound ports. All SNMP traffic is hub-initiated outbound.

| Requirement | v0.5.0 target |
|---|---|
| SNMP poll duration per device | p95 < 2 s for up to 48 interfaces |

## Known Limitations (v0.5.0)

1. **Polling only ‚Äî no traps.** State changes between poll intervals are invisible until the next tick. A link flap faster than `snmpPollIntervalSec` can go unnoticed.
2. **Fixed vendor dispatcher.** Cisco / MikroTik / UniFi / Synology / QNAP are first-class; everything else gets MIB-II only. User-defined OIDs are v0.6.0+.
3. **Topology is a hint, not a graph.** ARP-based gateway-child grouping; no LLDP/CDP neighbour walks, no visual layout.
4. **Synthetic health runs from the hub only.** Agent-side probes for "inside-the-LAN" perspective are deferred.
5. **Speedtest rate cap is hub-global.** Multi-user hubs see exactly one speedtest/hour shared across users. Acceptable for homelab; awkward for shared-hosted deployments.
6. **InterfaceSnapshot is single-tier retention.** No 1m ‚Üí 5m rollup for throughput sparklines; `retentionDays` applies uniformly. Will strain storage on high-port-count switches with long retention.
7. **Discovery results are ephemeral.** No "resume scan" or "rescan last CIDR" convenience; each scan is a fresh flow.
8. **No user-editable alert templates for the new metrics.** `NETWORK_HEALTH` and `INTERFACE_UTILISATION` use ship-default templates through the v0.3.0 channel modules, same as v0.3.0's new metrics did.
9. **Ookla CLI licence prompt.** The Ookla binary requires accepting its license/GDPR terms; unattended install via Dockerfile passes the flags, but a bring-your-own mount without those flags will simply hang the first run. Help Center documents; no server-side retry-with-accept.

## Rollback & Compatibility

- All schema changes additive. Existing monitoring resources, uptime checks, alerts, channels, and routes keep working.
- **ResourceType compatibility:** existing `HOMELAB | SERVER | PC` rows unaffected. The new `NETWORK_DEVICE` enum value is optional in every query; enum extensions are safe in PostgreSQL without a table rewrite.
- **AlertMetric compatibility:** existing alerts with `metric IN (cpu, memory, disk, UPTIME)` unaffected. Routes using the `*` wildcard automatically include the new metrics; routes with explicit `metrics[]` filters keep their filter set verbatim (existing routes don't retroactively gain new metrics ‚Äî adding them is an explicit user action).
- **Agent compatibility:** v0.2.0 through v0.4.0 agents continue to connect without any v0.5.0-related change. Network devices don't run an agent.
- **Capability compatibility:** a hub lacking `net-snmp` / speedtest binary / ICMP cap boots fine, serves its capability matrix, and hides the affected UI affordances. No forced dependency on any of the new libraries/binaries.
- **Socket.IO compatibility:** all new events are additive; pre-v0.5.0 dashboards ignore them. `lab:alert` gains an optional `ifIndex` field that older clients safely ignore.
- Rolling the hub back to v0.4.0 requires: stop any pending SNMP jobs, drop the new tables (or leave them ‚Äî v0.4.0 simply won't touch them), revert env vars. No destructive cross-release data reshape.

## Open questions for owner review

1. **Storage decision: extend `HomeLab` vs parallel `NetworkDevice` table.** Proposing the `HomeLab`-extension route because it reuses every existing ownership/label/alert scaffold for free; the cost is that agent-related columns sit unused on `NETWORK_DEVICE` rows. Parallel-table camp wants cleaner schemas at the cost of duplicating those scaffolds. Confirm?

2. **SNMPv3 default auth/priv protocols.** Proposing `SHA` auth + `AES` (128) priv as the defaults in the create wizard ‚Äî MD5 and DES are weak but still dominant in older gear.
