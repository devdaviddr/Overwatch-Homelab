# Overwatch Homelab — Product Specification v0.1.1

## Overview

v0.1.1 is a minor UX and app configuration release focused on five coordinated changes:

1. Collapsible sidebar with a thin variant (icon-only) for space-efficient navigation.
2. Make the resource hero icon visually more square and consistent across list and detail views.
3. Render resource label(s) on the same line as the description in the resource header/card.
4. Include agent configuration fields as part of the **New Resource** wizard so operator onboarding is a single flow.
5. **macOS zero-intervention agent launch** — the wizard detects macOS, locks native mode, and automatically starts the lab-agent process after resource creation. A new hub-server API manages the agent lifecycle (start/stop/status), and the Configuration tab exposes Start/Stop controls.

This release is intentionally small and scoped so the UX changes are well-specified and reversible.

---

## Goals

- Provide a compact sidebar mode so power users can reclaim screen real estate while keeping navigation discoverable.
- Improve visual consistency by standardising icon aspect ratio to a square container.
- Reduce friction: labels should not wrap to a separate line when space permits; group descriptive metadata inline.
- Streamline onboarding: allow creating a resource and configuring the agent (HUB_URL, heartbeat, metrics interval) in one wizard.
- **Zero-intervention agent startup on macOS**: when the hub is running natively the user should be able to go from wizard to live metrics without touching a terminal.

## UX Details & Acceptance Criteria

### 1) Collapsible Sidebar (Expanded <-> Thin)

Behavior:
- Sidebar has two modes: Expanded and Thin (collapsed).
- Expanded width: 240px (default). Thin width: 72px (icon-only).
- Toggle control: a chevron button at the top of the sidebar toggles modes and uses `aria-expanded`.
- Hovering a sidebar item in Thin mode shows a tooltip with Resource name + type + up to 2 labels.
- Collapse state persists per-browser using `localStorage` key: `ui.sidebarCollapsed` (value `"true"|"false"`).
- Transition: smooth width transition (`transition-width 160ms ease-out`) and fade/translate for label text.

Visuals:
- In Thin mode, only the heroicon and an optional small status dot (online/offline) are visible; text labels are hidden.
- Provide a small expand affordance (chevron) visible in both states.

Accessibility:
- Toggle button: role="button" aria-pressed, aria-expanded
- Tooltip uses role="tooltip" and appears on focus as well as hover.

Acceptance criteria:
- Toggling persists on page reload.
- Keyboard users can toggle with Enter/Space and traverse items with Tab.
- Tooltips show on focus and hover.

Files changed:
- `apps/hub-client/src/components/Sidebar.tsx`


### 2) Square hero icon

Behavior:
- All hero icons use a square container (recommended: 40x40 px) with `rounded-md` border radius.
- Icon container should center the heroicon and preserve consistent padding; use `flex items-center justify-center`.
- On active/hover state, show a subtle ring using brand colour and/or fill.

Suggested Tailwind classes:
- `w-10 h-10 rounded-md flex items-center justify-center border-2 border-transparent` (default)
- Active: `border-brand-400 bg-brand-950` (or `bg-brand-500/10` depending on theme)

Acceptance criteria:
- Visuals for heroicon in Overview, Sidebar, and Detail pages match and are square.

Files changed:
- `apps/hub-client/src/components/ResourceCard.tsx`
- `apps/hub-client/src/pages/HomeLabPage.tsx` (header icon)


### 3) Label inline with description

Behavior:
- Resource header (and small card views) show: Name (bold) on first line, then Description *followed by* label chips on the same line (if width permits).
- Layout: second line is `display: flex; gap: 0.5rem; align-items: center;` with `flex-wrap: wrap;` so long descriptions wrap but label chips remain adjacent to the end of the visible description when possible.

Edge cases:
- If the description is empty, labels should remain visible on second line (aligned left).

Acceptance criteria:
- Label chips appear on same line as description for standard content lengths and wrap gracefully at narrow widths.

Files changed:
- `apps/hub-client/src/pages/OverviewPage.tsx`
- `apps/hub-client/src/pages/HomeLabPage.tsx`


### 4) Agent configuration inside New Resource wizard

Behavior:
- 4-step wizard: Basic Info → Type & Labels → Agent Configuration → Review.
- Step 3 (Agent Configuration) includes:
  - **Platform selector** — macOS / Linux / Windows pill buttons, auto-detected from `navigator.platform` / `navigator.userAgent`. The detected option gets a small "detected" badge.
  - **Hub URL** — text field, default `http://localhost:3002`.
  - **Heartbeat interval (ms)** / **Metrics interval (ms)** — numeric fields with minimum validation.
  - **macOS**: Docker checkbox is hidden; replaced by an amber notice: "Docker on macOS reports VM specs — native mode is required."
  - **Linux/Windows**: Docker checkbox remains visible.
  - Live `.env` preview that updates as fields change, with a copy-to-clipboard button.

Fields & validation:
- `agentHubUrl` — optional, valid URL (default `http://localhost:3002` in preview)
- `heartbeatIntervalMs` — integer >= 1000 (default 15000)
- `metricsIntervalMs` — integer >= 5000 (default 60000)

API / Data model:
- `POST /api/homelabs` accepts optional `agentHubUrl`, `heartbeatIntervalMs`, `metricsIntervalMs`.
- `packages/shared-types` `CreateHomeLabSchema` updated to accept these optional fields.
- No DB migration required — columns already exist on `HomeLab`.

Acceptance criteria:
- Creating a resource with agent fields persists those fields and the Configuration tab shows them.
- The `.env` snippet is accurate and copyable.
- On macOS the Docker checkbox is absent; on Linux/Windows it is visible.

Files changed:
- `apps/hub-client/src/components/CreateHomeLabWizard.tsx`
- `apps/hub-client/src/hooks/useHomeLabMutations.ts`
- `packages/shared-types/src/events.ts`
- `apps/hub-server/src/routes/homelabs.ts`


### 5) macOS zero-intervention agent launch

#### Wizard auto-launch (post-creation)

When `platform === 'mac'` and the user clicks **Create Resource**:
1. The wizard creates the resource.
2. It calls `GET /api/agent/capabilities` to check if the hub can spawn processes.
3. If `canLaunch: true` (hub is native, not Docker): calls `POST /api/agent/:labId/launch` automatically.
4. Shows "Agent started — will connect shortly" in the Review step UI.
5. Calls `onCreated(labId)` regardless of launch outcome — launch failure is non-blocking.

If `canLaunch: false` (hub inside Docker), auto-launch is silently skipped. The Configuration tab provides a one-liner fallback.

#### Hub-server agent launcher API

New router mounted at `/api/agent` (auth required, rate-limited):

| Endpoint | Description |
|---|---|
| `GET /api/agent/capabilities` | `{ canLaunch: boolean, reason?: string }` — false when `/.dockerenv` exists |
| `POST /api/agent/:labId/launch` | Spawns `tsx apps/lab-agent/src/index.ts` as a child process with env vars; tracks in module-level Map |
| `GET /api/agent/:labId/launch` | `{ running: boolean, pid?: number }` |
| `DELETE /api/agent/:labId/launch` | Kills the managed process; `{ stopped: boolean }` |

Implementation details:
- Docker detection: `existsSync('/.dockerenv')`
- Agent binary: `<repo_root>/node_modules/.bin/tsx` (resolved relative to source file via `__dirname`)
- Agent entry: `<repo_root>/apps/lab-agent/src/index.ts`
- Child process: `spawn(tsx, [entry], { stdio: 'ignore', detached: false, env: { ...process.env, LAB_ID, HUB_URL, ... } })`
- On child `exit` event: removed from map automatically
- Killing an existing process for the same `labId` before re-launching

#### Configuration tab — Local Agent section

New section above "Detected Agents" in `AgentConfigPanel`, visible only when `canLaunch === true`:
- Shows running/stopped status with animated pulse dot and PID when running.
- **Start on this Mac** / **Stop** buttons with loading spinners.
- Status polled every 5 seconds via `setInterval`.
- Docker warning on the Docker snippet tab is gated behind `isMac` (not shown to Linux users).

#### Native one-liner (fallback)

The native snippet in the Configuration tab is now a single copyable command (no build step):
```
LAB_ID=<id> HUB_URL=<url> HEARTBEAT_INTERVAL_MS=<hb> METRICS_INTERVAL_MS=<m> npx tsx apps/lab-agent/src/index.ts
```

## Rollback & Compatibility

- All changes are additive: existing resources without agent fields continue to work.
- Agent launcher is stateless beyond in-process Map — restarting hub-server clears tracked processes (agents are not auto-restarted on hub restart by design in v0.1.1).
- If auto-launch causes issues it can be disabled by returning `canLaunch: false` from the capabilities endpoint without any other changes.
