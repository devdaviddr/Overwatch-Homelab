# Overwatch Homelab — Product Specification v0.1.0

## Overview

Overwatch Homelab v0.1.0 is the initial working release. It establishes the core platform: user authentication, homelab registration, storage pool tracking, and a real-time agent telemetry pipeline. The focus of this version is foundational correctness — a working end-to-end system that can be run locally via Docker Compose.

---

## Scope

### In scope

- User account management (register, login)
- HomeLab CRUD (create, read, update, delete)
- StoragePool management (create, delete) per HomeLab
- Real-time agent connection (register, heartbeat, metrics push)
- Web UI: login, overview dashboard, homelab detail, create wizard, edit/delete flows
- Local Docker Compose development environment

### Out of scope (planned for future versions)

- Multi-user roles and permissions (admin vs viewer)
- Real-time metrics display in the UI (agent metrics are received but not yet rendered)
- Disk metrics collection (placeholder, pending native library integration)
- User profile management (name/email/password changes)
- HomeLab sharing between users
- Alert thresholds and notifications
- Production deployment configuration (secrets management, TLS, container registry)
- Storage pool editing (update name/capacity)
- Pagination on list endpoints

## User Stories

### Authentication

**US-01** — As a new user, I can register an account with my name, email address, and a password (min 8 characters), so that I can access the platform.

**US-02** — As a registered user, I can log in with my email and password and receive a session token, so that I can authenticate subsequent requests.

**US-03** — As a logged-in user, I can sign out, clearing my session from the browser.

### HomeLab Management

**US-04** — As a user, I can create a new HomeLab by providing a name (required) and an optional description, so that I can register my physical lab environment.

**US-05** — As a user, I can optionally add one or more storage pools during HomeLab creation, specifying a name and total capacity (in GB or TB), so that I can track disk resources from the start.

**US-06** — As a user, I can see all my HomeLabs listed on the Overview page, with each card showing the lab name, description, and storage pool count.

**US-07** — As a user, I can click on a HomeLab card to navigate to a detail page showing aggregate capacity stats (pool count, total capacity, used capacity) and individual storage pool usage bars.

**US-08** — As a user, I can edit a HomeLab's name and description from the detail page.

**US-09** — As a user, I can delete a HomeLab from the detail page. Deletion is permanent, removes all associated storage pools, and requires explicit confirmation.

### Agent Telemetry

**US-10** — As an operator, I can run the lab-agent on a machine and have it automatically connect to the hub, register itself with a lab ID, and send heartbeats every 15 seconds.

**US-11** — As an operator, the lab-agent will automatically reconnect to the hub if the connection drops.

**US-12** — As an operator, the lab-agent collects and pushes CPU and memory metrics to the hub every 60 seconds.

## API Reference

Base URL: `/api`  
All protected endpoints require `Authorization: Bearer <token>`.

### Auth

#### `POST /api/auth/register`
Create a new user account.

Request:
```json
{ "email": "string", "name": "string", "password": "string (min 8)" }
```
Response `201`:
```json
{ "success": true, "data": { "token": "string", "user": { "id", "email", "name" } } }
```
Errors: `400 VALIDATION_ERROR`, `409 CONFLICT` (email taken)

---

#### `POST /api/auth/login`
Authenticate and receive a JWT.

Request:
```json
{ "email": "string", "password": "string" }
```
Response `200`:
```json
{ "success": true, "data": { "token": "string", "user": { "id", "email", "name" } } }
```
Errors: `400 VALIDATION_ERROR`, `401 UNAUTHORIZED`

---

### HomeLabs  *(all require authentication)*

#### `GET /api/homelabs`
List all homelabs owned by the authenticated user, including nested storage pools.

Response `200`:
```json
{ "success": true, "data": [ HomeLab ] }
```

---

#### `POST /api/homelabs`
Create a new HomeLab.

Request:
```json
{ "name": "string (min 1)", "description": "string (optional)" }
```
Response `201`:
```json
{ "success": true, "data": HomeLab }
```

---

#### `GET /api/homelabs/:id`
Get a single HomeLab with nested storage pools.

Response `200`: `{ "success": true, "data": HomeLab }`  
Errors: `404 NOT_FOUND`

---

#### `PATCH /api/homelabs/:id`
Update a HomeLab's name and/or description.

Request:
```json
{ "name": "string (optional)", "description": "string | null (optional)" }
```
Response `200`: `{ "success": true, "data": HomeLab }`  
Errors: `400 VALIDATION_ERROR`, `404 NOT_FOUND`

---

#### `DELETE /api/homelabs/:id`
Delete a HomeLab and all its storage pools (cascade).

Response `204 No Content`  
Errors: `404 NOT_FOUND`

---

### Storage Pools  *(all require authentication)*

#### `GET /api/homelabs/:homeLabId/storage-pools`
List all storage pools for a HomeLab.

Response `200`: `{ "success": true, "data": [ StoragePool ] }`  
Errors: `404 NOT_FOUND`

---

#### `POST /api/homelabs/:homeLabId/storage-pools`
Add a storage pool to a HomeLab.

Request:
```json
{ "name": "string (min 1)", "totalBytes": "integer >= 0", "usedBytes": "integer >= 0 (default 0)" }
```
Response `201`: `{ "success": true, "data": StoragePool }`  
Errors: `400 VALIDATION_ERROR`, `404 NOT_FOUND`

---

#### `DELETE /api/homelabs/:homeLabId/storage-pools/:poolId`
Remove a storage pool.

Response `204 No Content`  
Errors: `404 NOT_FOUND`

## Agent Protocol

The lab-agent communicates with `hub-server` over a persistent Socket.IO connection.

### Agent → Hub events

| Event | When | Payload |
|---|---|---|
| `agent:register` | On connect | `{ labId, agentVersion, hostname }` |
| `agent:heartbeat` | Every 15 s | `{ labId, timestamp }` |
| `agent:metrics` | Every 60 s | `LabMetrics` (see below) |

### Hub → Agent events

| Event | When | Payload |
|---|---|---|
| `hub:ack` | After register | `{ success: true, message }` |
| `hub:error` | On error | `{ success: false, message }` |

### Hub → Dashboard (broadcast)

| Event | When | Payload |
|---|---|---|
| `lab:metrics` | On each metrics push | `LabMetrics` broadcast to room `lab:<labId>` |

### LabMetrics schema

```typescript
{
  labId: string          // UUID
  timestamp: string      // ISO 8601
  cpu: {
    usagePercent: number // 0–100
    cores: number
    model: string
  }
  memory: {
    totalBytes: number
    usedBytes: number
    freeBytes: number
  }
  disks: DiskMetrics[]   // empty in v0.1.0 (placeholder)
  uptimeSeconds: number
}
```

## Environment Variables

### hub-server

| Variable | Required | Default | Description |
|---|---|---|---|
| `DATABASE_URL` | Yes | — | PostgreSQL connection string |
| `JWT_SECRET` | Yes (prod) | `dev_secret_change_me` | HS256 signing secret; throws on startup if unset in production |
| `JWT_EXPIRES_IN` | No | `7d` | JWT lifetime |
| `PORT` | No | `3001` | HTTP listen port |
| `CORS_ORIGIN` | No | `http://localhost:5173` | Allowed CORS origin |
| `NODE_ENV` | No | — | Set to `production` to enforce JWT_SECRET |

### lab-agent

| Variable | Required | Default | Description |
|---|---|---|---|
| `LAB_ID` | Yes | — | UUID identifying this lab; must match a HomeLab in the DB |
| `HUB_URL` | No | `http://localhost:3001` | WebSocket URL of hub-server |
| `HEARTBEAT_INTERVAL_MS` | No | `15000` | Heartbeat frequency |
| `METRICS_INTERVAL_MS` | No | `60000` | Metrics push frequency |

## Non-Functional Requirements

| Requirement | v0.1.0 status |
|---|---|
| Auth rate limiting | ✅ 20 req / 15 min on auth endpoints |
| API rate limiting | ✅ 120 req / min on API endpoints |
| Password security | ✅ bcrypt, cost factor 12 |
| JWT enforcement | ✅ Required in production, checked at startup |
| Resource isolation | ✅ All queries scoped to authenticated user |
| Cascade deletes | ✅ HomeLab delete cascades to StoragePool |
| Agent reconnection | ✅ Automatic, infinite retries, 5 s delay |
| Docker Compose | ✅ Full local stack with one command |

## Known Limitations (v0.1.0)

1. **Disk metrics are not collected** — `disks` is always an empty array. Integration with `systeminformation` or a similar native library is required.
2. **Agent metrics are not displayed in the UI** — The hub receives and re-broadcasts metrics but the client does not yet subscribe to or render them.
3. **No storage pool editing** — Capacity and name cannot be changed after creation; pools must be deleted and re-created.
4. **No pagination** — All list endpoints return all records. May become a performance concern with many homelabs or pools.
5. **LAB_ID must be a valid UUID** — The agent's `LAB_ID` env var must be a UUID to satisfy Zod validation, but it is not currently validated against an existing HomeLab record in the database.
6. **Dev credentials in compose** — `docker-compose.yaml` contains hardcoded database password and JWT secret suitable only for local development.
