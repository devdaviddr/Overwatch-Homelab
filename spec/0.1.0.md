# Overwatch Homelab — Product Specification v0.1.0

## Overview

Overwatch Homelab v0.1.0 is the initial working release. It establishes the core platform: user authentication, homelab registration, and a real-time agent telemetry pipeline. The focus of this version is foundational correctness — a working end-to-end system that can be run locally via Docker Compose, with real system metrics collected natively from monitored machines and displayed live in the browser.

---

## Scope

### In scope

- User account management (register, login)
- HomeLab CRUD (create, read, update, delete)
- Real-time agent connection (register, heartbeat, metrics push)
- Per-homelab agent configuration (hub URL, heartbeat interval, metrics interval) from the UI
- Agent setup panel: tabbed snippets for native run (macOS/Linux) and Docker run (Linux), with copyable `.env` block and pre-filled env vars
- Live metrics dashboard: CPU, memory, disks, network interfaces, OS info, uptime
- Web UI: login, overview dashboard, homelab detail, create wizard, edit/delete flows
- Local Docker Compose development environment

### Out of scope (planned for future versions)

- Historical metrics storage and time-series graphs
- Multi-user roles and permissions (admin vs viewer)
- User profile management (name/email/password changes)
- HomeLab sharing between users
- Alert thresholds and notifications
- Production deployment configuration (secrets management, TLS, container registry)
- Pagination on list endpoints

## User Stories

### Authentication

**US-01** — As a new user, I can register an account with my name, email address, and a password (min 8 characters), so that I can access the platform.

**US-02** — As a registered user, I can log in with my email and password and receive a session token, so that I can authenticate subsequent requests.

**US-03** — As a logged-in user, I can sign out, clearing my session from the browser.

### HomeLab Management

**US-04** — As a user, I can create a new HomeLab by providing a name (required) and an optional description, so that I can register my physical lab environment.

**US-05** — As a user, I can see all my HomeLabs listed on the Overview page, with each card showing the lab name and description.

**US-06** — As a user, I can click on a HomeLab card to navigate to a detail page showing agent status, creation date, metrics interval, the agent configuration panel, and the live metrics dashboard.

**US-07** — As a user, I can edit a HomeLab's name and description from the detail page.

**US-08** — As a user, I can delete a HomeLab from the detail page. Deletion is permanent and requires explicit confirmation.

### Agent Telemetry

**US-09** — As an operator, I can run the lab-agent on a machine and have it automatically connect to the hub, register itself with a lab ID, and send heartbeats every 15 seconds.

**US-10** — As an operator, the lab-agent will automatically reconnect to the hub if the connection drops.

**US-11** — As an operator, the lab-agent collects and pushes full system metrics to the hub every 60 seconds (and immediately on first connect), including CPU usage and info, memory (with swap), disk filesystems, network interfaces, OS info, and uptime.

**US-12** — As an operator, I can view and configure the agent connection settings for each HomeLab from the UI, including the hub URL and polling intervals, so that I can connect an agent on any machine without editing config files manually.

**US-13** — As an operator, the HomeLab detail page provides ready-to-use agent setup snippets:
  - **Native** tab (default): `.env` file + `npm run build && npm start` — works on macOS and Linux with real host hardware access
  - **Docker** tab: `docker run --pid=host --privileged` command — Linux hosts only, with a warning that Docker on macOS reports VM specs rather than real host hardware

**US-14** — As a user, the HomeLab detail page shows a live metrics dashboard that updates every time the agent pushes new data, displaying CPU load, memory usage, filesystem usage, and network interface state.

## API Reference

Base URL: `/api`  
All protected endpoints require `Authorization: Bearer <token>`.

### Auth

#### `POST /api/auth/register`
Create a new user account.

Request:
```json
{ "email": "string", "name": "string", "password": "string (min 8)" }
```
Response `201`:
```json
{ "success": true, "data": { "token": "string", "user": { "id", "email", "name" } } }
```
Errors: `400 VALIDATION_ERROR`, `409 CONFLICT` (email taken)

---

#### `POST /api/auth/login`
Authenticate and receive a JWT.

Request:
```json
{ "email": "string", "password": "string" }
```
Response `200`:
```json
{ "success": true, "data": { "token": "string", "user": { "id", "email", "name" } } }
```
Errors: `400 VALIDATION_ERROR`, `401 UNAUTHORIZED`

---

### HomeLabs  *(all require authentication)*

#### `GET /api/homelabs`
List all homelabs owned by the authenticated user.

Response `200`:
```json
{ "success": true, "data": [ HomeLab ] }
```

---

#### `POST /api/homelabs`
Create a new HomeLab.

Request:
```json
{ "name": "string (min 1)", "description": "string (optional)" }
```
Response `201`:
```json
{ "success": true, "data": HomeLab }
```

---

#### `GET /api/homelabs/:id`
Get a single HomeLab by ID.

Response `200`: `{ "success": true, "data": HomeLab }`  
Errors: `404 NOT_FOUND`

---

#### `PATCH /api/homelabs/:id`
Update a HomeLab's name, description, and/or agent configuration.

Request:
```json
{
  "name": "string (optional)",
  "description": "string | null (optional)",
  "agentHubUrl": "string (URL) | null (optional)",
  "heartbeatIntervalMs": "integer >= 1000 (optional)",
  "metricsIntervalMs": "integer >= 5000 (optional)"
}
```
Response `200`: `{ "success": true, "data": HomeLab }`  
Errors: `400 VALIDATION_ERROR`, `404 NOT_FOUND`

---

#### `DELETE /api/homelabs/:id`
Delete a HomeLab (permanent).

Response `204 No Content`  
Errors: `404 NOT_FOUND`

## Agent Protocol

The lab-agent communicates with `hub-server` over a persistent Socket.IO connection.

### Agent → Hub events

| Event | When | Payload |
|---|---|---|
| `agent:register` | On connect | `{ labId, agentVersion }` |
| `agent:heartbeat` | Every 15 s | `{ labId, timestamp }` |
| `agent:metrics` | Immediately after register ACK, then every 60 s | `LabMetrics` |

### Hub → Agent events

| Event | When | Payload |
|---|---|---|
| `hub:ack` | After register | `{ success: true, message }` |
| `hub:error` | On error | `{ success: false, message }` |

### Dashboard → Hub events

| Event | When | Payload |
|---|---|---|
| `dashboard:subscribe` | On page mount | `{ labId: string }` |

The browser emits `dashboard:subscribe` when the HomeLab detail page loads. Hub-server adds the socket to room `lab:<labId>`, which causes it to receive all subsequent `lab:metrics` broadcasts for that lab.

### Hub → Dashboard (broadcast)

| Event | When | Payload |
|---|---|---|
| `lab:metrics` | On each agent metrics push | `LabMetrics` broadcast to room `lab:<labId>` |

### LabMetrics schema

```typescript
{
  labId: string          // UUID matching a HomeLab
  timestamp: string      // ISO 8601
  uptimeSeconds: number
  os: {
    platform: string     // "linux" | "darwin" | "win32"
    distro: string       // e.g. "Ubuntu", "macOS"
    release: string
    arch: string         // e.g. "x64", "arm64"
    hostname: string
  }
  cpu: {
    manufacturer: string // e.g. "Intel", "Apple"
    brand: string        // e.g. "Core i7-9700K", "M4 Pro"
    cores: number        // logical cores
    physicalCores: number
    speedGHz: number     // base clock
    usagePercent: number // 0–100
    temperatureCelsius: number | null  // null on unsupported platforms
  }
  memory: {
    totalBytes: number
    usedBytes: number
    freeBytes: number
    availableBytes: number    // free + reclaimable cache
    swapTotalBytes: number
    swapUsedBytes: number
  }
  disks: Array<{
    fs: string           // device path, e.g. "/dev/sda1"
    type: string         // filesystem type, e.g. "ext4", "APFS"
    mountPoint: string
    totalBytes: number
    usedBytes: number
    availableBytes: number
    usePercent: number   // 0–100
  }>
  network: Array<{
    iface: string
    ip4: string
    mac: string
    operstate: string    // "up" | "down" | "unknown"
    speedMbps: number | null
  }>
}
```

Metrics are collected using the [`systeminformation`](https://www.npmjs.com/package/systeminformation) library (v5). All collection is async and runs in parallel via `Promise.all`. Only external (non-loopback) network interfaces are included.

## Environment Variables

### hub-server

| Variable | Required | Default | Description |
|---|---|---|---|
| `DATABASE_URL` | Yes | — | PostgreSQL connection string |
| `JWT_SECRET` | Yes (prod) | `dev_secret_change_me` | HS256 signing secret; throws on startup if unset in production |
| `JWT_EXPIRES_IN` | No | `7d` | JWT lifetime |
| `PORT` | No | `3001` | HTTP listen port |
| `CORS_ORIGIN` | No | `http://localhost:5173` | Allowed CORS origin |
| `NODE_ENV` | No | — | Set to `production` to enforce JWT_SECRET |

### lab-agent

| Variable | Required | Default | Description |
|---|---|---|---|
| `LAB_ID` | Yes | — | UUID of an existing HomeLab in the DB |
| `HUB_URL` | No | `http://localhost:3001` | WebSocket URL of hub-server |
| `HEARTBEAT_INTERVAL_MS` | No | `15000` | Heartbeat frequency in ms |
| `METRICS_INTERVAL_MS` | No | `60000` | Metrics push frequency in ms |

## Known Limitations (v0.1.0)

1. **Metrics are not persisted** — Each `lab:metrics` push is broadcast in real-time and discarded. There is no time-series storage; refreshing the page clears the last-seen metrics until the next push (up to 60 s wait, or immediately if agent is still connected).

2. **No pagination** — All list endpoints return all records. May become a performance concern with many homelabs.

3. **LAB_ID must be a valid UUID matching an existing HomeLab** — The agent's `LAB_ID` env var must be a UUID to pass Zod validation. If it doesn't match any HomeLab in the database, metrics are broadcast to an empty room and silently dropped. The UI shows the correct UUID in the agent config panel.

4. **Agent config changes require agent restart** — Updating `agentHubUrl` or intervals in the UI persists the change to the database, but a running agent reads these values only at startup from env vars. The operator must restart the agent process to apply new settings.

5. **Docker on macOS reports VM specs, not real hardware** — Docker Desktop on macOS runs inside a Linux VM. The lab-agent in Docker will report the VM's virtualised CPU and RAM rather than the host's real hardware (e.g., it will not show an M4 Pro or 24 GB). Run the agent natively on macOS for accurate metrics.

6. **CPU temperature may be null** — On macOS and Linux systems without `lm-sensors`, `temperatureCelsius` will be `null`.

7. **Dev credentials in compose** — `docker-compose.yaml` contains hardcoded database password and JWT secret suitable only for local development.
