# Overwatch Homelab — Product Specification v0.2.0

## Overview

v0.2.0 turns Overwatch from a live-only monitor into a platform that answers "what happened?" as well as "what is happening now?". It adds **historical metrics persistence**, **time-range dashboards**, and **threshold-based alerting**, together with a small set of **platform hardening fixes** uncovered during the pre-release review of the v0.1.x codebase.

The release has two tracks that ship together:

1. **Feature track** — historical metrics, alerts, profile management, pagination, summary cards (the original v0.2.0 scope).
2. **Hardening track** — fix the current typecheck break, close Socket.IO auth gap, add agent reconnect backoff, tighten startup config validation, and add a small automated test safety net. These are pre-requisites for shipping history/alerts with confidence.

---

## Foundation from v0.1.x

The following capabilities already ship and are **not** in scope for v0.2.0 to re-implement:

- Agent configuration fields (`agentHubUrl`, `heartbeatIntervalMs`, `metricsIntervalMs`) on `HomeLab` — v0.1.1.
- Agent launcher API (`GET /api/agent/capabilities`, `/api/agent/:labId/launch` CRUD) — v0.1.1.
- Platform-aware wizard with macOS auto-detection, native-only lock, auto-launch post-creation — v0.1.1.
- Configuration tab Local Agent section (Start/Stop + status polling) — v0.1.1.
- In-app Help Center with markdown rendering and sidebar accordion — v0.1.2.
- Collapsible sidebar (thin mode) with persistence, square hero icons, inline labels — v0.1.1.

v0.2.0 may call the existing `/api/agent/:labId/launch` status endpoint if it needs to correlate live agent state with alert evaluation, but must not change the launcher API contract.

## Scope

### In scope — feature track

- **Historical metrics persistence** — store every `agent:metrics` push in Postgres with a timestamp, creating a time-series record per resource.
- **Time-range dashboard views** — 1 h / 6 h / 24 h / 7 d selectors in the Monitor tab; line charts replace or supplement the live sparklines for historical windows.
- **Retention policy** — configurable per-resource retention (default: 30 days); automatic pruning via a scheduled job.
- **Alert thresholds** — per-resource configurable CPU %, memory %, and disk % high-watermarks; alert fires when any metric exceeds the threshold for N consecutive readings.
- **In-app alert notifications** — alert banner in the UI and a persisted alert log per resource (acknowledged / resolved states).
- **User profile management** — change display name and password from a new `/profile` page.
- **Pagination** — cursor-based pagination on `GET /api/homelabs` (50-per-page default).
- **Resource detail summary cards** — overview stats (avg CPU last 1 h, peak memory 24 h, disk used %) on the Monitor tab.

### In scope — hardening track

- **H1** Fix the recharts Tooltip formatter typing in `MetricsDashboard.tsx`.
- **H2** Make root `npm run typecheck` reliable by ensuring `packages/shared-types` is rebuilt before downstream apps typecheck (via npm workspace script ordering or TypeScript project references).
- **H3** Authenticate Socket.IO connections with JWT on handshake; authorize `dashboard:subscribe` against `HomeLab.ownerId`.
- **H4** Agent reconnect backoff — exponential (2 s → 30 s cap) with jitter; keep `reconnectionAttempts: Infinity`.
- **H5** Remove hardcoded `JWT_SECRET` from `docker-compose.yaml`; remove the `"dev_secret_change_me"` fallback and fail fast in any environment when `JWT_SECRET` is missing (use a documented dev-env placeholder in `.env.example` instead).
- **H6** Tighten password policy: `min(12)` + require at least one letter and one digit in `CreateUserSchema` and the new profile-password-change path.
- **H7** Stale-agent pruner: every 60 s, evict entries from `connectedAgents` where `lastHeartbeat > 2.5 * heartbeatIntervalMs`; emit `lab:agent-status { connected: false }` to affected rooms.
- **H8** Add a minimal automated test safety net: Vitest wired up in `hub-server` with tests for `POST /api/auth/register`, `POST /api/auth/login`, `GET /api/homelabs` (paginated), `GET /api/homelabs/:id/metrics`, and `POST /api/homelabs/:id/alerts/:alertId/acknowledge`.
- **H9** Startup env validation in `hub-server`: zod-parse `process.env` at boot; fail with a clear message if `DATABASE_URL` / `JWT_SECRET` / `CORS_ORIGIN` are missing or malformed.

### Out of scope (deferred to v0.3.0+)

- Email / webhook / PagerDuty alert delivery (in-app only for v0.2.0).
- Multi-user roles (admin vs viewer).
- HomeLab sharing between accounts.
- TLS / production deployment hardening beyond env-var hygiene.
- Mobile-responsive layout.
- Exportable reports (CSV / PDF).
- Agent process management (already delivered in v0.1.1).
- Alert min/max/percentile downsampling (averages only in v0.2.0).
- Auto-restart of managed agents on hub-server restart.

## User Stories

### Historical Metrics

- **US-15** — As a user, the dashboard retains and displays historical metric data so I can see trends over time, not just the live snapshot.
- **US-16** — As a user, I can select a time window (1 h, 6 h, 24 h, 7 d) in the Monitor tab to view CPU, memory, and disk charts over that range.
- **US-17** — As an admin, I can configure how long historical metrics are retained per resource (default 30 days) before they are automatically pruned.

### Alerts

- **US-18** — As a user, I can define CPU, memory, and disk usage thresholds for each resource. An alert fires when the metric exceeds the threshold for 3 consecutive readings.
- **US-19** — As a user, I am notified in the dashboard UI when an alert fires. The alert includes resource name, metric, current value, and threshold.
- **US-20** — As a user, I can view an alert log per resource showing all past events, their resolution time, and acknowledgement status.
- **US-21** — As a user, I can acknowledge or dismiss an active alert from the UI.

### User Profile

- **US-22** — As a logged-in user, I can update my display name from the profile page.
- **US-23** — As a logged-in user, I can change my password from the profile page (requires current password confirmation and meets the new complexity rules).

### Security & Reliability (new in this spec)

- **US-24** — As a user, live metrics for a resource are only delivered to Socket.IO clients that present a valid JWT for that resource's owner. (H3)
- **US-25** — As an operator, the lab-agent reconnects to the hub with exponential backoff after a network outage, not a constant-rate loop. (H4)
- **US-26** — As an operator, hub-server refuses to start with a clear error message when any required env var is missing or invalid. (H9)

## API Changes

### New endpoints

#### `GET /api/homelabs/:id/metrics`
Retrieve historical metrics for a resource.

Query parameters:
- `from` — ISO 8601 start datetime (default: 1 hour ago)
- `to` — ISO 8601 end datetime (default: now)
- `resolution` — `raw` | `1m` | `5m` | `1h` (default: `5m` — server-side time-bucket averaging)

Response `200`:
```json
{
  "success": true,
  "data": {
    "labId": "uuid",
    "from": "ISO8601",
    "to": "ISO8601",
    "points": [
      {
        "timestamp": "ISO8601",
        "cpuPercent": 42.1,
        "memActivePercent": 61.5,
        "diskUsedPercent": { "/": 55.2 }
      }
    ]
  }
}
```

---

#### `GET /api/homelabs/:id/alerts`
List alerts for a resource (paginated).

Query: `status` (`active` | `resolved` | `all`), `limit`, `cursor`.

---

#### `POST /api/homelabs/:id/alerts/:alertId/acknowledge`
Acknowledge an active alert. Response `204 No Content`.

---

#### `PATCH /api/homelabs/:id` — additional fields (extends v0.1.1)

v0.1.1 already supports patching `agentHubUrl`, `heartbeatIntervalMs`, `metricsIntervalMs`. v0.2.0 extends the same endpoint to also accept:

```json
{
  "alertThresholds": {
    "cpuPercent": 85,
    "memPercent": 90,
    "diskPercent": 80,
    "consecutiveBreaches": 3
  },
  "retentionDays": 30
}
```

The v0.1.1 agent-field handling remains unchanged.

---

#### `PATCH /api/auth/profile`
Update the authenticated user's name or password.

Request:
```json
{
  "name": "string (optional)",
  "currentPassword": "string (required if changing password)",
  "newPassword": "string — must satisfy v0.2.0 password policy (optional)"
}
```
Response `200`: `{ "success": true, "data": { "id", "email", "name" } }`.

Password policy (H6): minimum 12 characters, at least one letter and one digit. Applies to register + profile-change.

---

#### `GET /api/homelabs` — now paginated

Query: `limit` (default 50, max 200), `cursor` (opaque string).

Response:
```json
{
  "success": true,
  "data": {
    "items": [ /* HomeLab[] */ ],
    "nextCursor": "string|null",
    "hasMore": false
  }
}
```

Cursor encodes `(createdAt, id)` so results are stable across inserts. Backwards compat: clients that omit `limit`/`cursor` receive the first page.

---

### Socket.IO contract changes (H3)

- **Handshake**: clients must connect with `auth: { token: "<jwt>" }`. The server validates the token in `io.use()` middleware and attaches `userId` to the socket.
- `dashboard:subscribe` — server verifies `HomeLab.ownerId === socket.userId` before joining the `lab:${labId}` room; rejects with `hub:error` otherwise.
- Agent sockets (`agent:register`) remain token-less for now; this is acceptable because `labId` is a UUID acting as a capability, and moving agent auth to tokens is a v0.3.0 concern. Document this trade-off in the security section of the Help Center.

## Architecture Changes

### Hub-server additions

- **Env validator** (H9) — `src/lib/env.ts` zod-parses `process.env` at import; `DATABASE_URL`, `JWT_SECRET`, `CORS_ORIGIN` required, `PORT` / `JWT_EXPIRES_IN` optional with defaults. Throws on missing/invalid; `src/index.ts` imports it first.
- **Socket.IO auth middleware** (H3) — `io.use((socket, next) => verify(socket.handshake.auth.token))`; attaches `socket.data.userId`.
- **Stale-agent pruner** (H7) — `setInterval(60_000)` sweeps `connectedAgents` and evicts entries older than `2.5 * heartbeatIntervalMs`.
- **MetricSnapshot writer** — on each `agent:metrics` event, persist a row async (does not block the broadcast path). Errors are logged, never surface to the agent.
- **Retention pruner** — `setInterval(6 * 60 * 60_000)` deletes snapshots older than each lab's `retentionDays` in batches of 1000.
- **Alert evaluator** — after each metrics persist, query the last `consecutiveBreaches` snapshots for that lab; fire / resolve alerts based on thresholds.
- **`GET /api/homelabs/:id/metrics`** — queries `MetricSnapshot`, applies server-side time-bucket averaging for non-raw resolutions.
- **Cursor pagination** on `GET /api/homelabs` (feature-scope F6 fix).

The `/api/agent/*` process-management routes from v0.1.1 are not modified.

### Hub-client additions

- Time-range selector component (1h / 6h / 24h / 7d pill tabs).
- Historical line charts (recharts `LineChart`) for CPU, memory, disk.
- Alert badge in the Sidebar next to affected resources.
- Alert log tab on the resource detail page.
- Profile page at `/profile`.
- Socket.IO client sends `auth: { token: localStorage.token }` on connect (H3).
- Recharts Tooltip formatter typing fix (H1) — either widen the inline `formatter` signature to `(v, name, item) => ...` with `v as number` guard, or update the Tooltip's `formatter` generic.

### Lab-agent additions

- Exponential reconnect backoff with jitter (H4): `min(30_000, 2_000 * 2^n) + rand(0..500)`.
- Structured startup banner reporting `HUB_URL`, `LAB_ID`, `HEARTBEAT_INTERVAL_MS`, `METRICS_INTERVAL_MS` so operators don't need to enable verbose logs to debug config.

### Shared-types additions

- `MetricSnapshotSchema`, `AlertSchema`, `AlertThresholdsSchema`.
- `UpdateProfileSchema` (name + optional current/new password with H6 policy).
- Pagination helper types: `Paginated<T>`, `PaginationQuery`.
- **Publish workflow fix (H2)**: root `typecheck` script runs `npm run build --workspace=packages/shared-types` first (or switch to TS project references in `tsconfig.base.json`). This is what stabilises the current broken-build state.

## Known Limitations (v0.2.0)

1. **Alert delivery is in-app only** — no email, SMS, or webhook integrations.
2. **No multi-user roles** — all resources remain owner-scoped.
3. **Downsampling is averages only** — `resolution` applies simple time-bucket averaging; no min / max / percentile rollups.
4. **`rawPayload` grows large** — full `LabMetrics` blob stored; for long-running labs with large disk/network arrays, this may become a storage concern at the 30-day boundary. Compaction is v0.3.0.
5. **Agent process management is not persisted** — hub-server restart clears the managed-process map (v0.1.1 behaviour carried forward).
6. **Agent-to-hub socket is still `labId`-only authenticated** — H3 tightens only the dashboard side. Agent-side JWT is v0.3.0.
7. **Hub-client and lab-agent have no automated tests** — H8 adds hub-server tests only.

## Addendum — Shipped beyond the planned spec

The following scope landed during v0.2.0 implementation after the spec above was frozen. Everything here is implemented, tested, and live on `release/0.2.0`.

### Recovery-token-based password reset (no email verification)

Context: the spec covered auth'd password change (US-23) but not a forgot-password flow. Since v0.2.0 has no email integration, the reset flow is built around **recovery tokens** — the same pattern as TOTP backup codes.

**User stories (new):**
- **US-27** — As a new user signing up, I receive a one-time recovery token that lets me reset my password later.
- **US-28** — As a user who forgot my password, I can reset it by presenting my email + recovery token + a new password. The old recovery token is invalidated on success.
- **US-29** — As a logged-in user, I can regenerate a fresh recovery token from the profile page, invalidating the previous one immediately.

**New endpoints:**
- `POST /api/auth/register` response is now `{ token, user, recoveryToken }`. `recoveryToken` is a 64-char hex string (32 bytes of entropy) shown to the user **once**; the server stores only a bcrypt hash.
- `POST /api/auth/reset-password` — body `{ email, recoveryToken, newPassword }`. Verifies the token against the stored hash, updates the password (H6 policy), rotates the recovery token, and returns `{ token, user, recoveryToken }`. Dummy bcrypt run on unknown-user paths for timing parity. Public route (no JWT required — that's the point of reset).
- `POST /api/auth/recovery-token` — authenticated. Generates a new recovery token, invalidates the old one, returns `{ recoveryToken }`.

**Data model (additive):**

| Field on `User` | Type | Notes |
|---|---|---|
| recoveryTokenHash | String? | bcrypt hash (cost 12). Nullable so existing users pre-upgrade can still log in; first signup or first reset populates it. |

**Shared-types:** `ResetPasswordSchema` (email + 64-char hex token + password policy).

**Client:** LoginPage redesigned with three tabs — **Sign in / Sign up / Reset**. `RecoveryTokenModal` shows the token once with a copy button and an explicit "save this, shown once" warning. After signup or reset, the modal's "continue" button completes the login. Profile page has a **Regenerate recovery token** button.

### Live alert banner + in-app threshold configuration

The planned spec said "alert banner in the UI" but did not specify a live mechanism or threshold-configuration surface. Both shipped:

- **`AlertSettingsPanel`** on the Configuration tab — toggle enable/disable, CPU/memory/disk % fields, consecutive-breaches (1–60), retentionDays. Saves via `PATCH /api/homelabs/:id` (which already accepts these fields).
- **`AlertBanner`** mounts at the dashboard layout and subscribes to every owned lab's Socket.IO room. When the server emits `lab:alert` (fire) or `lab:alert-resolved`, the banner renders a dismissable toast for 10 s and invalidates the sidebar's active-alert count query. Clicking the toast routes to the lab's Alerts tab.

### Platform hardening

- **Agent `agent:register` now rejects unknown labIds** with `hub:error UNKNOWN_LAB` and disconnects the socket. Prevents log spam from stale-config agents and saves the DB from FK-violation traffic.
- **`persistMetricSnapshot` downgrades P2003** (FK violation) to a single-line warning instead of a full Prisma stack trace — covers the race where a lab is deleted while an agent is mid-push.
- **Lab-agent gated behind compose profile.** Default `docker compose up` brings only postgres + hub-server + hub-client. To include the Linux-in-Docker agent: `docker compose --profile agent up -d` with `LAB_ID` set in `.env`.
- **Root `.env` for docker-compose.** `JWT_SECRET` (and optional `LAB_ID`) loaded from `${repo_root}/.env` instead of requiring a shell export. `JWT_SECRET` uses `${JWT_SECRET:-}` so `docker compose down / logs / ps` work without interpolation errors; the hub-server's own env validator still fails fast at runtime if the secret is missing or < 16 chars.

### Testing (actual vs planned)

The spec called for supertest route tests backed by a test Postgres. We deferred the DB-backed tests (per the spec's "minimal automated test safety net" scope) and instead extracted pure logic into dedicated modules so it's unit-testable without a database. Final count: **47 Vitest tests across 9 files, ~1.1 s end-to-end.**

| `pagination.test.ts` | Cursor encode/decode round-trip + malformed input |

| `env.test.ts` | Fail-fast on missing/short JWT_SECRET, missing DATABASE_URL |
| `metrics.test.ts` | Alert evaluator — fire / no-fire / peak update / resolve |
| `retention.test.ts` | pruneOnce per-lab cutoff + 1000-row batching + partial-batch stop |
| `socketAuth.test.ts` | Middleware — agent / dashboard / missing / invalid / expired / valid token |
| `recoveryToken.test.ts` | Token generation entropy + hash round-trip + wrong-token rejection |
| `reset-schema.test.ts` | ResetPasswordSchema edge cases |

Deferred to v0.3.0: supertest route tests (need a CI Postgres harness), client component tests.

### Manual QA (completed 2026-04-20)

End-to-end verified against the dockerised stack:

- Register / login / weak-password rejection / rename / password change / re-login
- Cursor pagination (5 labs × 3 pages at limit=2)

## Rollback & Compatibility

- All schema changes are additive; no existing columns change semantics.
- Socket.IO JWT requirement is a breaking change for any external dashboard client (none known). The built-in hub-client is updated in the same release.
- Removing hardcoded `JWT_SECRET` and the dev fallback means running `hub-server` without `JWT_SECRET` set now fails at boot — documented in the release notes and `.env.example`.
- Feature flags not planned; v0.2.0 is shipped as a single release.
