# Overwatch Homelab — Product Specification v0.2.0

## Overview

v0.2.0 builds on the real-time telemetry foundation of v0.1.x by adding **historical metrics storage**, **time-range dashboards**, and **threshold-based alerting**. It turns Overwatch from a live-only monitor into a platform capable of answering "what happened?" as well as "what is happening now?".

---

## Foundation from v0.1.1

The following capabilities ship in v0.1.1 and are **not** in scope for v0.2.0 to re-implement:

- Agent configuration fields (`agentHubUrl`, `heartbeatIntervalMs`, `metricsIntervalMs`) on the `HomeLab` model and wizard.
- Agent launcher API (`/api/agent/capabilities`, `/api/agent/:labId/launch` CRUD) — hub-server spawns `lab-agent` natively on macOS; lifecycle tracked in-process.
- Platform-aware wizard (macOS auto-detection, native-only lock, auto-launch post-creation).
- Configuration tab Local Agent section (Start/Stop buttons, status polling).

v0.2.0 may call the existing `/api/agent/:labId/launch` status endpoint if it needs to correlate live agent state with alert evaluation, but must not change the launcher API contract.

## Scope

### In scope

- **Historical metrics persistence** — store every `agent:metrics` push in Postgres with a timestamp, creating a time-series record per resource
- **Time-range dashboard views** — 1 h / 6 h / 24 h / 7 d selectors in the Monitor tab; line charts replace or supplement the live sparklines for historical windows
- **Retention policy** — configurable per-resource retention (default: 30 days); automatic pruning of old records via a scheduled job
- **Alert thresholds** — per-resource configurable CPU %, memory %, and disk % high-watermarks; alert fires when any metric exceeds the threshold for N consecutive readings
- **In-app alert notifications** — alert banner in the UI and a persisted alert log per resource (acknowledged / resolved states)
- **User profile management** — change display name and password from the profile page
- **Pagination** — cursor-based pagination on `GET /api/homelabs` (50-per-page default)
- **Resource detail summary cards** — overview stats (avg CPU last 1 h, peak memory 24 h, disk used %) on the Monitor tab

### Out of scope (deferred to v0.3.0+)

- Email / webhook / PagerDuty alert delivery (in-app only for v0.2.0)
- Multi-user roles (admin vs viewer)
- HomeLab sharing between accounts
- TLS / production deployment hardening
- Mobile-responsive layout
- Exportable reports (CSV / PDF)
- Agent process management (already delivered in v0.1.1)

## New User Stories

### Historical Metrics

**US-15** — As a user, the dashboard retains and displays historical metric data so I can see trends over time, not just the live snapshot.

**US-16** — As a user, I can select a time window (1 h, 6 h, 24 h, 7 d) in the Monitor tab to view CPU, memory, and disk charts over that range.

**US-17** — As an admin, I can configure how long historical metrics are retained per resource (default 30 days) before they are automatically pruned.

### Alerts

**US-18** — As a user, I can define CPU, memory, and disk usage thresholds for each resource. An alert fires when the metric exceeds the threshold for 3 consecutive readings.

**US-19** — As a user, I am notified in the dashboard UI when an alert fires. The alert includes the resource name, metric, current value, and threshold.

**US-20** — As a user, I can view an alert log per resource showing all past alert events, their resolution time, and acknowledgement status.

**US-21** — As a user, I can acknowledge or dismiss an active alert from the UI.

### User Profile

**US-22** — As a logged-in user, I can update my display name from the profile page.

**US-23** — As a logged-in user, I can change my password from the profile page (requires current password confirmation).

## API Changes

### New endpoints

#### `GET /api/homelabs/:id/metrics`
Retrieve historical metrics for a resource.

Query parameters:
- `from` — ISO 8601 start datetime (default: 1 hour ago)
- `to` — ISO 8601 end datetime (default: now)
- `resolution` — `raw` | `1m` | `5m` | `1h` (default: `5m` — server-side downsampling)

Response `200`:
```json
{
  "success": true,
  "data": {
    "labId": "uuid",
    "from": "ISO8601",
    "to": "ISO8601",
    "points": [
      {
        "timestamp": "ISO8601",
        "cpuPercent": 42.1,
        "memActivePercent": 61.5,
        "diskUsedPercent": { "/": 55.2 }
      }
    ]
  }
}
```

---

#### `GET /api/homelabs/:id/alerts`
List alerts for a resource (paginated).

Query parameters: `status` (`active` | `resolved` | `all`), `limit`, `cursor`

---

#### `POST /api/homelabs/:id/alerts/:alertId/acknowledge`
Acknowledge an active alert.

Response `204 No Content`

---

#### `PATCH /api/homelabs/:id` — additional fields (extends v0.1.1)

v0.1.1 already supports patching `agentHubUrl`, `heartbeatIntervalMs`, and `metricsIntervalMs`. v0.2.0 extends the same endpoint to also accept:

```json
{
  "alertThresholds": {
    "cpuPercent": 85,
    "memPercent": 90,
    "diskPercent": 80,
    "consecutiveBreaches": 3
  },
  "retentionDays": 30
}
```

Do not remove or break the existing agent-field handling.

---

#### `PATCH /api/auth/profile`
Update the authenticated user's name or password.

Request:
```json
{
  "name": "string (optional)",
  "currentPassword": "string (required if changing password)",
  "newPassword": "string min 8 (optional)"
}
```
Response `200`: `{ "success": true, "data": { "id", "email", "name" } }`

## Data Model Changes

### New table: `MetricSnapshot`

| Field | Type | Notes |
|---|---|---|
| id | UUID | PK |
| labId | UUID | FK → HomeLab (CASCADE DELETE) |
| recordedAt | DateTime | indexed |
| cpuPercent | Float | |
| memTotalBytes | BigInt | |
| memActiveBytes | BigInt | |
| diskSnapshots | JSON | `[{ mountPoint, usedBytes, totalBytes }]` |
| rawPayload | JSON | full `LabMetrics` blob |

Index: `(labId, recordedAt DESC)` for efficient range queries.

### Updated `HomeLab` fields

| Field | Type | Notes |
|---|---|---|
| retentionDays | Int | default 30 |
| alertThresholds | JSON? | `{ cpuPercent, memPercent, diskPercent, consecutiveBreaches }` |

These are additive — all existing fields (including v0.1.1 agent fields) remain unchanged.

### New table: `Alert`

| Field | Type | Notes |
|---|---|---|
| id | UUID | PK |
| labId | UUID | FK → HomeLab |
| metric | String | `cpu` \| `memory` \| `disk` |
| threshold | Float | configured watermark |
| peakValue | Float | highest value during alert |
| firedAt | DateTime | |
| resolvedAt | DateTime? | null = still active |
| acknowledgedAt | DateTime? | |

## Architecture Changes

### Hub-server additions

- `MetricSnapshot` writer — on each `agent:metrics` event, persist a `MetricSnapshot` row to Postgres (async, non-blocking to the broadcast path)
- Retention pruner — a `setInterval` job (runs every 6 h) that deletes snapshots older than `retentionDays` per lab
- Alert evaluator — after each metrics persist, query the last N snapshots and fire/resolve alerts based on thresholds
- `GET /api/homelabs/:id/metrics` — queries `MetricSnapshot`, applies server-side downsampling for non-raw resolutions

Note: the `/api/agent/*` process-management routes are already in place from v0.1.1 — do not modify them.

### Hub-client additions

- Time-range selector component (1h / 6h / 24h / 7d pill tabs)
- Historical line charts using `recharts` `LineChart` for CPU, memory, disk over selected window
- Alert badge in the Sidebar next to affected resources
- Alert log tab on the resource detail page
- Profile page at `/profile`

## Non-Functional Requirements

| Requirement | v0.2.0 target |
|---|---|
| Metrics write latency | < 50 ms per snapshot insert (async, does not block broadcast) |
| Historical query performance | < 500 ms for 24 h range with 5 m resolution (indexed) |
| Retention pruner | Runs every 6 h; prunes old rows in batches of 1000 |
| Alert evaluation | After every metrics write; no polling |
| Pagination | Cursor-based, 50 results default, configurable up to 200 |

## Known Limitations (v0.2.0)

1. **Alert delivery is in-app only** — No email, SMS, or webhook integrations. Notifications surface only within the dashboard.
2. **No multi-user roles** — All resources remain owner-scoped; no viewer or shared-access model.
3. **Downsampling is averages only** — `resolution` parameter applies simple time-bucket averaging; no min/max or percentile rollups.
4. **rawPayload grows large** — The `rawPayload` JSON column stores the full LabMetrics blob. For long-running labs with large disk/network arrays, this may become a storage concern at the 30-day boundary.
5. **Agent process management is not persisted** — The v0.1.1 launcher tracks agents in-process only; hub-server restart clears the map. Auto-restarting managed agents on hub restart is deferred to v0.3.0+.
