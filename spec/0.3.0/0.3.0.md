# Overwatch Homelab — Product Specification v0.3.0 (DRAFT)

> **Status:** Design draft for review. Supersedes the earlier repo-analysis draft (archived at `spec/0.3.0/0.3.0-repo-analysis-archived.md`) which drifted from the platform thesis. No code yet. The choices called out as "Open questions" at the bottom need owner review before implementation starts.

## Overview

v0.3.0 pushes Overwatch past host-only metrics into the **services and reachability layer** a homelab operator actually lives in day-to-day, and gets alerts out of the browser. Three coherent features ship together under the theme **"Operate what you run."**

1. **Docker container monitoring** — the agent ships an opt-in Docker collector; the UI gains a Services tab per HomeLab showing per-container CPU/mem, state, restart count, and health-check result.
2. **Uptime checks** — users add HTTP / TCP / ICMP probes, scheduled from the hub; the dashboard shows uptime %, response-time sparklines, and an outage timeline; a new `UPTIME` alert metric flows through the existing alert machinery.
3. **Multi-channel alert delivery** — a first-class `AlertChannel` model (generic webhook / Discord / ntfy / SMTP email) plus `AlertRoute` rules so existing threshold alerts and new uptime alerts actually reach the operator when they're not looking at the tab.

The spec deliberately keeps v0.2.0 monitoring and its alert evaluator backward-compatible. Every new data model is additive; every new agent capability is negotiated at handshake so old agents keep working unchanged.

Two tracks that ship together:

1. **Feature track** — Docker collector + Services tab, Uptime checks (probes + UI + alert integration), Alert channels + routing.
2. **Supporting track** — Agent capability negotiation, maintenance windows / silences, hub-server Prometheus `/metrics`, agent token auth, client + E2E test harnesses, structured logging, Help Center entries.

The earlier repo-analysis draft for v0.3.0 is archived at `spec/0.3.0/0.3.0-repo-analysis-archived.md` and is not part of this milestone.

## Scope

### In scope — feature track

#### F1. Docker container monitoring

1. **Agent Docker collector** — new `apps/lab-agent/src/collectors/docker.ts`. Reads `/var/run/docker.sock` (Unix) or `//./pipe/docker_engine` (Windows) via the `dockerode` library. Per tick, collects for each running container: `id`, `name`, `image`, `state`, `status`, `startedAt`, `restartCount`, `cpuPercent`, `memUsageBytes`, `memLimitBytes`, `healthStatus` (when Docker health-check is defined), `labels` (filtered to `com.docker.compose.project` / `com.docker.compose.service` only).
2. **Opt-in** — driven by a new agent env var `COLLECT_DOCKER` (default `false`). When enabled at startup, the agent declares `docker: true` in its handshake capability set (S1). When the socket is unavailable or permission-denied, the agent logs once, declares `docker: false`, and continues with host metrics only.
3. **Socket payload** — `agent:metrics` gains an optional `containers: ContainerSnapshot[]` field. Payload size is capped at 200 containers per tick; overflow is logged and truncated.
4. **Hub-server persistence** — `ContainerSnapshot` rows written on the same out-of-band path as `MetricSnapshot`, keyed by `(labId, recordedAt, containerId)`. Retention uses the same `retentionDays` setting as host metrics.
5. **Services tab UI** — new tab on the HomeLab detail page (between Monitor and Alerts). Table of containers with: name, image (abbreviated), state badge, health badge, CPU%, mem%/limit, restart count, uptime. Row click opens a drawer with the last 1 h of CPU/mem sparklines for that container. Filter box (name / image / compose-project). Sticky "stopped containers (N)" collapsible section below the running list.
6. **Overview card** — the HomeLab card gains a small "N containers · M unhealthy" sub-line when the lab reports `docker: true`.

#### F2. Uptime checks

1. **New resource** — `UptimeCheck` is a first-class model, **not** a HomeLab subtype. A check can be either stand-alone (user-added, not bound to a HomeLab) or attached to a HomeLab (runs only while that lab has a connected agent, to avoid false-positive alerts during a known-down host window — see *Dependency suppression* below).
2. **Probe types**
   - `HTTP` — method (GET / HEAD / POST), URL, expected status range (default `200..399`), optional body regex, optional headers (values treated as secrets if key matches `/authorization|token|secret|key/i`), TLS verify toggle.
   - `TCP` — host, port, connect timeout.
   - `ICMP` — host, timeout; implemented via `net-ping` (requires elevated caps on Linux — the hub's compose runs `cap_add: [NET_RAW]` so it works out of the box; the spec documents the implication).
3. **Scheduling** — intervals `30s | 1m | 5m | 15m`, default `1m`. Hub-server owns the scheduler; probes run from the hub, not from agents (a homelab user wants to know "is it reachable from the outside / the LAN" — running from each host defeats the point; agent-side reachability is a future option).
4. **Storage** — `UptimeResult` table, one row per probe execution. Fields: `checkId`, `executedAt`, `status` (`up | down | degraded`), `responseTimeMs`, `httpStatus?`, `errorMessage?`. Same retention pruner as metrics. Index `(checkId, executedAt DESC)`.
5. **Alerting** — `Alert.metric` enum gains `UPTIME`. A check that fails N consecutive times (`UptimeCheck.consecutiveFailuresToAlert`, default 2) fires an alert; first successful probe afterwards resolves it. Reuses existing `Alert` table and Socket.IO events.
6. **UI — Uptime page** — new top-level route `/uptime` showing a list of checks with live status pills, uptime % over 24h/7d/30d, current response-time sparkline. Detail route `/uptime/:checkId` with the outage timeline, response-time chart, last-N results table, and a "Test now" button.
7. **UI — HomeLab attachment** — the HomeLab detail Configuration tab adds a "Linked uptime checks" panel for attaching existing stand-alone checks to the lab.
8. **Create flow** — "New Uptime Check" button on the Uptime page opens a modal: pick type → fill fields → test probe inline (single-shot run against the filled form, show result) → save. No multi-step wizard — uptime checks are lightweight.

#### F3. Multi-channel alert delivery

1. **`AlertChannel` model** — user-scoped. Types: `WEBHOOK` (generic JSON POST with HMAC signature header), `DISCORD` (Discord webhook URL shape), `NTFY` (server URL + topic + optional auth), `SMTP_EMAIL` (requires hub-level SMTP config).
2. **`AlertRoute` model** — joins a `HomeLab` (or `UptimeCheck`, or `*` for all) to an `AlertChannel` with a severity filter (`all | warning+ | critical-only`) and an optional metric filter (subset of `cpu / memory / disk / uptime`). A user can have multiple routes per channel.
3. **Delivery** — on alert fire/resolve, the existing alert evaluator enqueues a delivery job (see Supporting track S8 for the job runner — a **new, minimal** runner scoped to alert delivery only). Worker picks up, renders the channel-specific payload template, POSTs, retries up to 5 times with exponential backoff capped at 15 min, records `AlertDelivery { alertId, channelId, status, attempts, lastAttemptAt, errorMessage? }`.
4. **Templates** — each channel type has a default template for fire / resolve / acknowledge events. Templates are server-owned in v0.3.0 (not user-editable); user-editable templates land in v0.4.0+.
5. **SMTP config** — hub-level env vars (`SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`, `SMTP_FROM`). Lazy validation at the point of need (email channel creation) — the hub boots fine without them; attempting to create an SMTP channel fails fast with a clear error.
6. **UI — `/alert-channels`** — list/create/edit channels and routes. "Test channel" button fires a synthetic event through the real pipeline (so transport, template, and retry path are all exercised).
7. **Signing (webhook only)** — `X-Overwatch-Signature: sha256=<hmac>` header computed over the request body using a per-channel shared secret. Receiver can verify.

### In scope — supporting track

- **S1 — Agent capability negotiation.** `agent:register` payload grows a `capabilities: { host: boolean, docker: boolean }` field (host is always `true`; future collectors — gpu, smart, ups — slot in here without another protocol change). Hub persists the latest-seen capability set on the HomeLab row; UI hides tabs/cards whose capability is false. Unknown capability keys from newer agents are ignored gracefully.
- **S2 — Maintenance windows / silences.** `MaintenanceWindow` model: `scope: 'lab' | 'check' | 'user'`, optional `labId` / `checkId`, `startsAt`, `endsAt`, `reason`. The alert evaluator and the uptime-alert path both check for an active window before firing. Active silences also suppress delivery on resolve (so a window crossing a fire→resolve transition doesn't page with the resolve). UI: "Silence" button on alerts + a `/maintenance` page for bulk management.
- **S3 — Prometheus `/metrics` on hub-server.** Exposes: `overwatch_active_agents`, `overwatch_subscribed_dashboards`, `overwatch_alerts_active`, `overwatch_uptime_checks_up` / `_down`, `overwatch_alert_delivery_attempts_total{channel_type,status}`, `overwatch_job_runner_queue_depth`. Endpoint `GET /metrics` (unauthenticated by default — deployments put it behind network-level ACL; an env flag `METRICS_REQUIRE_AUTH=true` switches to requiring a shared `METRICS_TOKEN`).
- **S4 — Agent token auth.** The current `labId`-only handshake is preserved as a **legacy mode** (default `true`) so existing deployments keep working. A new `AGENT_AUTH=token` mode adds: the hub generates a rotatable agent token (shown once, stored bcrypt-hashed on the HomeLab row) in the Configuration tab; the agent sends `{ kind: "agent", labId, token }` on handshake; the hub verifies. Switching a lab from legacy to token mode is a per-lab setting, not global, to allow gradual migration.
- **S5 — Client test harness.** Vitest + React Testing Library for `apps/hub-client`. Minimum coverage: `AuthContext`, `useLabMetrics` hook, `AlertBanner`, new `ContainersTable`, new `UptimeStatusList`, form validation paths for the new create modals. Target: 25+ tests, < 3 s runtime.
- **S6 — Playwright E2E smoke.** One scenario covering the golden path: sign up → create a HomeLab → start the agent (mocked Socket.IO server harness, not a real agent binary) → see live metrics → create an uptime check against a test HTTP server → create a webhook channel pointed at a test sink → trigger a threshold breach → confirm alert fires in UI and webhook is delivered. Runs against the compose stack in CI (deferred — CI pipeline itself is a v0.4.0+ item).
- **S7 — Structured logging (pino) + request IDs.** Replace `console.log` with pino. Every request gets an `X-Request-Id` (generated if absent) threaded into the log context via `AsyncLocalStorage`. Socket.IO events get a `connectionId`. Redaction filter blocks fields matching `/token|secret|password|credential|authorization/i`; verified by unit test.
- **S8 — Minimal job runner.** Scoped to **alert delivery only** — DB-backed `Job` table, single in-process worker with a lease lock, statuses `queued | running | succeeded | failed`, exponential-backoff retries. Deliberately narrow in scope; future milestones can grow it into a general-purpose runner.
- **S9 — Help Center entries.** Four new topics: `containers`, `uptime-checks`, `alert-channels`, `maintenance`. Each includes a "how to test" section.
- **S10 — Test coverage for new code.** Vitest unit tests for: Docker collector payload shape (mocked dockerode), uptime scheduler tick behaviour, each probe type's success/failure classification, alert route matching logic, webhook HMAC signing, maintenance-window suppression, capability negotiation (old agent → features hidden), agent token verification path.

### Out of scope (deferred to v0.4.0+)

- **Process / systemd / listening-port health** — natural next step after container monitoring; deferred to keep F1 focused.
- **SMART / ZFS / mdadm** — requires a smartctl wrapper + privilege handling; separate agent collector.
- **GPU telemetry** (`nvidia-smi`, `rocm-smi`) — separate collector; depends on vendor tooling.
- **NUT / UPS** — separate daemon to talk to.
- **SNMP network-device discovery** — large enough to warrant its own milestone.
- **Speedtest / gateway latency / DNS health** (synthetic internet probes) — belongs with a dedicated "network observability" milestone.
- **Backup job ingestion** (restic / borg / rsnapshot exit codes) — inbound push API design needed.
- **Multi-user RBAC / resource sharing / team workspaces.**
- **TLS / mTLS between agent and hub** — S4 closes the capability-leak side of agent auth; full mTLS is broader.
- **Public / shared status pages + PWA mobile layout.**
- **Wake-on-LAN + remote restart.**
- **Kubernetes / Podman pod awareness** beyond what `dockerode` sees on a single host.
- **User-editable alert templates.**
- **Agent packaging** (deb / rpm / Homebrew formula / signed releases / auto-update).
- **TimescaleDB migration / tiered downsampling.**
- **OpenAPI spec generation** from Zod.
- **Repository analysis** (the archived v0.3.0 draft — deferred indefinitely; platform thesis is homelab/server/networking, not code).

## User Stories

### Containers (F1)

- **US-41** — As a user, I can enable Docker container collection on any agent by setting `COLLECT_DOCKER=true`. If the Docker socket isn't accessible the agent keeps running with host metrics only and tells me why in its logs.
- **US-42** — As a user, I see a **Services** tab on each HomeLab that reports containers, showing state, health, CPU%, mem/limit, and restart count. Stopped containers are grouped below running ones and collapsed by default.
- **US-43** — As a user, clicking a container row shows a 1 h sparkline of that container's CPU% and memory usage so I can spot the noisy neighbour.
- **US-44** — As a user viewing the HomeLab list, I see a "N containers · M unhealthy" sub-line on each lab that reports Docker capability.

### Uptime (F2)

- **US-45** — As a user, I can add an **Uptime Check** (HTTP, TCP, or ICMP) against any host or URL, either as a stand-alone check or linked to a HomeLab.
- **US-46** — As a user, the Uptime page shows every check's live status, 24 h / 7 d / 30 d uptime %, and a response-time sparkline.
- **US-47** — As a user, a check detail page shows the outage timeline, response-time chart, last-N probe results, and a "Test now" button.
- **US-48** — As a user, when a check fails N consecutive times an `UPTIME` alert fires through the same alert system as CPU/memory/disk breaches.
- **US-49** — As a user, I can attach a stand-alone check to a HomeLab from the Configuration tab so it inherits that lab's maintenance windows and alert routes.

### Alert delivery (F3)

- **US-50** — As a user, I can create **Alert Channels** of type Webhook / Discord / ntfy / Email and route alerts to them by lab (or check), severity, and metric.
- **US-51** — As a user, I can press "Test channel" and receive a synthetic alert through the real delivery pipeline so I know my webhook URL is correct.
- **US-52** — As a user, when an alert fires, the delivery worker retries on transient failure up to 5 times with exponential backoff and records every attempt.
- **US-53** — As a user configuring a Webhook channel, I get a signing secret I can use to verify the `X-Overwatch-Signature` header on my receiver.

### Maintenance & safety (S-track)

- **US-54** — As a user doing planned maintenance, I can silence a lab or a check for a window. Alerts fired during the window are suppressed, and the matching resolve event doesn't deliver either (no spurious "all clear" page).
- **US-55** — As an operator, I can scrape `GET /metrics` on the hub-server for Prometheus-style hub telemetry.
- **US-56** — As an operator, I can enable agent-token auth per lab — the legacy labId-only mode stays the default so existing deployments don't break.
- **US-57** — As a user, all dashboard features I see reflect what my agent actually supports — if my agent can't collect Docker, I don't see an empty Services tab.

## API Changes

### New endpoints

#### `POST /api/uptime-checks`

Create an uptime check. Body varies by type:

```json
{
  "name": "Jellyfin LAN",
  "type": "HTTP",
  "intervalSec": 60,
  "consecutiveFailuresToAlert": 2,
  "labId": null,
  "config": {
    "url": "https://jellyfin.lab.local/health",
    "method": "GET",
    "expectedStatusRange": [200, 399],
    "bodyRegex": null,
    "headers": {},
    "tlsVerify": true,
    "timeoutMs": 5000
  }
}
```

Response includes the persisted check plus the first-run probe result when `runImmediately: true` is passed (default).

#### `GET /api/uptime-checks?labId=…&status=…&cursor=…`

Cursor-paginated list. `labId` optional (omitted = all user-owned checks). `status` filter: `up | down | any` (default `any`). Response items include `currentStatus`, `lastResultAt`, `uptimePercent24h`.

#### `GET /api/uptime-checks/:checkId`

Single check detail.

#### `PATCH /api/uptime-checks/:checkId`

Update any mutable field. Changing `type` is rejected (delete + recreate).

#### `DELETE /api/uptime-checks/:checkId`

Cascades `UptimeResult` and any open alerts.

#### `POST /api/uptime-checks/:checkId/run`

Fires one synchronous probe and returns the result without persisting a historical row. Used by the "Test now" UI button. Rate-limited 10/min/user.

#### `GET /api/uptime-checks/:checkId/results?from=…&to=…&limit=…&cursor=…`

Probe history for charts and outage timeline.

---

#### `GET /api/homelabs/:labId/containers/latest`

Returns the most recent container snapshot per `containerId` for the lab. Shape:

```json
{
  "success": true,
  "data": {
    "recordedAt": "ISO8601",
    "containers": [
      {
        "containerId": "…",
        "name": "jellyfin",
        "image": "lscr.io/linuxserver/jellyfin:latest",
        "state": "running",
        "status": "Up 3 days",
        "health": "healthy",
        "restartCount": 0,
        "cpuPercent": 2.4,
        "memUsageBytes": 734003200,
        "memLimitBytes": 8589934592,
        "composeProject": "media",
        "composeService": "jellyfin"
      }
    ]
  }
}
```

#### `GET /api/homelabs/:labId/containers/:containerId/history?resolution=…&from=…&to=…`

Time-bucketed CPU% / mem% for a single container. Reuses the v0.2.0 downsample library.

---

#### `POST /api/alert-channels`

```json
{
  "name": "On-call Discord",
  "type": "DISCORD",
  "config": { "webhookUrl": "https://discord.com/api/webhooks/…" }
}
```

`config` shape is a discriminated union by `type`. Secrets in `config` are encrypted at rest with a `CREDENTIAL_ENCRYPTION_KEY` env var (32-byte hex) — introduced here, scoped to channel config only.

#### `GET /api/alert-channels`

Lists user's channels. `config` secrets are never returned.

#### `PATCH /api/alert-channels/:id` / `DELETE /api/alert-channels/:id`

Standard. Deleting a channel also deletes its routes and marks any in-flight deliveries as `cancelled`.

#### `POST /api/alert-channels/:id/test`

Enqueues a synthetic `alert:test` delivery. Returns the `deliveryId` so the UI can poll.

#### `POST /api/alert-routes`

```json
{
  "channelId": "uuid",
  "scope": { "kind": "lab", "labId": "uuid" },
  "severity": "warning+",
  "metrics": ["cpu", "memory", "disk", "uptime"]
}
```

`scope.kind` accepts `lab | check | all`. `severity` accepts `all | warning+ | critical-only`.

#### `GET /api/alert-routes` / `PATCH` / `DELETE`

Standard.

---

#### `GET /api/alerts/:alertId/deliveries`

Delivery attempts for an alert (all channels, all retries). Helpful debugging surface.

---

#### `POST /api/maintenance-windows`

```json
{
  "scope": { "kind": "lab", "labId": "uuid" },
  "startsAt": "ISO8601",
  "endsAt": "ISO8601",
  "reason": "Firmware upgrade"
}
```

`scope.kind` accepts `lab | check | user` (the last silences every alert source owned by the user).

#### `GET /api/maintenance-windows?active=true|false|all`

List. `active=true` returns only windows overlapping now.

#### `DELETE /api/maintenance-windows/:id`

Cancels early.

---

#### `GET /metrics`

Prometheus text format. Unauthenticated by default; `METRICS_REQUIRE_AUTH=true` + `METRICS_TOKEN` env pair flips to bearer-token-required. Never returns user data — only aggregate counters and gauges.

### Socket.IO contract changes

- **`agent:register`** payload gains `capabilities: { host: true, docker?: boolean }` and (when token auth is enabled on this lab) requires `token`. Backwards compatible: absent `capabilities` is treated as `{ host: true }`.
- **`agent:metrics`** payload gains optional `containers?: ContainerSnapshot[]`. Capped at 200.
- **`lab:containers`** — new broadcast to `lab:<labId>` room carrying the containers slice. Lets the dashboard update the Services tab without re-querying REST. Ownership check identical to `lab:metrics`.
- **`lab:uptime-update`** — new broadcast to a new `uptime:<checkId>` room when an uptime probe completes. Payload: `{ checkId, status, responseTimeMs, executedAt }`. Subscription model mirrors `dashboard:subscribe` — `uptime:subscribe { checkId }` with ownership check.
- **`lab:alert`** (existing) gains an `alertMetric` value of `UPTIME` and optional `checkId` alongside the existing `labId`.

## Architecture Changes

### Hub-server additions

- **`src/lib/crypto.ts`** — AES-256-GCM helpers for `AlertChannel.configCiphertext`. Reads `CREDENTIAL_ENCRYPTION_KEY` (32-byte hex). Lazily validated: hub boots without it; creating an alert channel fails fast with a clear error.
- **`src/lib/env.ts`** — adds `CREDENTIAL_ENCRYPTION_KEY`, `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`, `SMTP_FROM`, `METRICS_REQUIRE_AUTH`, `METRICS_TOKEN`. All lazy.
- **`src/lib/logger.ts`** — pino logger with redaction for fields matching `/token|secret|password|credential|authorization/i`. Request-ID middleware threads `X-Request-Id` through `AsyncLocalStorage`.
- **`src/workers/alertDeliveryWorker.ts`** — single in-process worker, 2 s poll, lease-lock pattern. On startup resets any `running` jobs to `queued` (no lock-up on hub restart). Exponential backoff `min(2^attempts * 5s, 15 min)`.
- **`src/channels/`** — per-channel modules implementing a `Channel` interface: `send(payload, config): Promise<DeliveryResult>`. Files: `webhook.ts`, `discord.ts`, `ntfy.ts`, `smtp.ts`. Webhook signs with HMAC-SHA256 over the body.
- **`src/uptime/scheduler.ts`** — ticks every 1 s, picks up checks whose next due-time has passed (computed from `lastResultAt + intervalSec`), runs probes in parallel bounded by a concurrency limit (default 50), records results, updates `currentStatus`, hands off to the alert evaluator on state transitions.
- **`src/uptime/probes/`** — `http.ts`, `tcp.ts`, `icmp.ts`. Each exports `run(config): Promise<ProbeResult>` with a strict timeout.
- **`src/alerts/evaluator.ts`** — extended to handle `UPTIME` alerts. Refactor of the v0.2.0 evaluator to take an abstract "source" (lab metric stream or uptime check) + "rule" rather than hard-coding CPU/mem/disk.
- **`src/alerts/router.ts`** — on alert fire/resolve, finds matching routes (by owner, scope, severity, metric), creates an `AlertDelivery` per matched channel, enqueues a `Job`.
- **`src/maintenance/suppress.ts`** — `isSilenced(labId?, checkId?, userId, at): boolean` consulted by both evaluator and router.
- **`src/capabilities.ts`** — parses `agent:register.capabilities`, persists to `HomeLab.agentCapabilities`, exposed via `GET /api/homelabs/:id` so the UI can gate features.
- **`src/metrics/prometheus.ts`** — `/metrics` endpoint. Uses `prom-client`. Collectors wired at module load.
- **`src/containers/persist.ts`** — on-receive `agent:metrics` handler extracts `containers[]` and batch-inserts `ContainerSnapshot` rows. Shares the out-of-band pattern with v0.2.0 `MetricSnapshot` persistence.
- **Retention pruner** extended: prunes `MetricSnapshot`, `ContainerSnapshot`, `UptimeResult` on the same 6 h schedule using each lab's `retentionDays` (uptime checks inherit the owner's current v0.2.0 default of 30 days in v0.3.0 — per-check retention is deferred).

### Hub-client additions

## Security Considerations

1. **Alert-channel secrets at rest** — AES-256-GCM via `CREDENTIAL_ENCRYPTION_KEY`. Includes webhook URLs that embed tokens (Discord webhooks), ntfy auth, SMTP password, and per-channel HMAC signing secrets.
2. **Secrets in logs** — pino redaction covers the header fields above; verified by unit test.
3. **Webhook receiver verification** — `X-Overwatch-Signature: sha256=<hmac-hex>` over the raw body. Documented in Help Center with a node+python verifier snippet.
4. **Uptime probe abuse** — intervals are gated to `{30, 60, 300, 900}` seconds; `POST /uptime-checks/:id/run` rate-limited to 10/min/user. TCP/ICMP probes refuse RFC1918 ranges only when the hub is running with `UPTIME_LAN_RESTRICTED=true` (default `false`, because homelab use-case is exactly LAN probing). Public SaaS deployments flip that flag.
5. **Agent token auth (S4)** — opt-in per lab. When enabled, the labId alone is no longer sufficient; rotating the token invalidates prior agents for that lab.
6. **Container metrics exposure** — `lab:containers` broadcasts inherit the v0.2.0 ownership-scoped room model. Image tags are treated as public info; container env vars are **never** collected or transmitted.
7. **Prometheus `/metrics`** — unauthenticated by default for the common single-tenant homelab deployment, but switchable to bearer-token via `METRICS_REQUIRE_AUTH=true`. Never exposes user identifiers or check configs — only counters and gauges.
8. **Maintenance windows** — scope is owner-gated (`MaintenanceWindow.ownerId` enforced on every write). A user can't silence another user's alerts.
9. **SMTP relay abuse** — the hub only sends to addresses listed in the user's own channel config; no open-relay mode.

## Non-Functional Requirements

| Requirement | v0.3.0 target |
|---|---|
| Container collector tick overhead | < 150 ms additional per agent tick for 100 containers |
| Container payload size | Cap 200 containers/tick; overflow truncated + logged |
| Uptime scheduler drift | < 2 s drift at sustained 500 checks/min |
| HTTP probe default timeout | 5 s; max 30 s |
| TCP/ICMP probe default timeout | 3 s; max 10 s |
| Alert delivery latency (fire → first attempt) | p95 < 5 s |
| Alert delivery retry backoff | `min(2^attempts * 5s, 15 min)`; max 5 attempts |
| Prometheus scrape cost | < 50 ms per scrape with 1000 checks + 50 labs |
| Maintenance suppression check | < 1 ms per alert evaluation (in-memory index of active windows) |
| Retention pruner run time | < 30 s for a 30-day window covering all three snapshot tables |

## Known Limitations (v0.3.0)

1. **Probes run from the hub only.** Agent-side reachability probes (useful for LAN-from-inside vs WAN-from-outside perspective) are deferred.
2. **Container collector is Docker-only.** Podman socket is compatible in rootless-with-docker-socket mode but not officially supported; containerd / CRI-O / Kubernetes are out of scope.
3. **Alert templates are not user-editable.** Ship-default copy only; customisation lands in v0.4.0+.
4. **No agent packaging.** Users still run the agent as a native `node dist/index.js` or the existing Linux Docker profile.
5. **No TLS between agent and hub** beyond whatever the reverse proxy provides. S4 addresses auth, not confidentiality on the wire.
6. **Single-process alert-delivery worker.** Multi-host horizontal scaling needs Redis + BullMQ — deferred.
7. **Retention is single-tier.** No 1m→1h rollup tiering; `MetricSnapshot` + `ContainerSnapshot` + `UptimeResult` all prune on the same `retentionDays` scalar.
8. **Uptime SLO reporting is basic.** 24 h / 7 d / 30 d uptime % only — no SLO budget burn rates or weekly email summary.
9. **No multi-user RBAC.** Every model stays owner-scoped as today.

## Rollback & Compatibility

- All schema changes additive. Existing monitoring resources and alerts keep working.
- **Agent compatibility:** v0.2.0 agents connect without a `capabilities` field — hub treats that as `{ host: true }` and hides Docker UI affordances for that lab. v0.3.0 agents without `COLLECT_DOCKER=true` behave identically.
- **Auth compatibility:** `agentAuthMode` defaults to `legacy` for every existing lab; the per-lab token flip is opt-in. No forced migration.
- **Alert compatibility:** existing `Alert` rows with `metric IN (cpu, memory, disk)` are unaffected by the enum extension.
- Socket.IO event names are additive. v0.2.0 dashboards ignore `lab:containers` and `lab:uptime-update`.
- Rolling the hub back to v0.2.0 requires: stop agents with `COLLECT_DOCKER=true`, drop the new tables (or leave them — v0.2.0 simply won't touch them), revert env vars. No destructive cross-release data reshape.

## Open questions for owner review

1. **Per-lab vs per-user retention for uptime results.** Uptime checks aren't strictly tied to a lab, and 30 days of 1-probe-per-minute is ~43k rows per check. Proposal: single `UPTIME_RETENTION_DAYS` env (default 30) rather than reusing lab `retentionDays`. Confirm, or flip to per-check overrides.

2. **Probe origin: hub-only vs agent-optional.** Proposing hub-only for v0.3.0 (simpler, matches the "is it reachable?" mental model). Agent-side probes would answer "is it reachable from inside the LAN" which is valuable but doubles the surface. Defer?

3. **SMTP vs webhook-first for email.** Proposing SMTP because it's ubiquitous and doesn't require a third-party account. Alternative: ship Resend / Postmark / SendGrid API adapters. Keep SMTP-only for v0.3.0?

4. **Agent token auth default.** Proposing `legacy` default with opt-in per-lab flip. Alternative: switch default to `token` for *newly-created* labs while leaving existing labs on legacy. Safer migration story or UX churn?

5. **Containers for non-Docker runtimes.** Explicitly Docker-only for v0.3.0. Podman with the compat socket works in practice but we don't test it. Document as "works on best-effort" or stay silent?

6. **Maintenance-window scope expressiveness.** Proposing `lab | check | user`. Do you want tag-based silences too (e.g. "silence everything labelled `production-nas`")? That's a slightly larger change to the alert evaluator's filter pipeline.

7. **Prometheus `/metrics` auth default.** Proposing unauthenticated by default for the single-tenant homelab case. Flip to auth-required if the typical deployment is actually exposed beyond the LAN.

8. **Do we ship a minimal public status page in this milestone** (read-only `/status/:shareId` view of selected HomeLabs + uptime checks), or punt entirely to v0.4.0+? It's small code-wise but adds unauth routing + share-token model.

9. **v0.4.0 scope.** `spec/0.4.0/0.4.0.md` currently mirrors this spec and needs to be replaced with a forward-looking milestone (SMART/ZFS, GPU, NUT/UPS, SNMP, network probes, agent packaging, etc.). Want me to rewrite it now as a follow-on draft, or leave it empty until v0.3.0 converges?
