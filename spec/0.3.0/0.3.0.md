# Overwatch Homelab — Product Specification v0.3.0 (DRAFT)

> **Status:** Design draft for review. No code yet. The choices called out as "Open questions" at the bottom need owner review before implementation starts.

## Overview

v0.3.0 extends Overwatch beyond physical-machine monitoring to **source code repositories as a first-class resource type**. A user adds a Git URL + credential, the hub clones the repo, and a Claude-powered analyzer produces a structured findings report: architecture summary, security risks, dependency health, code quality notes. The UI surfaces these findings the same way v0.2.0 surfaces metrics — overview cards, a detail page, filterable findings log.

Two tracks that ship together:

1. **Feature track** — new `GITHUB_REPO` / `GITLAB_REPO` / `GIT_REPO` resource types, credential storage, repo cloning pipeline, Claude analyzer, findings model + UI.
2. **Supporting work** — an async job runner (analyses take minutes), secret-at-rest encryption, per-user analysis quota, docs + Help Center entries.

The spec deliberately keeps the v0.2.0 monitoring surface untouched. Adding a repo resource is purely additive — existing `HOMELAB` / `SERVER` / `PC` resources behave identically.

## Foundation from v0.2.0

Not in scope to re-implement:

- Auth (signup / signin / reset password with recovery tokens) — v0.2.0.
- Cursor pagination on `GET /api/homelabs` — v0.2.0.
- Profile page, Help Center, sidebar with alert badges — v0.2.0.
- Socket.IO JWT handshake + ownership-checked `dashboard:subscribe` — v0.2.0.
- Env validator at boot, root `.env`, rate limiting — v0.2.0.
- Vitest safety net (47 tests) — v0.2.0.

Repo resources **share** the `HomeLab` model and its `GET /api/homelabs` list endpoint, so they automatically inherit pagination, ownership scoping, sidebar rendering, and the create wizard's first step (name + type selection). They diverge on a separate detail page and a separate PATCH payload shape.

## Scope

### In scope — feature track

1. **New resource types** — extend `ResourceType` enum with `GITHUB_REPO`, `GITLAB_REPO`, `AZURE_DEVOPS_REPO`, `GIT_REPO`. (Bitbucket, Gitea deferred.)
2. **Repo credentials** — per-resource `repoUrl` + optional `repoUsername` + encrypted `repoCredential` (PAT or password). HTTPS only for v0.3.0; SSH keys deferred. Per-provider auth semantics handled transparently (see *Provider quirks* below).

#### Provider quirks — URL shapes & auth patterns

| Provider | URL shape (accepted) | Auth pattern we use against `git clone` |
|---|---|---|
| `GITHUB_REPO` | `https://github.com/{owner}/{repo}(.git)?` or GitHub Enterprise host | username = `x-access-token`, password = PAT |
| `GITLAB_REPO` | `https://gitlab.com/{path}(.git)?` or self-hosted GitLab host | username = `oauth2`, password = PAT |
| `AZURE_DEVOPS_REPO` | `https://dev.azure.com/{org}/{project}/_git/{repo}` **or** legacy `https://{org}.visualstudio.com/{project}/_git/{repo}` | username = `pat` (any non-empty string works — Azure ignores it), password = PAT. Minimum PAT scope: **Code (Read)**. |
| `GIT_REPO` | Any `https://…(.git)?` URL | username = `repoUsername` field (user-provided), password = `repoCredential` |

The hub never embeds credentials in the URL. It shells out to `git clone` with `GIT_ASKPASS` pointing at a short-lived helper script that reads the decrypted credential from stdin — the plaintext never hits the process environment, command line, or git config.

**URL validation** accepts both modern `dev.azure.com` and legacy `*.visualstudio.com` forms for Azure DevOps, normalising internally but preserving the original for display.
3. **Clone pipeline** — shallow clone (`--depth 50`) on creation; re-pull on demand from the Configuration tab or the scheduled re-analyze job. Per-repo disk cap (default 500 MB).
4. **Claude-powered analyzer** — hub-server runs an analysis job that:
   - Pre-extracts deterministic facts (file tree, LOC, primary language, package manifests, dependency list, git log stats, README).
   - Sends a structured prompt to Claude Sonnet 4.6 via the Anthropic SDK, with prompt caching on the system prompt + per-repo static context.
   - Parses the response into typed `Finding` records (severity, category, title, detail, optional filePath/lineRange).
   - Streams progress + partial findings over Socket.IO.
5. **Findings storage & UI**
   - `RepoAnalysis` table holds a full analysis (summary, stats, findings[], token usage, cost).
   - Repo detail page with three tabs: **Summary** (markdown exec summary + stats), **Findings** (severity-filtered list), **Configuration** (repo URL, credential rotate, schedule, re-analyze).
   - Overview cards show provider icon, repo URL, last-analyzed timestamp, critical-finding count.
6. **Create wizard flow branch** — Type-selection step routes repo types into a "Repo details" step (URL + credential) instead of the labels + agent step used for monitoring resources.
7. **Scheduled re-analysis** — per-repo cron-ish interval (daily default, configurable: manual only / hourly / daily / weekly).

### In scope — supporting track

- **S1 — Credential encryption at rest.** AES-256-GCM using a dedicated `CREDENTIAL_ENCRYPTION_KEY` env var (32-byte hex). Plaintext never logged, never returned in API responses.
- **S2 — Async job runner.** Analysis jobs take minutes; they must survive the request/response lifecycle. DB-backed `Job` table with statuses `queued | running | succeeded | failed | cancelled`. Single in-process worker with a lease lock. On hub-server restart, mark any `running` jobs as `failed` (user can retry).
- **S3 — Per-user analysis quota.** Env-configurable daily cap on analyses per user (default 20/day) to prevent runaway Claude API spend. Quota consumed on job start.
- **S4 — Secret scanning pre-flight.** Before sending repo contents to Claude, reject files whose content matches common secret patterns (AWS keys, private key headers, GitHub tokens, `.env` with likely secrets). Log and skip; never include in Claude request body.
- **S5 — Cost + token telemetry.** Every analysis records `inputTokens`, `outputTokens`, `cacheReadTokens`, `cacheWriteTokens`, and estimated USD. Surface in the analysis detail for transparency.
- **S6 — `ANTHROPIC_API_KEY` env var** added to the env validator (required iff at least one repo resource exists; optional otherwise). Separate from `JWT_SECRET`.
- **S7 — Help Center entry** for repo resources.
- **S8 — Test coverage** for the new code: Vitest unit tests for the encryption helper, job runner state machine, secret scanner, analyzer prompt construction, and findings parser.

### Out of scope (deferred to v0.4.0+)

- **Branch-aware analysis** (default branch only in v0.3.0).
- **PR analysis / write-back to GitHub** (comment on PRs, open issues).
- **SSH key auth** — HTTPS + PAT only for v0.3.0.
- **Bitbucket / Gitea providers.**
- **Webhook-triggered re-analysis** (GitHub push → re-run). Scheduled + manual only for v0.3.0.
- **Private registry dep analysis** (authenticated `npm install`, Python package auth, etc.).
- **Multi-repo aggregated reports** ("across all my repos, what's my dep-vulnerability surface?").
- **Findings acknowledgement workflow** similar to alerts (ack / dismiss / mark as won't-fix). Findings are read-only in v0.3.0; user re-runs analysis to get a fresh set.
- **Managed Agents / server-side agentic tool use.** v0.3.0 uses the Claude API directly with a single structured request per analysis section, not a tool-using loop.
- **Exportable reports (PDF / JSON)** — can be derived from the API later.

## User Stories

### Adding a repo

- **US-31** — As a user, I can create a new resource of type GitHub Repo, GitLab Repo, Azure DevOps Repo, or Generic Git Repo from the dashboard wizard.
- **US-32** — As a user, I provide the repo URL and a PAT / password at creation time (and a username for generic Git repos). The credential is stored encrypted; I never see it again in the UI.
- **US-32a** — As a user adding an Azure DevOps repo, I can paste the full `https://dev.azure.com/{org}/{project}/_git/{repo}` URL (or a legacy `*.visualstudio.com` URL) and a PAT with at least "Code (Read)" scope. The hub handles the Azure-specific auth pattern (PAT-as-password with any username) transparently.
- **US-33** — As a user, after I save the repo the hub immediately clones it and kicks off the first analysis. I see progress live in the dashboard.

### Viewing analysis results

- **US-34** — As a user, the repo detail page shows an executive summary of the repo (architecture, primary language, LOC, notable characteristics) in plain English.
- **US-35** — As a user, I can view a list of findings from the analysis, filterable by severity (critical / warning / info) and category (security / architecture / dependencies / quality).
- **US-36** — As a user, each finding includes a title, explanation, and — where applicable — the specific file path and line range it refers to.
- **US-37** — As a user, I can see the timestamp of the most recent analysis and a history of prior analyses for the same repo.

### Managing repo resources

- **US-38** — As a user, I can trigger a re-analysis manually at any time. If an analysis is already running for that repo, the button is disabled.
- **US-39** — As a user, I can configure how often the repo is auto-re-analyzed (manual / hourly / daily / weekly).
- **US-40** — As a user, I can rotate the stored credential from the Configuration tab (enter new value; old ciphertext replaced in one transaction).
- **US-41** — As a user, deleting a repo resource also deletes its clone directory, all analyses, and the encrypted credential.

### Safety / observability

- **US-42** — As an operator, the hub never sends file contents to Claude that match known secret patterns (AWS keys, private keys, tokens).
- **US-43** — As an operator, each analysis records its token usage and estimated cost so I can audit Claude API spend.
- **US-44** — As an operator, the hub refuses to start repo analysis if `ANTHROPIC_API_KEY` or `CREDENTIAL_ENCRYPTION_KEY` are missing — with a clear startup diagnostic.
- **US-45** — As a user, I have a per-day analysis quota. Exceeding it returns a clear `QUOTA_EXCEEDED` error; the quota resets at UTC midnight.

## API Changes

### New endpoints

#### `POST /api/homelabs` — extended for repo types

Existing endpoint. When `resourceType` is a repo type, the request body accepts:

```json
{
  "name": "my-app",
  "resourceType": "GITHUB_REPO",
  "labels": ["production"],
  "repoUrl": "https://github.com/devdaviddr/example.git",
  "repoUsername": null,
  "repoCredential": "ghp_…",
  "analysisSchedule": "daily"
}
```

`repoUsername` is optional. The server defaults it per provider:

- `GITHUB_REPO` → `x-access-token`
- `GITLAB_REPO` → `oauth2`
- `AZURE_DEVOPS_REPO` → `pat` (any non-empty string works; Azure ignores the username for PAT auth)
- `GIT_REPO` → required from the client (no safe default)

An Azure DevOps example:

```json
{
  "name": "platform-api",
  "resourceType": "AZURE_DEVOPS_REPO",
  "repoUrl": "https://dev.azure.com/contoso/Platform/_git/platform-api",
  "repoCredential": "<PAT with Code (Read) scope>",
  "analysisSchedule": "daily"
}
```

Response omits both `repoUsername` (when defaulted) and `repoCredential` entirely. A successful create queues the first analysis job and returns the HomeLab row plus `{ job: { id, status: "queued" } }`. If the initial clone fails (bad credential, bad URL, over the size cap), the job is marked failed and the HomeLab row is left in place with a `lastAnalysisError` set — the user can fix the credential via PATCH and retry.

#### `POST /api/homelabs/:id/analyze`

Manually trigger re-analysis. Returns `{ job: { id, status: "queued" } }` or `409 ANALYSIS_IN_PROGRESS` if one is already running for this repo. Enforces the per-user daily quota.

---

#### `GET /api/jobs/:jobId`

Poll a job's status. Useful as a fallback when the Socket.IO stream drops. Returns:
```json
{ "id": "uuid", "kind": "repo-analysis", "status": "running", "progress": 0.4, "startedAt": "ISO8601" }
```

---

#### `PATCH /api/homelabs/:id` — additional fields for repo types

```json
{
  "repoUrl": "https://…",
  "repoUsername": "optional-for-GIT_REPO",
  "repoCredential": "new-pat-value",
  "analysisSchedule": "weekly"
}
```

`repoUsername` and `repoCredential` are never returned on reads. Supplying `repoCredential` replaces the stored ciphertext atomically (encrypted in one transaction with any other changes). Supplying an empty string clears the credential — analysis will fail until one is set again. The server rejects `repoUrl` changes that switch providers (e.g., a GitHub URL on an AZURE_DEVOPS_REPO resource) with `400 PROVIDER_MISMATCH` — delete and recreate to switch.

---

### Socket.IO contract changes

- **`repo:job-progress`** — emitted to `lab:<labId>` room while a repo-analysis job is running. Payload: `{ jobId, stage: "cloning" | "scanning" | "analyzing" | "persisting", progress: 0..1, message?: string }`.
- **`repo:finding`** — emitted per finding as the analyzer parses them, so the UI fills the list progressively. Payload is the Finding shape above.
- **`repo:analysis-complete`** — emitted when a job finishes. Payload: `{ jobId, analysisId, findingCounts, costUsd }` on success; `{ jobId, status: "failed", error: string }` on failure.

All three events respect the v0.2.0 ownership check — only the user who owns the lab receives them.

## Data Model Changes

### Extended `HomeLab`

| Field | Type | Notes |
|---|---|---|
| repoUrl | String? | HTTPS clone URL; required when `resourceType` is a repo type |
| repoUsername | String? | Optional. Defaulted per-provider on write when null (`x-access-token` / `oauth2` / `pat`); required for `GIT_REPO` |
| repoCredentialCiphertext | Bytes? | AES-256-GCM-encrypted credential (IV + ciphertext + auth tag in a single blob) |
| analysisSchedule | `AnalysisSchedule` | `manual \| hourly \| daily \| weekly`; default `daily` |
| lastAnalyzedAt | DateTime? | Populated on first successful analysis |
| lastAnalysisCommitSha | String? | Git SHA of the last analyzed revision |
| lastAnalysisError | String? | Last clone / analysis error message (cleared on next success) |

All fields nullable — monitoring-type resources don't populate them. For repo resources, `repoUrl` is required and the provider is derived from the `resourceType` enum, not from URL sniffing — this keeps the credential-handling path unambiguous (an `AZURE_DEVOPS_REPO` resource always uses the Azure auth pattern even if the URL looks generic).

### New enum: `AnalysisSchedule`

```
manual | hourly | daily | weekly
```

```
FindingSeverity: critical | warning | info
FindingCategory: security | architecture | dependencies | quality | performance | documentation
```

All schema changes additive. No existing columns change semantics.

## Architecture Changes

### Hub-server additions

- **`src/lib/crypto.ts`** — AES-256-GCM encrypt/decrypt helpers reading `env.CREDENTIAL_ENCRYPTION_KEY`. One function per direction, returns `Buffer`. Hard-fail on bad auth tag (tampering).
- **`src/lib/env.ts`** — add `ANTHROPIC_API_KEY` (required if any repo resource exists — validated lazily on analysis start, not at boot, so monitoring-only deployments don't need it), `CREDENTIAL_ENCRYPTION_KEY` (required iff any repo resource exists). Monitoring-only users can ignore both.
- **`src/workers/jobRunner.ts`** — single in-process worker polling `Job` table every 2 s for `status = 'queued'` or `status = 'running' AND leaseExpiresAt < now()` (dead-worker cleanup). Worker sets status → `running`, refreshes `leaseExpiresAt` every 30 s while working, marks `succeeded` / `failed` at the end. On hub-server startup, any `running` jobs get reset to `failed` with a "hub restarted" error.
- **`src/analyzers/repo/clone.ts`** — thin wrapper around `simple-git` (or `child_process` to `git`). Clones to `/var/lib/overwatch/repos/<labId>`. Enforces size cap. Runs `git rev-parse HEAD` to capture commitSha.
- **`src/analyzers/repo/providers.ts`** — per-provider module. Exposes `urlValidator(resourceType, url)` (regex + URL parser accepting both `dev.azure.com/{org}/{project}/_git/{repo}` and `{org}.visualstudio.com/{project}/_git/{repo}` for Azure DevOps, and `github.com`/GitHub Enterprise, `gitlab.com`/self-hosted for the others) and `resolveGitAuth(resourceType, repoUsername, repoCredential)` which returns the `{ username, password }` pair passed to `GIT_ASKPASS`. Keeps Azure's "any username works" quirk encapsulated in one file instead of sprinkled through the analyzer pipeline.
- **`src/analyzers/repo/preprocess.ts`** — deterministic preprocessing: walks the tree, computes LOC with `cloc`-style heuristic, parses `package.json` / `go.mod` / `requirements.txt` / `Cargo.toml`, extracts README, git log stats.
- **`src/analyzers/repo/secretScan.ts`** — regex-based preflight: scans file contents for AWS keys (`AKIA[0-9A-Z]{16}`), GitHub tokens (`gh[ps]_[A-Za-z0-9]{36}`), private key headers, `.env` with `SECRET|KEY|TOKEN|PASSWORD=` lines. Flagged files are skipped from the Claude request and recorded as a `security` finding.
- **`src/analyzers/repo/claude.ts`** — builds the Claude request:
  - **System prompt:** Cached. Describes the analyzer's role, required output JSON schema, severity definitions.
  - **User prompt:** The preprocessing result + selected source files. Target token budget: ~60k input tokens (Sonnet 4.6 has a much bigger context; we deliberately stay modest for cost).
  - **Output format:** JSON-only response shaped like `{ summary, findings: Finding[] }`. Uses `output_format: { type: "json_object" }` or a response schema.
  - **Caching:** System prompt + preprocessing context marked as cache breakpoint. Re-analysis of the same repo at the same commit is a cache hit (but still rerun — we don't want to stub live analysis with stale data).
- **`src/analyzers/repo/index.ts`** — orchestrates: clone/pull → preprocess → secretScan → claude → persist. Emits `repo:job-progress` + `repo:finding` + `repo:analysis-complete` via the `io` handle.
- **`src/lib/scheduler.ts`** — cron-ish scheduler that runs every 10 min, enumerates repos with `analysisSchedule != 'manual'` whose last analysis is older than their interval, and enqueues a job. Per-user quota check happens at enqueue time; jobs over quota are silently skipped (next tick reconsiders).

### Hub-client additions

- New wizard branch: when the type-selection step picks a repo type, the wizard switches to a **Repo details** step — URL + credential + schedule, plus a username field that's shown only for `GIT_REPO`. The step shows provider-specific hints inline (e.g., "Paste the full `https://dev.azure.com/…/_git/…` URL; PAT needs Code (Read) scope" for Azure DevOps).
- `RepoDetailPage` — parallel to `HomeLabPage` but with **Summary / Findings / Configuration** tabs. Route: `/labs/:labId` stays the same; the page branches on `resourceType` to render the correct detail shell.
- `FindingsList` component — severity + category filters, virtualised list, click to expand for markdown detail + file/line context.
- `AnalysisProgress` component — live progress bar driven by the `repo:job-progress` socket events.
- Overview card gains a repo variant with provider icon (GitHub / GitLab / Azure DevOps logos; generic fork icon for `GIT_REPO`), last-analyzed relative time, and critical-finding count badge.

### Shared-types additions

- `AnalysisSchema`, `FindingSchema`, `RepoAnalysisSchema`, `JobSchema`.
- `RepoCreateSchema` — extends `CreateHomeLabSchema` with `repoUrl`, `repoUsername?`, `repoCredential`, `analysisSchedule`. Refinement rule: `repoUsername` required when `resourceType === "GIT_REPO"`, otherwise optional.
- `AnalysisScheduleSchema`, `FindingSeveritySchema`, `FindingCategorySchema`, and extended `ResourceTypeSchema` (adds `GITHUB_REPO`, `GITLAB_REPO`, `AZURE_DEVOPS_REPO`, `GIT_REPO`).
- Per-provider URL regex exported from `providers.ts` and referenced from the client-side form validator so we don't duplicate the Azure DevOps URL patterns.

## Security Considerations

1. **Credentials at rest** — AES-256-GCM with a key from env. Key rotation is manual in v0.3.0: set new `CREDENTIAL_ENCRYPTION_KEY`, re-submit credentials via PATCH (documented in Help Center). Key-rotation automation is v0.4.0.
2. **Credentials in transit to Git** — passed to `git` via the credential helper over stdin (never via URL embedding, never via env). The helper exits immediately after the clone.
3. **Credentials in logs** — structured logger filter blocks field names matching `/credential|token|password|secret/i`. Verified by a unit test.
4. **Source code to Claude** — preflight secret scanner drops files matching high-entropy or known-pattern secrets. A dropped file becomes a `security` finding ("Possible secret detected in path X, review manually") so the user isn't surprised by the omission.
5. **Rate limiting** — `/api/homelabs/:id/analyze` gets its own limiter (5 req / hour / user) on top of the per-user daily quota.
6. **Cloned repo blast radius** — clones live in a directory owned by the hub-server user, `0700`. No shell execution from the repo. Dependency install is **not** run in v0.3.0 — we only read manifests, never execute `npm install` or equivalent.
7. **ANTHROPIC_API_KEY** — stored only in env, never in DB, never logged, never echoed in error responses.
8. **Prompt-injection resistance** — system prompt instructs Claude to ignore any instructions found inside repo content. Output is validated against the `FindingSchema` — any response that fails validation is rejected and the job marked failed.

## Non-Functional Requirements

| Requirement | v0.3.0 target |
|---|---|
| Clone time | < 60 s for repos up to 100 MB shallow |
| Preprocessing time | < 5 s for repos up to 200k LOC |
| Claude analysis time | Varies with repo size; target p95 < 120 s |
| Analysis cost | Target < $0.10 per analysis for a typical 20k LOC repo |
| Job polling overhead | Worker polls at 2 s; < 10 ms per empty poll |
| Repo disk cap | 500 MB per repo, enforced before clone completes |
| Socket.IO progress update rate | Max 4/s to avoid UI thrash |
| Analysis quota | 20/day/user default, env-configurable |

## Known Limitations (v0.3.0)

1. **Default branch only** — no per-branch analyses. Switching the default branch in the remote means the next analysis follows it.
2. **Analyses are never partial** — if the Claude call fails, the whole analysis fails. No resuming.
3. **No dependency install** — dep analysis is manifest-only (no transitive vulnerability scan via Snyk / OSV in v0.3.0).
4. **Cost is best-effort** — USD estimates use the price list embedded in `analyzer/claude.ts` at ship time and may drift if Anthropic updates pricing. Token counts themselves are authoritative.
5. **Findings are read-only** — no ack / dismiss / mark-as-won't-fix workflow. User re-runs analysis to refresh findings.
6. **Single-worker job runner** — analyses run serially per hub-server instance. For multi-user or high-throughput deployments, the job runner will need a proper queue (BullMQ + Redis) in v0.4.0.
7. **HTTPS + PAT only** — no SSH key auth.
8. **No webhook-triggered re-analysis.**

## Rollback & Compatibility

- All schema changes additive. Existing monitoring resources keep working unchanged.
- Feature is fully opt-in: a deployment that never creates a repo resource doesn't need `ANTHROPIC_API_KEY` or `CREDENTIAL_ENCRYPTION_KEY`. The env validator enforces them lazily on first repo operation.
- The new `Job` table is general-purpose — later features (bulk imports, scheduled exports) can use it without schema changes.
- Socket.IO event names are new (`repo:*`), so v0.2.0 clients that connect to a v0.3.0 server simply ignore them.

## Open questions for owner review

These are the judgement calls worth confirming before we start building. Redirect any that don't match your mental model.

1. **Resource-type extension vs separate model.** I'm proposing **extending `HomeLab`** with nullable repo columns rather than creating a separate `Repository` model. Pro: reuses pagination, ownership, sidebar, wizard scaffolding; one "resources" concept for the user. Con: schema becomes mildly polymorphic. Alternative would be a `Repository` model with its own routes.

2. **Analyzer architecture: structured single-shot vs tool-using agent.** I'm proposing **structured single-shot** — one Claude request per analysis returning JSON-structured findings, with deterministic preprocessing in front. Pro: predictable cost, easier to test, faster. Con: less deep than an agent that can traverse files. The "Claude agent" phrasing in the request could point toward a tool-using loop (Claude Agent SDK); that's doable but roughly 3–5× the cost per analysis and harder to time-bound. Happy to switch if the cost tradeoff is worth it to you.

3. **Credential encryption key source.** I'm proposing a **dedicated env var** `CREDENTIAL_ENCRYPTION_KEY`. Alternative: derive from `JWT_SECRET` using HKDF. The separate-key approach is cleaner but means another secret to manage. HKDF-from-JWT is ergonomically nicer for solo homelabbers but ties two concerns to one key.

4. **Analysis findings — immutable snapshot vs living list.** I'm proposing **immutable snapshots** — each analysis creates a new `RepoAnalysis` with its own `Finding` rows. The user sees the latest by default; history is preserved. Alternative is a "living" findings list that's updated in place. Snapshots are simpler to reason about and make history meaningful; they also cost more storage. Default feels right for a homelab tool — speak up if you want in-place updates.

5. **Schedule granularity.** I'm proposing `manual | hourly | daily | weekly`. Anything finer (every-N-minutes) seems like it's really a webhook use case, which is deferred. Anything coarser (monthly) probably doesn't warrant a preset. Confirm?

6. **Per-user quota default.** 20 analyses/day feels safe for a homelab tool at Sonnet 4.6 pricing (roughly $2 of API spend at the $0.10/analysis target). Do you want this higher, lower, or configurable-only (no default cap)?

7. **Model selection.** Proposing **Claude Sonnet 4.6** as the default model. Pro: best quality-per-dollar for code reasoning. Alternatives: Haiku 4.5 for much cheaper but weaker analysis, or Opus 4.7 for highest quality at ~3× the cost. Should this be a per-deployment env var, a per-repo setting, or hardcoded to Sonnet?

8. **Where do repo clones live?** Proposing `/var/lib/overwatch/repos/<labId>` inside the hub-server container, backed by a new Docker volume. Concern: the `hub-server` Dockerfile currently doesn't declare a writable app-data volume. Adding one is straightforward but changes the compose shape. Alternatively, clone to an OS tempdir and re-clone on every analysis — simpler, but wastes bandwidth for daily re-analyses of a 200 MB repo.
