# Overwatch Homelab — Product Specification v0.6.0 (DRAFT)

> **Status:** Design draft for review. Assumes v0.4.0 (SMART / GPU / UPS / environmental sensors) and v0.5.0 (SNMP + network health) have shipped — their resources, alert metrics, and snapshot tables are referenced here without re-specifying them. No code yet. The choices called out as "Open questions" at the bottom need owner review before implementation starts.

## Overview

v0.6.0 is the milestone where Overwatch stops being a single-user homelab tool and becomes a **shareable platform**. Every track below exists because the thesis we've been shipping against — "one operator, one JWT, one set of resources" — stops holding the moment anyone wants to hand someone else a login, expose a read-only board to a community, or let a second admin acknowledge alerts overnight.

Six coherent features ship together under the theme **"From single-user homelab tool to shareable platform."**

1. **Multi-user RBAC + resource sharing** — `Organisation`, `Membership`, `ResourceShare`. Every existing resource gets an owning org; every route becomes org-scoped; JWT gains `orgId`.
2. **Agent packaging + auto-update** — first-class `.deb` / `.rpm` / Homebrew / MSI / signed tarball, systemd / LaunchAgent / Windows service units, and a signed auto-update protocol over Ed25519.
3. **TLS / mTLS between agent and hub** — optional ACME-issued server cert plus a hub-rooted CA that can mint per-agent client certs. Closes the confidentiality gap that v0.3.0 S4 explicitly left open.
4. **TimescaleDB + tiered downsampling** — convert the five snapshot tables into hypertables, materialise 1 m → 5 m → 1 h continuous aggregates, and apply per-tier retention. Feature-flagged so deployments without the extension keep working.
5. **Public status pages** — read-only `/status/:shareSlug` route rendering up/down + 30-day uptime in the Statuspage.io style.
6. **PWA + mobile-responsive dashboard** — installable web app manifest, service worker cache, responsive breakpoints on every existing page, and Web Push as a new `AlertChannel`.

Two tracks ship together:

1. **Feature track** — the six F-items above.
2. **Supporting track** — RBAC migration scripts, release-signing pipeline, per-feature hub-side feature flags, and expanded test harnesses (permission-matrix tests, signed-release verification tests).

**This milestone's scope is the largest in the project's history.** Two of its data changes — the RBAC ownership reshape (F1) and the Timescale hypertable conversion (F4) — are **not additively reversible** the way every previous milestone has been. See *Rollback & Compatibility* for what that means concretely. The hub-side feature flags in S5 exist precisely so individual features can be disabled for rollback without reverting the whole release.

## Foundation from v0.2.0 → v0.5.0

Not in scope to re-implement:

- Auth, cursor pagination, Help Center, Socket.IO JWT handshake — v0.2.0.
- Threshold alert evaluator, `MetricSnapshot`, retention pruner — v0.2.0.
- Docker container monitoring, uptime checks, alert channels (webhook / Discord / ntfy / SMTP), maintenance windows, Prometheus `/metrics`, agent token auth, job runner, pino logging — v0.3.0.
- SMART / ZFS / mdadm, GPU telemetry, NUT / UPS, environmental sensors — v0.4.0 (assumed shipped).
- SNMP device discovery, `NetworkDevice` + `NetworkSnapshot` + `InterfaceSnapshot`, gateway / DNS / speedtest probes — v0.5.0 (assumed shipped).

v0.6.0 **reuses**:

- `AlertChannel` and `AlertRoute` from v0.3.0 — a new `WEB_PUSH` channel slots into the discriminated union without evaluator changes.
- `UptimeCheck`, `NetworkDevice`, and `HomeLab` — each becomes a shareable resource via the new `ResourceShare` table.
- The v0.3.0 capability-negotiation protocol — agent TLS / mTLS mode is advertised as a new capability key (`transport: "tls" | "mtls" | "plaintext"`).
- The job runner from v0.3.0 — gains a new `kind: "auto-update-check"` and `kind: "cert-rotation"`.
- The retention pruner from v0.2.0 / v0.3.0 — superseded by Timescale retention policies when the extension is present; falls back to the existing pruner when it isn't.

## Scope

### In scope — feature track

#### F1. Multi-user RBAC + resource sharing

1. **`Organisation` model** — top-level tenant boundary. Every resource (`HomeLab`, `UptimeCheck`, `NetworkDevice`, `AlertChannel`, `AlertRoute`, `MaintenanceWindow`, `StatusPage`) gains an `orgId` FK. There is no user-scoping after this milestone — `ownerId` columns are dropped in favour of `orgId` + the membership matrix.
2. **`Membership` model** — joins `User` ↔ `Organisation` with a `role` enum: `owner | admin | editor | viewer`. A user can be in many orgs; a user's default org is tracked on `User.defaultOrgId`. Role semantics:
   - `owner` — full control, including destroying the org itself, managing billing-adjacent settings, and inviting owners. Every org has at least one owner; the last owner cannot be demoted or removed.
   - `admin` — everything an owner can do except org destruction and owner-role assignment.
   - `editor` — create/update/delete resources, acknowledge alerts, manage uptime checks, edit alert routes.
   - `viewer` — read-only across every resource the org owns or is shared with them. Explicitly cannot acknowledge alerts, trigger `POST /uptime-checks/:id/run`, or edit channels. Can subscribe to Socket.IO rooms.
3. **`ResourceShare` model** — grants a resource from one org to another membership context with a given role cap. Shape: `{ resourceKind, resourceId, sourceOrgId, targetOrgId?, targetMembershipId?, role }`. Two forms:
   - **Org-to-org share** — `targetOrgId` set; every member of the target org inherits `min(role, their org role)` on the shared resource. Useful for MSP-style "I manage my friend's homelab" flows.
   - **Direct-user share** — `targetMembershipId` set; only that specific membership gets the grant. Useful for one-off "let this one user see this one dashboard" cases.
4. **JWT shape** — gains `orgId` (the active org for this session) and `role` (the user's role in that org, resolved server-side at issue time). `GET /api/me/orgs` lists memberships; `POST /api/auth/switch-org { orgId }` issues a new JWT scoped to that org. The old token is not invalidated server-side (stateless JWT), but the client discards it immediately.
5. **Migration path (critical — see Rollback)** — existing single-user installs get a **default org per user**, named `"{user.name}'s Workspace"`, with that user as sole `owner` and `defaultOrgId` set. Every existing `HomeLab`, `UptimeCheck`, `NetworkDevice`, `AlertChannel`, `AlertRoute`, `MaintenanceWindow` row has its previous `ownerId` rewritten to that user's default org's `id`. The migration is idempotent and transactional at the row-group level. Rolling back to v0.5.0 requires reconstructing `ownerId` from `Membership` — the spec ships a `down-migration` script that does this deterministically for any org with exactly one `owner` member; multi-member orgs block the downgrade with a clear error.
6. **Route scoping** — every authenticated route gains a `requireOrgRole(role)` middleware layered after `authenticate`. The middleware reads `req.user.orgId`, verifies membership, and rejects with `403 FORBIDDEN` if the user's effective role for the addressed resource (direct-membership role or `ResourceShare`-granted role, whichever is higher) is below the required level. Write endpoints default to `editor`; destructive endpoints default to `admin`; org-settings endpoints default to `owner`.
7. **Socket.IO scoping** — `dashboard:subscribe { labId }` now checks "user has effective role >= viewer on this resource, via membership in the owning org OR via a `ResourceShare`". The subscription layer learns to join a union of rooms: `lab:<labId>` for direct sockets and `org:<orgId>` for fan-out of cross-resource events (e.g. a new lab being created).
8. **UI — `/org` page** — org list, member list with role pills, invite-by-email flow (the invite sends the recipient a one-time signup/join token by email via the existing SMTP config, or by copy-paste URL if SMTP isn't configured). Admins can demote / promote / remove members. Owners see a "Danger zone" with "Transfer ownership" and "Delete organisation".
9. **UI — share dialog** — "Share" button on each resource detail page opens a dialog: org-to-org (autocomplete of orgs the inviter has write access to — practically their own) and direct-user (email input → creates a pending `ResourceShare` resolved on signup). Role selector defaults to `viewer`.
10. **Audit trail (minimal)** — every membership change and share creation writes a row to a new `AuditEvent` table (actor, action, target, at). No UI in v0.6.0; the table exists so v0.7.0+ can build one without a schema bump. (The audit-log UI is deferred explicitly.)

#### F2. Agent packaging + auto-update

1. **Release artefacts** — the agent build pipeline produces, for each tagged release `lab-agent@X.Y.Z`:
   - Debian `.deb` for `amd64` and `arm64` (built via `fpm` in a Docker builder image).
   - RPM for `x86_64` and `aarch64`.
   - Homebrew formula in a `homebrew-overwatch` tap repo — `brew install overwatch/tap/lab-agent`.
   - Windows MSI (best-effort — packaged via `wix` in a Windows GitHub Actions runner; documented as beta-quality if the runner isn't available).
   - Signed tarball `lab-agent-X.Y.Z-<os>-<arch>.tar.gz` for distributions we don't natively package.
   - Detached Ed25519 signature file `.sig` per artefact plus an `SHA256SUMS` file signed as a unit.
2. **Install-time service integration**
   - **Linux `.deb` / `.rpm`** — installs `overwatch-lab-agent.service` systemd unit under `/usr/lib/systemd/system/`. Unit runs as a new `overwatch` system user (created postinst), reads config from `/etc/overwatch/lab-agent.env`, logs to journald. `systemctl enable --now overwatch-lab-agent` runs postinstall.
   - **macOS Homebrew** — installs a LaunchAgent at `~/Library/LaunchAgents/com.overwatch.lab-agent.plist`. `brew services start lab-agent` enables.
   - **Windows MSI** — installs as a Windows service via `nssm`-style wrapping, configured for `Automatic (Delayed Start)`.
   - **Tarball** — ships a `scripts/install.sh` that auto-detects `systemd` and offers a one-line setup; otherwise documents the `node dist/index.js` invocation.
3. **Config location convention**
   - Linux: `/etc/overwatch/lab-agent.env` (system-wide) or `~/.config/overwatch/lab-agent.env` (per-user fallback).
   - macOS: `~/Library/Application Support/overwatch/lab-agent.env`.
   - Windows: `%ProgramData%\Overwatch\lab-agent.env`.
4. **Auto-update protocol**
   - New env var `AGENT_UPDATE_CHANNEL` with values `stable | beta | pinned` (default `pinned` — opt-in upgrade only; "auto" is never the default, because homelab operators don't want their monitoring daemon restarting unannounced).
   - Agent polls `GET /api/agent-releases/latest?channel=<channel>&currentVersion=<ver>&os=<os>&arch=<arch>` every 6 h. The hub returns `{ version, downloadUrl, signatureUrl, minimumSupportedVersion, rolloutPercent }` or `204 No Content` if no upgrade is available.
   - Agent downloads the artefact + `.sig` over HTTPS, verifies Ed25519 signature against a **pinned hub public key** baked into the agent binary at build time (the key rotates across major versions; rotation is documented in the runbook referenced — not included here).
   - Agent stages the new artefact to a temp path, then on next SIGHUP or scheduled 03:00-local restart hands off to the system package manager (`dpkg -i`, `rpm -U`, `brew upgrade`, or `msiexec /i`) and exits cleanly so the supervisor restarts it.
   - **Staged rollout** — `rolloutPercent` on the release record gates what percentage of polling agents see the new version. The hub assigns each agent a stable hash bucket by `labId`, so a 10 % rollout is the same 10 % of labs every time the agent polls.
5. **Hub side**
   - New `AgentRelease` table — `version`, `channel`, `os`, `arch`, `downloadUrl`, `signatureSha256`, `publishedAt`, `rolloutPercent`, `minimumSupportedVersion`, `yanked` (bool).
   - `GET /api/agent-releases` (admin+) for management; `GET /api/agent-releases/latest?…` is authenticated via the agent's existing token/mTLS path.
   - Upload flow is deliberately **out-of-band**: artefacts live on object storage (operator-provided — S3, R2, whatever), and the hub stores only metadata + URL. Reason: the hub is not a release CDN and shouldn't stream package bytes under a JWT-authenticated path.
6. **Auto-update is opt-in** — `AGENT_UPDATE_CHANNEL=pinned` (default) disables polling entirely. An operator has to actively set `stable` or `beta` to enable auto-update. Rationale: self-updating daemons are a common support-ticket source; default should be "you picked this version, we won't change it without you".

#### F3. TLS / mTLS between agent and hub

1. **Server TLS on the hub**
   - New `HUB_TLS_MODE` env: `off | provided | acme` (default `off` — preserves the v0.5.0 plaintext-behind-reverse-proxy convention).
   - `provided` — operator supplies `HUB_TLS_CERT_PATH` and `HUB_TLS_KEY_PATH`; the hub loads them at boot and serves HTTPS on `HUB_TLS_PORT` (default 3443) **in parallel** with the plaintext port. File changes are picked up by SIGHUP without a full restart.
   - `acme` — embedded lightweight ACME client (likely `acme-client` or `@peculiar/acme-client` — dependency choice deferred to implementation). Requires `HUB_TLS_DOMAIN`, `HUB_ACME_CONTACT_EMAIL`, and either HTTP-01 (hub must be reachable on port 80) or DNS-01 (supply `HUB_ACME_DNS_PROVIDER` + credentials for providers: Cloudflare, Route53, DigitalOcean — three to start). Certificate auto-renews at 30-days-to-expiry via a background job (`kind: "cert-renewal"`).
2. **Mutual TLS (mTLS) for agents**
   - A new `HUB_AGENT_CA` mode turns on the hub's internal certificate authority. On first boot with this mode, the hub generates a self-signed Ed25519 CA keypair and stores it encrypted at rest via the existing `CREDENTIAL_ENCRYPTION_KEY` (introduced in v0.3.0). Public CA cert is exposed at `GET /api/ca.pem` (authenticated as `admin`).
   - When mTLS is enabled for a given lab (new per-lab setting `HomeLab.agentTransport: plaintext | tls | mtls`, migrates to `tls` for everything when the hub runs in TLS-required mode), the hub issues a **per-agent client certificate** on demand: `POST /api/homelabs/:id/agent-cert` returns a PKCS#12 bundle plus a one-time unwrap password. Agent stores this under its config directory (see F2).
   - Agent's Socket.IO handshake presents the client certificate. The hub's Socket.IO middleware verifies: chain to the hub CA, CN matches `labId`, not revoked (checked against an in-memory revocation cache that refreshes every 60 s from `AgentCertRevocation`).
   - Optional: in mTLS mode the `AGENT_TOKEN` from v0.3.0 becomes redundant. Per-lab setting chooses whether both are required (defence in depth) or whether mTLS alone is sufficient (operational simplicity). Default when mTLS is on: mTLS alone.
3. **Certificate rotation** — client certs have a 90-day validity. An auto-rotation job (`kind: "cert-rotation"`) inside the agent re-runs `POST /api/homelabs/:id/agent-cert` at 15-days-to-expiry, atomically swaps the bundle on disk, and signals its Socket.IO client to reconnect with the new cert. Server cert auto-rotates via the ACME client in `acme` mode; in `provided` mode, rotation is the operator's responsibility (hub logs a warning 14 days out).
4. **Revocation** — `POST /api/homelabs/:id/agent-cert/revoke` writes to `AgentCertRevocation` (serial number, revokedAt, reason). The revocation check runs inline on Socket.IO handshake — a revoked agent's connection is rejected immediately.
5. **Compatibility** — plaintext mode remains the default for backwards compatibility with v0.2.0–v0.5.0 deployments. TLS is an operator choice, not a forced upgrade. `HUB_TLS_MODE=off` + `HomeLab.agentTransport=plaintext` is the no-op path.

#### F4. TimescaleDB + tiered downsampling

1. **Hypertable conversion** — `MetricSnapshot`, `ContainerSnapshot`, `UptimeResult`, `NetworkSnapshot`, `InterfaceSnapshot` each become Timescale hypertables partitioned on `recordedAt` / `executedAt` with a **7-day chunk interval**. Primary-key shape changes from `id` alone to `(id, recordedAt)` to satisfy Timescale's chunk-pruning requirements; the `id` column stays a unique UUID so application code that looks up by id still works (join through the hypertable index).
2. **Continuous aggregates (CAggs)** — three tiers per source table:
   - **1 m** — `(labId OR checkId OR deviceId, bucket_1m)` with `avg(cpuPercent)`, `avg(memPercent)`, `p95(responseTimeMs)`, `sum(uptimeSamples)`, etc.
   - **5 m** — rollup of the 1 m CAgg.
   - **1 h** — rollup of the 5 m CAgg.
3. **Retention policies** — per-tier, per-lab-or-default:
   - Raw: 7 days (down from the v0.3.0 lab-configurable `retentionDays`; operators who need longer raw retention can override per lab).
   - 1 m: 7 days (deliberately same as raw — 1 m is the "recent history" resolution, not a long-term tier; we keep it short to keep the CAgg refresh cheap).
   - 5 m: 30 days.
   - 1 h: 365 days.
   These are Timescale drop-chunks policies — cheap, no row-by-row deletes.
4. **Query routing** — the existing `/api/homelabs/:id/metrics?resolution=…` endpoint now resolves the resolution-to-table mapping server-side: `raw → hypertable`, `1m → cagg_1m`, `5m → cagg_5m`, `1h → cagg_1h`. Time windows that span tier boundaries query the highest-resolution tier that fully covers the window (conservatism — a 20-day window falls through to 5 m because 7 days of raw doesn't cover it).
5. **Feature flag** — `TIMESCALE_ENABLED=true|false` (default: `true` if the extension is present at boot, `false` otherwise). The startup sequence issues `CREATE EXTENSION IF NOT EXISTS timescaledb` guarded by a superuser check; if the extension is unavailable, the hub logs once at `info`, sets the flag to `false`, and keeps the v0.3.0 plain-Postgres retention pruner running. All read paths branch on the flag so Postgres-only deployments keep working with a degraded retention story (single-tier, same as today).
6. **Migration risk (critical — see Rollback)** — converting an existing table to a hypertable is a one-way operation from the application's perspective. The migration runs inside a transaction per table, but on large existing installs (>1 M rows) it can take minutes. The spec provides an explicit offline migration script (`scripts/migrate-to-timescale.ts`) that operators run once, not an automatic boot-time conversion. Rolling back to v0.5.0 requires either (a) keeping a pre-migration backup, or (b) the spec's `down-migration.sql` which un-hypertables each table (Timescale supports this via `decompress_chunk` + manual `INSERT INTO … SELECT` — slow but correct).

#### F5. Public status pages

1. **`StatusPage` model** — owned by an org; fields: `id`, `orgId`, `slug` (URL-safe, org-unique), `title`, `description`, `themeJson` (primary colour, logo URL, footer markdown), `includedResourcesJson` (`[{ kind: "homelab" | "uptime-check" | "network-device", id }]`), `isPublic` (bool, default `true`), `passwordHash` (bcrypt, optional — for soft-gated pages), `createdAt`, `updatedAt`.
2. **Public route** — `GET /status/:slug` is **unauthenticated** and served by the hub-server (not behind `/api`). Returns an HTML shell plus a JSON state document at `GET /status/:slug/state`. Rate-limited 60 req / min / IP. When `passwordHash` is set, the HTML prompts for a password that sets a short-lived cookie (`sp_<slug>`) with a signed payload; no account needed.
3. **Rendered content**
   - Overall status banner (green / yellow / red) computed from the included resources' current state.
   - 30-day uptime bar per included uptime check (one coloured cell per day; Statuspage.io style) — green if 100 %, yellow if < 99.9 %, red if any outage >= 30 min, grey if no data.
   - For included HomeLabs: connection status + last-heartbeat time. No CPU / memory / disk values (prevents info disclosure about internal capacity).
   - For included NetworkDevices: reachability + last poll. No SNMP OIDs or interface detail.
   - Incident timeline: last 10 incidents (auto-generated from `Alert` fire / resolve transitions for the included resources). Operators can optionally post a manual update on an incident via `POST /api/status-pages/:id/incidents/:alertId/updates { message }`.
4. **Privacy posture** — the status page is deliberately a **reachability surface**, not a metrics surface. If an operator wants to expose raw CPU % on a public page, that's a v0.7.0+ conversation with very different privacy defaults. v0.6.0 ships the narrow version.
5. **Custom domain** — out of scope for v0.6.0. Operators front the hub with their own reverse proxy if they want `status.example.com`. CNAME-based multi-tenancy is a v0.7.0+ feature.
6. **UI — Status Page builder** — new `/status-pages` page in the authenticated app. List existing pages, create new, pick theme + included resources with a checklist. Preview pane shows the public page in an iframe. "Copy public URL" button.

#### F6. PWA + mobile-responsive dashboard

1. **Web App Manifest** — `public/manifest.webmanifest` with `name: "Overwatch Homelab"`, `short_name: "Overwatch"`, icons at 192 / 512 / maskable, `display: "standalone"`, `theme_color` matching the Tailwind `brand` palette. `start_url: "/overview"`.
2. **Service worker** — generated via `vite-plugin-pwa` using Workbox. Caching strategy:
   - App shell (`index.html`, JS, CSS, fonts, icons) — **stale-while-revalidate** with a 24 h TTL.
   - `/api/homelabs`, `/api/uptime-checks`, `/api/network-devices` list endpoints — **network-first with 5 s timeout**, cached for 60 s so the app opens fast on flaky mobile networks.
   - Historical metrics / probe results — **network-only** (cannot safely cache time-ranged queries under a simple policy; not worth the complexity).
   - Socket.IO traffic is never cached.
3. **Mobile breakpoints** — every existing page gets explicit breakpoint work: Monitor (charts stack vertically; summary cards become swipeable), Alerts (list → card layout; acknowledge becomes a swipe action), Uptime (check rows collapse their sparkline into a current-status pill by default; tap to expand), Containers (table → stacked cards; sort controls move into a bottom sheet), Network (device list same; device detail's interface table scrolls horizontally with sticky interface name column). Tailwind breakpoints at `sm` (640 px), `md` (768 px), `lg` (1024 px). The sidebar collapses to a bottom nav at `< md`.
4. **Web Push alert channel**
   - `AlertChannel.type` enum gains `WEB_PUSH`. Config shape: `{ subscriptions: PushSubscription[] }` — a single channel can fan out to every device the user has opted in on.
   - Hub generates a VAPID keypair once at first boot (stored in a new `HubSettings` singleton table). Public key exposed at `GET /api/push/vapid-public-key`. Private key encrypted at rest via `CREDENTIAL_ENCRYPTION_KEY`.
   - Client opts in via a "Enable browser notifications" button in Settings. The resulting `PushSubscription` JSON is POSTed to `POST /api/push/subscribe { subscription }` and stored against the user + device.
   - Delivery uses the existing `web-push` library path in the delivery worker. A `WEB_PUSH` channel failure (expired subscription, 410 Gone) auto-prunes that subscription from the channel config.
   - Per-device on/off — the user can disable push on a specific device from Settings; the subscription stays in the channel but is flagged `paused`.
5. **Install prompt** — the client listens for `beforeinstallprompt` and surfaces a one-time toast after three successful logins ("Install Overwatch for faster access"). Dismissible; never re-prompts.

### In scope — supporting track

- **S1 — Per-feature hub-side feature flags.** New env vars — `FEATURE_RBAC=true`, `FEATURE_AGENT_AUTOUPDATE=true`, `FEATURE_MTLS=true`, `FEATURE_TIMESCALE=true`, `FEATURE_STATUS_PAGES=true`, `FEATURE_PWA_WEB_PUSH=true`. Each flag has a clear failure mode when disabled: RBAC-off falls back to single-org-per-user with every user as `owner` (i.e. v0.5.0 behaviour); mTLS-off falls back to token auth; Timescale-off runs the legacy pruner; status-pages-off 404s `/status/*`; web-push-off hides the subscribe button and rejects `POST /api/push/subscribe`. Reason: this milestone's scope is wide enough that a single bad feature shouldn't force rolling back the whole release.
- **S2 — RBAC migration scripts.** Two scripts in `apps/hub-server/scripts/`: `migrate-to-rbac.ts` (forward) and `rollback-from-rbac.ts` (reverse, with the multi-member-org guard described in F1.5). Both are idempotent; both log a per-row manifest to `rbac-migration-<timestamp>.log` for audit.
- **S3 — Release-signing pipeline.** GitHub Actions workflow `release-agent.yml` that: builds the matrix of artefacts, signs each with an Ed25519 key pulled from the workflow's secret store (key custody documented in the separate ops runbook — intentionally out of this spec), uploads to object storage, writes an `AgentRelease` row via an admin API call, and creates a GitHub Release with checksums. Key rotation procedure documented in the runbook.
- **S4 — Hub-rooted CA lifecycle.** The hub's self-generated CA keypair (F3) is generated exactly once per hub and is **not** portable — two hubs do not share a CA. Documented: to move agents between hubs, re-issue their client certs. CA rotation requires re-issuing every agent cert; the spec documents the procedure and provides a helper script `scripts/rotate-hub-ca.ts` that coordinates re-issuance over a configurable overlap window.
- **S5 — Test harness extensions**
  - **RBAC permission matrix tests** — generated Vitest suite that spins up an in-memory org-and-membership graph with N users × M roles × K resource kinds and asserts 403 / 200 outcomes against every authenticated route. Target: covers every route in the app, enforcing a "no new route lands without a matrix entry" CI check.
  - **Signed-release verification tests** — unit tests that assert the agent's Ed25519 verifier rejects: wrong key, tampered artefact, mismatched SHA256, expired signature (future work — we sign without expiry today), and accepts the happy path. Fixtures generated in-test; no network.
  - **mTLS handshake tests** — Socket.IO integration test with a real TLS socket pair: valid chain accepts, wrong CN rejects, revoked cert rejects, expired cert rejects.
  - **Timescale hypertable tests** — spin up a `timescale/timescaledb:latest-pg16` container in CI; assert hypertable creation, CAgg correctness (seed known points, read back bucket averages), and retention policy enforcement.
  - **Status page rate-limit + auth-gating tests** — 60/min/IP enforcement; password-protected page cookie round-trip; public page unauthenticated.
- **S6 — Client test extensions.** Vitest + RTL tests for: org-switcher, share dialog, mobile-breakpoint snapshot tests at `sm / md / lg` for each page, PWA install prompt state machine, push-subscription opt-in flow (mocked `PushManager`), status-page builder form validation.
- **S7 — Structured logging extensions.** Add `orgId`, `membershipId`, and (for agent sockets) `agentCertFingerprint` to the `AsyncLocalStorage` context so every log line from an authed request is org- and identity-scoped. Redaction filter extended to drop PKCS#12 password fields.
- **S8 — `/metrics` extensions.** New Prometheus gauges: `overwatch_orgs_total`, `overwatch_memberships_total{role}`, `overwatch_active_shares_total`, `overwatch_agent_tls_mode{mode}`, `overwatch_agent_release_adoption{version,channel}` (bucketed — how many agents on each version), `overwatch_status_page_requests_total{slug,status}`, `overwatch_web_push_delivery_total{status}`.
- **S9 — Help Center entries.** Six new topics: `organisations-and-sharing`, `agent-packaging`, `auto-update`, `tls-and-mtls`, `timescale-retention`, `status-pages`, `pwa-and-push`. Each includes a "rollback this feature" section pointing to the S1 flags.
- **S10 — OpenAPI generation (stretch).** Since RBAC blows up the route surface, this milestone is where auto-generating an OpenAPI document from the Zod schemas becomes genuinely useful. Stretch because it isn't load-bearing for any F-item — but the test-harness extensions in S5 benefit from it (the matrix generator can read the route list from the OpenAPI doc instead of a hand-maintained array).

### Out of scope (deferred to v0.7.0+)

- Kubernetes / Podman pod-level monitoring (beyond v0.3.0's Docker collector).
- Netflow / sFlow / packet-level network observability.
- Wake-on-LAN + remote restart UI for managed hosts.
- Log aggregation / tailing (journald, syslog, application logs).
- Backup job ingestion (restic / borg / rsnapshot / duplicati exit codes).

## User Stories

### RBAC + sharing (F1)

- **US-96** — As an existing v0.5.0 user, after the v0.6.0 upgrade I still see all my resources exactly as before, now inside "My Workspace" — without doing anything. I can invite a second user and give them `viewer` role on a specific HomeLab.
- **US-97** — As an org `owner`, I can invite members by email, assign roles (`admin | editor | viewer`), and demote / remove them later. The last owner cannot be demoted.
- **US-98** — As an `admin`, I can create, edit, and delete any resource in my org, but I cannot destroy the org itself or promote someone to `owner`.
- **US-99** — As an `editor`, I can manage resources and acknowledge alerts, but I cannot invite members or change org settings.
- **US-100** — As a `viewer`, I see every resource my org owns or has been shared with me, but every write button and acknowledge action is disabled or hidden — and if I try to POST directly, the server rejects with 403.
- **US-101** — As an org admin in org A, I can share a HomeLab with org B at role `viewer`; every member of org B sees that HomeLab read-only in their UI without it mingling with org B's own resources.
- **US-102** — As an org admin, I can share a single resource with a specific user by email; if they don't have an account yet, they get a signup link that completes the share on registration.
- **US-103** — As a user with memberships in multiple orgs, I can switch between them from the top bar; the JWT rotates and every query re-scopes.

### Agent packaging + auto-update (F2)

- **US-104** — As an operator on Debian / Ubuntu, I can run `apt install overwatch-lab-agent` (from the provided repo) and the agent starts as a systemd service without further config beyond editing `/etc/overwatch/lab-agent.env`.
- **US-105** — As an operator on macOS, I can run `brew install overwatch/tap/lab-agent && brew services start lab-agent` and the agent runs as a LaunchAgent.
- **US-106** — As an operator, I can set `AGENT_UPDATE_CHANNEL=stable` and the agent auto-upgrades to new releases on its next scheduled restart, after verifying the Ed25519 signature against the pinned hub public key.
- **US-107** — As a cautious operator, `AGENT_UPDATE_CHANNEL=pinned` (the default) leaves the agent on the version I installed; no polling, no upgrades, no surprises.
- **US-108** — As a hub operator, I can roll out a new agent release to 10 % of agents first (staged rollout via `rolloutPercent`) and then bump to 100 % once I've confirmed no regressions.

### TLS / mTLS (F3)

- **US-109** — As a hub operator, I can set `HUB_TLS_MODE=acme` + `HUB_TLS_DOMAIN=hub.example.com` and the hub automatically obtains and renews a Let's Encrypt certificate.
- **US-110** — As a hub operator, I can enable mTLS per-lab. The hub issues a client cert; the agent presents it on handshake; plaintext connections to that lab's labId are rejected.
- **US-111** — As a hub operator, I can revoke a specific agent's cert and its Socket.IO connection drops on the next handshake check, with no hub restart.
- **US-112** — As an agent, my client cert auto-rotates at 15 days before expiry without operator involvement.

### TimescaleDB + tiering (F4)

- **US-113** — As an operator, after running the one-shot Timescale migration my historical charts load noticeably faster for 7 d+ windows because the server reads from the 5 m continuous aggregate instead of raw rows.
- **US-114** — As an operator, I can keep 365 days of 1-hour rollups with storage cost comparable to keeping 30 days of raw 1-minute data in v0.5.0.
- **US-115** — As an operator running Postgres without the Timescale extension, the hub boots fine, logs the fallback once, and keeps working with the v0.3.0 single-tier pruner.

### Status pages (F5)

- **US-116** — As an org admin, I can create a public status page at `/status/mylab`, pick which HomeLabs and uptime checks to include, set a title and primary colour, and share the URL with non-users.
- **US-117** — As a visitor, I see current status and a 30-day uptime bar per included check without logging in. No raw metrics are exposed.
- **US-118** — As an org admin, I can password-protect a status page for a semi-private audience without creating accounts.

### PWA + mobile + push (F6)

- **US-119** — As a mobile user, I can install Overwatch to my home screen and open it as a standalone app; the shell loads offline (showing cached last-good state) and re-syncs when online.
- **US-120** — As a mobile user, every page is legible and usable on a 390 px-wide phone. The sidebar becomes a bottom nav; long tables become stacked cards.
- **US-121** — As a user, I can opt in to browser push notifications per device. When an alert fires for a resource I have access to, I get a push notification within the existing alert delivery pipeline.

### Safety & ops (supporting track)

- **US-122** — As a hub operator, I can disable any single v0.6.0 feature via a feature flag (e.g. `FEATURE_TIMESCALE=false`) and the hub keeps working in a degraded mode without a full rollback.
- **US-123** — As a hub operator, I can scrape `/metrics` and see per-version agent adoption so I know when it's safe to yank an old release.

### Hub-server additions

- **`src/middleware/rbac.ts`** — `requireOrgRole(min: 'viewer' | 'editor' | 'admin' | 'owner')` and `requireResourceRole(kind, idFromReq, min)`. Layered after `authenticate`. Resolves the caller's effective role for the resource (direct membership vs share) in a single query cached per request.

## Security Considerations

1. **RBAC bypass risk** — the biggest new attack surface is "a viewer silently becoming an editor via a missed middleware check". Mitigations: every route has a `requireOrgRole` or `requireResourceRole` call enforced in a CI lint (`tests/routes-require-rbac.test.ts` walks the Express router and asserts every handler's middleware stack contains one of the two). The RBAC permission matrix test (S5) is the backstop.
2. **JWT `orgId` tampering** — `orgId` is signed in the JWT alongside `userId`; the server never trusts a client-provided `orgId`. Switching orgs requires `POST /api/auth/switch-org` which re-verifies the membership server-side.
3. **Share elevation** — `ResourceShare.role` is capped by the sharer's role on the source resource. The server enforces `min(sharer.role, requestedRole)` on create; editing an existing share to raise its role goes through the same check.
4. **Agent release signing** — Ed25519 key custody is a named risk. Key lives in the release workflow's secret store; rotation procedure documented in the ops runbook (referenced, not included here). Agents ship with a **pinned public key** baked in at build time — a compromised hub cannot serve a malicious release to an agent that's already deployed.
5. **Auto-update as a vector** — default channel is `pinned` (no polling). Even on `stable`, the agent verifies signature before install. Minimum-supported-version gating prevents downgrade attacks.
6. **mTLS CN binding** — client certs bind CN = `labId`. An agent cannot impersonate another lab by presenting a valid cert with a different CN. Revocation check is inline on every handshake.
7. **CA private key at rest** — encrypted via the existing `CREDENTIAL_ENCRYPTION_KEY` (introduced in v0.3.0). Hub boot logs a warning if the key is missing while `HUB_AGENT_CA=true` and refuses to issue certs until the key is set.
8. **ACME DNS provider credentials** — stored encrypted at rest; redacted in logs by the pino filter. Providers ship with a documented least-privilege IAM policy in the ops runbook.
9. **Status-page info disclosure** — deliberately narrow. No metric values. Rate-limited 60 / min / IP. Password-protected pages use a signed short-lived cookie, not a persistent session.
10. **Web Push endpoints as PII** — `PushSubscription.endpoint` embeds a per-device token; treated as a secret. Never logged, encrypted in channel config.
11. **Service worker scope** — the SW is scoped to `/` but explicitly does not cache auth-sensitive endpoints (`/api/auth/*`, `/api/push/*`). Forced unregister on logout via `navigator.serviceWorker.controller.postMessage({ type: "logout" })`.
12. **Status-page slug collision** — slugs are globally unique across orgs. The builder validates against current slugs and a short reserved list (`status`, `api`, `admin`, `overview`, `auth`).
13. **Org deletion** — hard-delete cascades to every org-scoped resource. Confirmation requires typing the org name. No soft-delete in v0.6.0 — recover from backup is the answer.
14. **Audit events for sensitive actions** — membership changes, role promotions, shares, cert revocations, org deletions all write `AuditEvent` rows. Table exists even though the UI is deferred, so auditors have a trail.

## Known Limitations (v0.6.0)

1. **No SSO / OIDC.** All login is still email + password. v0.7.0+.
2. **No 2FA.** Not in v0.6.0 despite the RBAC expansion. v0.7.0+.
3. **No audit-log UI.** Events are recorded; reading them means a DB query.
4. **No custom domains for status pages.** Operators bring their own reverse proxy for `status.example.com`.
5. **Single-hub CA.** Hub CAs aren't portable; moving an agent to a different hub requires re-issuance.
6. **Auto-update is opt-in.** No "set it and forget it" default — by design, but means updates still require an explicit operator gesture.
7. **Web Push only for alerts.** Agent-connect, agent-disconnect, uptime-degrade events don't push. v0.7.0+.
8. **Timescale fallback is single-tier.** Without the extension, the v0.3.0 pruner runs and `resolution=1h` over 90 days is slow. Deployments should use Timescale if they want tiered performance.
9. **No per-org rate limits.** Global rate limits from v0.2.0 still apply uniformly. A noisy org can affect a quiet one under extreme load.
10. **Org deletion is hard-delete.** No soft-delete, no "undo" window. Operators keep backups.
11. **Windows MSI is best-effort.** Ships but without the test coverage the `.deb` / `.rpm` paths get.
12. **Status page 30-day uptime bar granularity is one day per cell.** Sub-day outage resolution is visible only by tapping a cell for that day's incidents.

## Rollback & Compatibility

Two of this milestone's changes are **not additively reversible**. Read carefully.

- **F1 RBAC ownership reshape.** Forward migration drops `ownerId` from seven tables and replaces it with `orgId`. Rolling back requires reconstructing `ownerId` from `Membership`. The shipped `rollback-from-rbac.ts` script does this deterministically **only for orgs with exactly one `owner` member** — which covers every install that upgraded from v0.5.0 without inviting anyone. Installs that actually used sharing cannot be rolled back without data loss; the rollback script refuses with a clear error and directs the operator to restore from backup. **Recommendation:** operators take a Postgres dump before running `migrate-to-rbac.ts`. The migration script prompts for confirmation of this.
- **F4 Timescale hypertable conversion.** Forward migration converts five tables to hypertables; primary keys expand to `(id, recordedAt)`. Rolling back to v0.5.0 means un-hypertabling — the provided `down-migration.sql` does `INSERT INTO tmp_<table> SELECT * FROM <table>` then swaps, which is slow on large installs but correct. CAggs drop cleanly. **Recommendation:** same as above — take a dump first.

Reversible changes:

- All new tables (`Organisation`, `Membership`, `ResourceShare`, `AuditEvent`, `AgentRelease`, `AgentCertRevocation`, `HubSettings`, `StatusPage`) are additive drops on rollback.
- `AlertChannel.type` enum gains `WEB_PUSH`; rollback drops the enum value but preserves any rows (they become invalid — the rollback script marks them `disabled`).
- Feature flags (S1) let any individual F-item be disabled in production without a schema revert. Use these first when something misbehaves; revert the release only if a flag isn't enough.

**Agent compatibility:**

- v0.5.0 agents connect without `transport` capability — hub treats that as `plaintext` and honours the per-lab `agentTransport=plaintext` default.
- v0.6.0 agents with `AGENT_UPDATE_CHANNEL=pinned` (default) behave identically to v0.5.0 agents once the packaging is installed.
- v0.5.0 agent token auth (S4 in v0.3.0) keeps working alongside mTLS — per-lab `agentAuthMode` and `agentTransport` compose.

**JWT compatibility:** v0.5.0 JWTs without `orgId` are accepted for `JWT_LEGACY_GRACE_DAYS` (default 7) after the upgrade and resolve to the user's default org. Clients that have refreshed their token within the grace window transition without a re-login.

**Socket.IO events:** additive — v0.5.0 dashboards ignore `org:resource-shared`, `hub:ca-rotated`, etc.

## Open questions for owner review

1. **Default RBAC role for invited members.** Proposing `viewer`. Alternative: `editor`, because the common homelab co-op case is "my co-admin, not my audit reviewer". Picks a side between safety and friction. Confirm `viewer` or flip to `editor`.

2. **Org-to-org share semantics vs. just direct-user share.** Proposing both forms. Org-to-org is powerful (MSP-style "I manage this org for you") but adds a second mental model on top of direct-user. Do you want the first release to be **direct-user only**, with org-to-org deferred to v0.6.1 or v0.7.0?

3. **JWT legacy grace window.** Proposing 7 days so existing active sessions don't get kicked mid-use. Alternative: 0 days (force re-login on upgrade) for a cleaner security story. Or 30 days if you want to be very conservative.

4. **Auto-update default channel.** Proposing `pinned` (opt-in auto-update). Strong argument for `stable` as the default: homelab users aren't going to remember to bump their agent, and then we ship a security fix that nobody picks up. Counter-argument: self-updating daemons cause support tickets. Which default matches your user's expectations?

5. **Staged rollout gating.** Proposing stable hash-bucketing by `labId`. Alternative: by `orgId` so all agents in an org roll together (avoids "my three servers are on different versions" confusion). Tradeoff: orgId bucketing means a single large org dominates the early buckets.

6. **ACME in-hub vs. operator-provided.** Proposing both modes. `acme` mode is genuinely convenient for homelab operators who don't run a cert manager; `provided` is for serious deployments. Are you comfortable with the hub owning the ACME client, or should we punt and tell operators to put Caddy in front?

7. **Hub-rooted CA vs. public CA for client certs.** Proposing hub-rooted (self-signed) because client certs don't need third-party trust — the hub itself is the only verifier. Alternative: let operators bring their own CA (e.g. step-ca). Ship with hub-rooted for v0.6.0 and add BYO CA in v0.7.0?

8. **Timescale as the default database image.** Proposing we switch the bundled `docker-compose.yaml` to `timescale/timescaledb:latest-pg16`. This is a larger image (~500 MB vs. ~80 MB) and a clear "this project has an opinion" move. Flip to Timescale-by-default, or keep the official Postgres image and make Timescale an explicit opt-in?

9. **Raw retention window.** Proposing 7 days raw, matching the 1m CAgg window. Alternative: keep the v0.3.0 lab-configurable `retentionDays` as the raw window, because operators used to setting it to 30 or 90 will be surprised by the 7-day drop. Tradeoff: longer raw windows blow up storage — the whole point of tiering.

10. **Status page under `/status/:slug` or as a subdomain.** Proposing path-based, because subdomains require operator DNS + TLS work. But path-based means the page lives on the same origin as the authenticated app, which is a bit uncomfortable for "this is truly public". Path-first for v0.6.0, subdomain support in v0.7.0?

11. **Web Push per-user or per-device channel.** Proposing a single per-user `WEB_PUSH` channel that fans out to N devices. Alternative: one channel per device, routed individually. Single-channel is simpler for routing (one route reaches all the user's devices); multi-channel is more granular ("quiet push on my work laptop, loud push on my phone"). Single for now?

12. **Where does release-signing key custody live?** The spec references an ops runbook. The owner-review decision is: do we want the signing key in GitHub's secret store (simple, single-point-of-failure), in a cloud KMS (better but adds a vendor), or on a hardware token operated by one person (best but creates a single human bottleneck)? Pick one for v0.6.0; documentation follows the choice.
